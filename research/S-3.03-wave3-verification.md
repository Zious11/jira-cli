# S-3.03 External-Claim Verification

This document verifies external factual claims that S-3.03's option matrix (A/B/C)
depends on, before any TDD work begins. It mirrors the pre-flight pattern that
produced DEC-010 and DEC-011.

## Claim 1 — Refresh-token grant on Atlassian Cloud OAuth 2.0 3LO

**Status:** VERIFIED — HIGH confidence

**Citation:** Atlassian Developer Docs, "Refresh tokens" — https://developer.atlassian.com/cloud/oauth/getting-started/refresh-tokens/ (retrieved 2026-05-08)

**Summary:** Atlassian Cloud 3LO supports `grant_type=refresh_token` at
`POST https://auth.atlassian.com/oauth/token`. Rotation is **enforced** — the
refresh token used in the request is **disabled** immediately and a new
refresh token is issued with each successful exchange. Access tokens are
short-lived (community sources consistently report ~1 hour / `expires_in: 3600`);
refresh tokens are long-lived (community/secondary sources cite ~90-day idle
expiry; the official docs page does not state an explicit lifetime, so this
sub-claim is MEDIUM confidence). The store-the-new-refresh-token requirement is
explicit in the docs; failure to persist the rotated value is one of the named
failure modes.

**Implication for jr:** `refresh_oauth_token` at `src/api/auth.rs:719` already
calls the correct endpoint with the correct parameter shape and persists
**both** the new access AND new refresh tokens via `store_oauth_tokens` — that
matches the rotation contract. No code-level correction needed for this claim.

## Claim 2 — Expired-access-token 401 response shape from Jira REST v3

**Status:** REFUTED — HIGH confidence

**Citation:** Atlassian KB, "401 unauthorized error when service account
accesses Jira or Confluence API"
(https://support.atlassian.com/atlassian-cloud/kb/401-unauthorized-error-when-service-account-accesses-jira-or-confluence-api/);
community-thread corroboration at
https://community.atlassian.com/forums/Jira-questions/Getting-a-401-Unauthorized-Error-on-REST-APi-In-one-App-and/qaq-p/1837018
(retrieved 2026-05-08).

**Summary:** The story's claim at line 121 of a `{"code": "EXPIRED"}` body is
**not** what Jira REST API v3 actually returns. The real shape is:

```
HTTP/1.1 401 Unauthorized
(no WWW-Authenticate Bearer error header in practice — RFC 6750 NOT followed)
{"errorMessages": ["The access token provided is expired, revoked, malformed, or invalid for other reasons."]}
```

There is **no** machine-readable `code: "EXPIRED"` field; there is **no**
RFC-6750-compliant `WWW-Authenticate: Bearer error="invalid_token"` header in
the typical response. Detection has to rely on substring matching against
`errorMessages[0]` (`contains("expired")`) — fragile and locale-dependent —
**or** on a blanket "401 → try one refresh" policy that does not differentiate
between expired tokens, revoked tokens, and missing scopes.

**Implication for jr:** Story Option A's trigger condition as written
(`{"code": "EXPIRED"}`) will **never fire**. Option A is only viable if the
spec is rewritten to either:
  - (a) treat any 401 on a request that previously authenticated successfully
    in this process as the refresh trigger, OR
  - (b) substring-match on `errorMessages[0]` (brittle).

This is a **material spec defect** in the same class as the S-2.06 v1 pivot:
the story's wire-up path is predicated on a response shape that does not exist.

## Claim 3 — Refresh-endpoint failure mode (refresh token expired/revoked)

**Status:** VERIFIED — HIGH confidence (with one nuance)

**Citation:** Atlassian Developer Docs, "Refresh tokens" — https://developer.atlassian.com/cloud/oauth/getting-started/refresh-tokens/ (retrieved 2026-05-08)

**Summary:** When the refresh token is unknown/invalid/expired/revoked, the
token endpoint returns:

```
HTTP/1.1 403 Forbidden        (per official docs)
{"error": "invalid_grant", "error_description": "Unknown or invalid refresh token."}
```

This is RFC-6749 §5.2-shaped (`error` + `error_description`). **Nuance:** the
official docs say HTTP **403**, while multiple secondary sources and the
"truto.one" / "nango.dev" community write-ups report the same body shape under
HTTP **400 Bad Request**. The discrepancy is consistent with Atlassian historically
returning 400 and updating to 403 (or vice versa). Implementations should
**treat any 4xx with `error: "invalid_grant"` as terminal** rather than
hardcoding a single status code.

**Implication for jr:** `refresh_oauth_token` (auth.rs:747) already does the
right thing — it bails on `!response.status().is_success()` and surfaces the
RFC-6749 body verbatim. The body-truncation (500 chars) and source-aware hint
are well-suited to triage both 400 and 403 variants. No code change required.

## Claim 4 — Concurrency / single-flight semantics

**Status:** VERIFIED (the risk is real) / INCONCLUSIVE (Atlassian docs do not
formally specify single-flight semantics) — MEDIUM confidence

**Citation:** Nango.dev blog, "Concurrency with OAuth token refreshes"
(https://nango.dev/blog/concurrency-with-oauth-token-refreshes/); Atlassian
community thread "OAuth rotating tokens — Unknown or invalid refresh token"
(https://community.developer.atlassian.com/t/oauth-rotating-tokens-unknown-or-invalid-refresh-token/54555)
(retrieved 2026-05-08).

**Summary:** Atlassian operates a **~10-minute reuse window** during which
the previous refresh token is briefly tolerated alongside the new one — but
this window is not formally guaranteed in the published API contract and is
known to fail in clustered deployments without coordination. Concurrent
parallel API calls that all 401, then all attempt to refresh, are a documented
foot-gun: multiple rotations race and at least one caller observes
`invalid_grant` ("unknown or invalid refresh token"). The community-recommended
mitigation is single-flight per `accountId` (or per profile, in jr's case).

**Implication for jr:** This is the highest-impact risk for Option A. jr is
single-process and uses `tokio` — multiple concurrent `JiraClient::send`
calls (e.g., `issue list` followed by per-issue enrichment, or any future
parallel fetch) sharing one expired access token would **all** see 401, **all**
call `refresh_oauth_token`, and the second-through-Nth refreshes would fail
with `invalid_grant` because the first refresh rotated the token out from
under them. **Option A as currently specified does not address this.** A
correct Option A must add a per-profile `tokio::sync::Mutex` or single-flight
primitive to serialize refreshes (see `tokio::sync::OnceCell` or the
`async-once-cell` / `singleflight-async` pattern). This adds a dependency or
a non-trivial concurrency primitive that the story does not currently scope.

## Claim 5 — `offline_access` scope

**Status:** VERIFIED — HIGH confidence

**Citation:** Atlassian Developer Docs, "Refresh tokens" page (same URL as
Claim 1); the docs explicitly state: "To obtain a refresh token, add
`offline_access` to the **scope** parameter of the authorization URL."
(retrieved 2026-05-08)

**Summary:** Required and unchanged. jr already requests `offline_access`
(verifiable in `oauth_login` flow). Sources also note the late-2024
"unioned scopes on refresh" change — refresh requests now return the union of
previously-granted scopes — but this affects scope semantics, not the
`offline_access` requirement itself.

**Implication for jr:** No code change required.

## Implications for S-3.03

The verification re-shapes the option matrix:

| Original option | Re-evaluated viability after verification |
|---|---|
| **A — Wire into `JiraClient::send`** | Trigger condition (`{"code": "EXPIRED"}`) is fictional (Claim 2). Concurrency story (Claim 4) is unaddressed. Option A is implementable but **its current spec is wrong on two material points** — it cannot be implemented as written. |
| **B — Remove as dead code** | Strengthened. `jr auth refresh` (the user-facing CLI) already covers manual recovery. The function is genuinely unreachable from production paths, and the spec for wiring it correctly (per A) would balloon scope. |
| **C — Keep dormant + test** | Strengthened. The function itself is correct (Claims 1, 3, 5 all verified). A wiremock unit test that asserts endpoint, parameter shape, rotation handling, and 4xx-`invalid_grant` failure surfacing is straightforward and locks in the verified contract for a future `send`-integration story. |

This is structurally analogous to **DEC-010 (S-2.06 v2 pivot)** and
**DEC-011 (S-2.07 v2 pivot)**: the story's external assumptions did not
survive contact with current Atlassian behavior, and a v2 spec is warranted
before any TDD red gate is written.
