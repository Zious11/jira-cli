# S-3.09 Wave-3 Verification: PKCE Deferral Re-Check

## Summary

The PKCE deferral decision recorded 2026-05-04 (DEC-006 / ADR-0013) **remains valid as of 2026-05-08**. No Atlassian announcement, changelog entry, or Developer Console change in the 30-day window leading up to today indicates that public PKCE support has been added to the OAuth 2.0 (3LO) authorization-code flow used by `jr`. **PROCEED** with S-3.09 closure (AC-001..AC-003) and use this report as fresh-evidence appendix to ADR-0013's monitoring trail.

## Per-Claim Verification

### Claim 1: Atlassian Cloud OAuth 2.0 (3LO) PKCE support status (2026-05)

**Verdict:** Unchanged — no public PKCE support.

The Atlassian developer changelog (`developer.atlassian.com/changelog/`) contains **zero entries between 2026-04-08 and 2026-05-08** referencing PKCE, `code_challenge`, `code_verifier`, OAuth 2.0 3LO public-client mode, or OAuth 2.1 enforcement on Jira Cloud. Verified directly via WebFetch.

The closest adjacent OAuth changelog entries:
- **2026-03-19** (cloud OAuth changelog): Multi-user app ownership, Confluence Folders beta, Smart Links v2 — no PKCE.
- **2026-05-04** (Bitbucket Cloud): OAuth changes (no refresh tokens for client credentials, rotating refresh tokens, Bearer-only tokens). **Bitbucket-scoped, not Jira 3LO; does not introduce PKCE.**

Confidence: **HIGH** (changelog and Perplexity cross-checked; both negative).

### Claim 2: Forge / Connect platform deviations

**Verdict:** Out of scope for `jr`; PKCE not a substitute path.

- **Forge** uses internal context tokens and a Forge proxy (`forge-proxy-authorization` header). No traditional OAuth authorization-code flow is exposed; PKCE is not applicable and not documented.
- **Atlassian Connect** uses JWT bearer tokens for app→Atlassian calls (server-to-server). PKCE is not documented as a feature.
- **OAuth 2.0 (3LO) apps** (the platform `jr` uses) remains the only relevant path for a CLI acting on behalf of a user. No public PKCE configuration option exists in Developer Console.

Migrating `jr` to Forge or Connect to obtain PKCE-equivalent protection is a **non-starter** — both are app-hosting platforms designed for in-product extensions, not standalone CLIs.

Confidence: **HIGH**.

### Claim 3: PKCE benefit for embedded-client threat model

**Verdict:** PKCE *would* meaningfully improve the threat model — the deferral is technical-infeasibility-driven, not benefit-driven.

The original ADR-0013 reasoning correctly identifies code-interception attacks as the primary PKCE-mitigated threat. Independent research confirms PKCE provides defense-in-depth that the existing mitigations (fixed callback port, first-listener-wins, XOR-obfuscated secret) **do not fully replicate**:

- **First-listener-wins is not absolute**: a daemon or pre-installed malicious app can register `127.0.0.1:53682` before `jr` runs. ADR-0013 §"Residual Risk" already acknowledges this.
- **Embedded-secret extraction**: XOR obfuscation is friction, not security. An attacker who reverse-engineers the binary AND wins the listener race can complete the token exchange.
- **PKCE survives both**: the `code_verifier` is per-flow ephemeral; an intercepted code without the verifier cannot be exchanged.
- **Short-lived tokens don't help**: the attacker exchanges the code immediately, before expiry.
- RFC 8252 §8.1 / §6 says native apps **MUST** implement PKCE; the embedded confidential-client status does not exempt `jr`.

**Implication:** ADR-0013's reactivation trigger is appropriate and important. The mitigations in place are reasonable for v0.5, but PKCE would be a strict upgrade — not marginal — once Atlassian exposes it. R-M1 (MEDIUM) classification is correctly retained.

Confidence: **HIGH**.
