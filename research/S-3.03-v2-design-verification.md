# S-3.03 v2 Design Verification

This document pre-flight-verifies the v2.0.0 design choices ("Option A-fixed":
wire auto-refresh into `JiraClient::send` with per-profile single-flight
coordination) BEFORE the spec is written. Goal: surface any wrong assumptions
now, not after another implementation cycle.

## Claim 1 — Blanket-401 trigger viability

**Status:** VERIFIED (blanket-401 is the right primary trigger) /
NEEDS-DESIGN-NOTE (revoked-token edge case) — HIGH confidence

**Citation:** Atlassian KB (verified in v1 research, Claim 2) — Jira REST v3
returns a single 401 shape with no machine-readable error code. Perplexity
search 2026-05-08: "Jira REST API v3 401 unauthorized: differentiate expired
access token vs revoked vs scope-removed vs tenant-deleted".

**Summary:** Jira Cloud does NOT differentiate 401 sub-cases (expired,
revoked, scope-changed, tenant-removed) in the response body or headers. All
return the same `{"errorMessages": ["..."]}` envelope. Substring-matching on
the message gains nothing meaningful — locale variability and Atlassian's
freedom to reword the message make it brittle, and the message itself
already lumps "expired, revoked, malformed, or invalid" together. Blanket-401
is the only practical primary trigger.

**Edge cases auto-refresh would mask:** revoked tokens, scope reductions,
tenant deletion, and 2FA step-up. In all four, the refresh attempt itself
will fail with `invalid_grant` (4xx), so jr surfaces the existing
"refresh failed — re-auth needed" message. The mask is one-redundant-network-call,
not a silent failure. ACCEPTABLE.

**Confidence:** HIGH that blanket-401 is correct primary trigger; HIGH that
substring-match alternative is not worth its brittleness.

## Claim 2 — Convention check (gh / aws-cli / gcloud)

**Status:** PARTIALLY-REFUTED — MEDIUM confidence; the v2 framing
"all use blanket-401 silent auto-refresh" is **not accurate**.

**Citation:** Perplexity search 2026-05-08 on aws-cli botocore + gcloud +
gh CLI behavior. Source-file verification on cli/cli was inconclusive
(WebFetch returned only authflow init code, not API-call refresh logic).

**Summary:** The actual industry pattern is **proactive expiry-time refresh
(primary) + 401-reactive retry (fallback)**, not blanket-401 alone:

| Tool      | Primary trigger           | Fallback        | Config flag |
|-----------|---------------------------|-----------------|-------------|
| aws-cli   | Expiry timestamp + buffer | 401/403 retry   | None (default-on) |
| gcloud    | Expiry timestamp          | 401 retry       | None (default-on) |
| gh CLI    | Inconclusive — likely 401-reactive only (PAT-centric); flock-based inter-process sync | retry w/ backoff | None |

**Implication:** The "default-ON, no config flag" decision is well-supported
by industry convention (none of the three tools expose a refresh flag). But
the **trigger pattern** is not what v2 currently describes. v2's blanket-401
is closer to gh's pattern than aws/gcloud's. This is **not a problem** for
jr v2.0.0 — interactive CLI bursts in jr are short enough that proactive
expiry-tracking adds complexity without much benefit — but the spec
should not claim "all three use blanket-401."

**Recommended spec adjustment:** Replace "follows gh/aws-cli/gcloud
convention of blanket-401 + silent auto-refresh" with: "follows the
default-ON, no-flag convention shared by gh/aws-cli/gcloud; uses
401-reactive trigger like gh, with proactive-expiry refresh deferred
as a future optimization (S-3.0X)."

**Confidence:** MEDIUM — aws-cli/gcloud claims well-cited; gh-specific
trigger source not directly verified, only inferred from community
discussions and behavior reports.

## Claim 3 — Rust idiom for per-profile single-flight in tokio

**Status:** NEEDS-DESIGN-CHANGE — HIGH confidence. The proposed
`OnceLock<DashMap<String, Arc<tokio::sync::Mutex<RefreshState>>>>` pattern
is workable but **`moka::future::Cache::get_with` is the canonical 2026
idiom** and would significantly reduce code complexity. However, it adds a
non-trivial dependency. Recommend the simpler tokio-only pattern given jr's
"prefer no new heavy dependency" stance.

**Citation:** Perplexity 2026-05-08; moka official docs at
docs.rs/moka/latest/moka/future/struct.Cache.html#method.get_with (WebFetch
2026-05-08).

**Summary of moka semantics (verbatim from docs):**

> "This method guarantees that concurrent calls on the same not-existing
> key are coalesced into one evaluation of the `init` future. Only one of
> the calls evaluates its future, and other calls wait for that future to
> resolve."

> "This method panics when the `init` future has panicked. When it happens,
> only the caller whose `init` future panicked will get the panic. If there
> are other calls in progress, this method will restart and resolve one of
> the remaining `init` futures."

This is exactly the single-flight-with-panic-recovery semantic jr needs.

**Comparison:**

| Pattern | Lines of code | New deps | Refresh-on-expiry support | Panic recovery |
|---------|--------------:|----------|---------------------------|----------------|
| `moka::future::Cache::get_with` | ~10 | +moka, +deps | TTL-native | Built-in (restarts) |
| `OnceLock<DashMap<String, Arc<tokio::sync::Mutex<RefreshState>>>>` | ~40-60 | +dashmap (medium) | Manual | Manual (poison handling) |
| `tokio::sync::OnceCell` | ~5 | None | NO (one-shot only) | N/A |
| `singleflight-async` crate | ~15 | +singleflight-async (dormant since 2022) | None | Unknown |
| `tokio::sync::broadcast` | ~30 | None | Manual | Manual |

**Recommendation:** Given jr's "prefer no new heavy dependency" constraint
AND the requirement to coordinate refreshes (not just init-once), the
proposed `tokio::sync::Mutex<RefreshState>` per profile via
`OnceLock<HashMap<...>>` is a defensible choice. Concretely:

```rust
// src/api/refresh_coordinator.rs
use std::collections::HashMap;
use std::sync::{Arc, OnceLock, Mutex as StdMutex};
use tokio::sync::Mutex as TokioMutex;

static COORDINATORS: OnceLock<StdMutex<HashMap<String, Arc<TokioMutex<RefreshState>>>>> = OnceLock::new();
```

**No DashMap is needed** — the outer `std::sync::Mutex<HashMap<...>>` is
held only briefly to look up/insert the per-profile `Arc<TokioMutex<...>>`,
then released before the async wait. This avoids a new dependency entirely.

**Footgun warning — `tokio::sync::Mutex` poison vs `std::sync::Mutex`:**
`tokio::sync::Mutex` does NOT poison on panic (unlike `std::sync::Mutex`).
This is the right semantic for refresh coordination — a panicked refresh
should let the next waiter retry. The spec must explicitly use
`tokio::sync::Mutex` here, not `std::sync::Mutex`. Document this in the
spec's "implementation notes" so a future contributor doesn't "fix" it.

**Confidence:** HIGH on idiom comparison; HIGH on the no-DashMap-needed
simplification.

## Claim 4 — Stale-token cache invalidation after refresh

**Status:** VERIFIED design pattern; NEEDS-DESIGN-NOTE — HIGH confidence

**Summary:** The proposed pattern (waiters wake from the Mutex guard, read
the freshly-stored access token from `RefreshState`, retry the original
request with the new token) is the right shape. Specifically:

```rust
struct RefreshState {
    // Most recent successful access token + epoch counter.
    // None means "no refresh has succeeded yet for this profile in
    // this process".
    last_access_token: Option<String>,
    last_refreshed_at: Option<Instant>,
}
```

The waiter re-reads `last_access_token` after acquiring the lock; if a
refresh completed within the last few seconds, it skips its own refresh
entirely and returns the cached token. **No keychain re-read needed in the
hot path** — the in-memory `RefreshState` is the source of truth for the
duration of the process; the keychain is the source of truth across
process restarts.

**Why not re-read keychain every send():** macOS Security framework
keychain access is ~5-50ms synchronous, blocking the tokio executor.
Doing this on every request is unacceptable given jr's typical command
fans out 5-50 requests (e.g., `issue list` + per-issue enrichment).

**Reference patterns:** aws-sdk-rust uses an in-memory credentials cache
with version counter; reqwest-middleware ecosystem uses moka with TTL.
Both keep the hot path off the persistence layer.

**Implication for spec:** AC must specify that `RefreshState` holds the
post-refresh access token in-memory and waiters read FROM IT, not from
the keychain. The keychain `store_oauth_tokens` call happens once in the
refresher path, not in every waiter.

**Confidence:** HIGH.

## Claim 5 — Atlassian's "10-minute reuse window" — single-flight necessity

**Status:** REFUTED — HIGH confidence. The 10-minute reuse window is
**not documented** by Atlassian and is at best an undocumented
implementation detail. **Single-flight is REQUIRED**, not over-engineering.

**Citation:** Perplexity 2026-05-08 — direct verification against
developer.atlassian.com/cloud/oauth/changelog (Mar 2026 update) and
Atlassian developer community threads. The v1 verification's mention of
"~10-minute reuse window" came from a Nango.dev secondary source, NOT from
Atlassian.

**Summary:** Atlassian's official position aligns with RFC 6749 standard
single-use rotation: the previous refresh token is invalidated immediately
on use, and a new one is issued. There is **no contractual grace window**.
Treat refresh as strictly single-use.

**Implication:** The original v1 framing that "single-flight might be
over-engineering if the 10-min window holds" is incorrect. Single-flight
is mandatory for any concurrent burst. This vindicates the v2 design
choice — proceed with single-flight as designed.

**Confidence:** HIGH — direct verification against current Atlassian docs
+ explicit absence of any reuse-window mention in 2026 changelogs.

## Claim 6 — Wiremock concurrency-assertion test pattern

**Status:** VERIFIED — HIGH confidence

**Citation:** docs.rs/wiremock/latest/wiremock/struct.Mock.html (WebFetch
2026-05-08).

**Summary verbatim from docs:**

> "Set an expectation on the number of times this Mock should match in the
> current test case. Expectations are verified when the MockServer is
> shutting down: if the expectation is not satisfied, the MockServer will
> panic."

> [mount_as_scoped] "MockServer will verify that the expectations set on
> the scoped Mock were verified - if not, it will panic."

**Recommended test pattern for AC-005 (N concurrent 401s → exactly 1
refresh):**

```rust
#[tokio::test]
async fn test_concurrent_401s_trigger_single_refresh() {
    let server = MockServer::start().await;

    // 401 mock matches every Jira API call (no expect — we don't care how
    // many times the API endpoint is hit, only the refresh endpoint).
    Mock::given(method("GET")).and(path_regex(r"/rest/api/3/.*"))
        .respond_with(ResponseTemplate::new(401).set_body_json(json!({
            "errorMessages": ["The access token provided is expired..."]
        })))
        .mount(&server).await;

    // Refresh endpoint: expect EXACTLY 1 hit even with N concurrent 401s.
    Mock::given(method("POST")).and(path("/oauth/token"))
        .respond_with(ResponseTemplate::new(200).set_body_json(json!({
            "access_token": "new", "refresh_token": "newrt", "expires_in": 3600
        })))
        .expect(1)
        .mount(&server).await;

    // Spawn N concurrent send() calls
    let handles: Vec<_> = (0..10).map(|_| tokio::spawn(send_call())).collect();
    futures::future::join_all(handles).await;

    // Verification happens automatically on server drop; explicit check optional.
}
```

`Mock::expect(1)` is the canonical idiom; no AtomicUsize needed.

**Confidence:** HIGH.

## Claim 8 — Anti-patterns (rotation-persist failure, inter-process race)

**Status:** VERIFIED both risks are real — HIGH confidence.

**Citation:** Perplexity 2026-05-08 — OAuth refresh-token concurrency
race-condition research; Atlassian community threads (cited in v1
verification, Claim 4).

**Risk 8a — Refresh succeeds, persist-to-keychain fails:** If
`store_oauth_tokens` returns an error after a successful refresh, the
process holds a valid in-memory access token but the keychain still has
the old, now-rotated, refresh token. On next process start, refresh fails
permanently with `invalid_grant`; user must re-auth.

**Mitigation:** v2 spec MUST require persist-before-cache-publish:

```rust
// In refresh path, inside Mutex guard:
let new_tokens = call_refresh_endpoint(...).await?;
store_oauth_tokens(profile, &new_tokens)?;  // FAIL HERE → return error
                                             // Don't update RefreshState.
state.last_access_token = Some(new_tokens.access);
state.last_refreshed_at = Some(Instant::now());
```

If keychain write fails, surface the error to the caller and **do not
update in-memory state** — next request triggers another refresh, which
will fail with `invalid_grant` (because the previous-call rotation was
real even though we couldn't persist the new token), and the user gets
a clear "rotation persisted failed; re-auth required" error.

This is recoverable via `jr auth login` and is the lesser of two evils
(better than a wedged process that thinks it's authenticated).

**Risk 8b — Two jr processes sharing a profile (concurrent CLI
invocations):** Process A and Process B both see expired access token,
both refresh. Process A succeeds (rotates token); Process B fails with
`invalid_grant` and surfaces NotAuthenticated despite auth being fine.

**Mitigation options for v2:**

- **Option A (do nothing in v2):** Document as known limitation.
  Inter-process coordination is out of scope — single-flight is
  intra-process only. User-facing impact: low (CLI usage is typically
  sequential), but scripted parallel invocations would see occasional
  spurious re-auth prompts.

- **Option B (file lock):** `flock(2)` on a per-profile lockfile
  in `~/.cache/jr/v1/<profile>/refresh.lock`. First process acquires,
  refreshes, persists, releases. Second process re-reads keychain
  AFTER lock release and skips refresh. Adds ~30 LOC + `fs2` crate
  (or std::os::unix flock).

- **Option C (post-hoc reconciliation):** On `invalid_grant` failure,
  re-read keychain — if refresh_token has changed since the in-memory
  copy was loaded, retry the original API request with the keychain's
  fresh access_token. ~15 LOC, no new dep, handles the race after-the-fact.

**Recommendation:** Spec the inter-process race as **Risk #8** in v2 with
Option C as the recommended mitigation (cheapest, no new dep, handles the
common case). Option A is also defensible if the spec scopes
"intra-process single-flight only" explicitly.

**Confidence:** HIGH that both risks are real and mitigation is needed.

## Implications for v2 spec

Concrete amendments needed before the spec ships:

1. **Trigger rationale (Claim 2):** Don't claim aws-cli/gcloud use
   blanket-401. Cite the convention as "default-ON, no flag" (correct) and
   the trigger as "401-reactive like gh CLI" (also correct). Note that
   proactive-expiry refresh is a future optimization, not v2.

2. **Single-flight primitive (Claim 3):** Use
   `OnceLock<std::sync::Mutex<HashMap<String, Arc<tokio::sync::Mutex<RefreshState>>>>>`.
   No DashMap dep needed. Document the `tokio::sync::Mutex` (not
   `std::sync::Mutex`) choice for the inner mutex with rationale (no
   poisoning on panic; `.lock().await` doesn't block executor).

3. **Stale-cache invalidation (Claim 4):** Spec waiters must read
   `RefreshState.last_access_token` from the in-memory mutex-guarded
   struct, NOT re-read keychain. Persist-before-publish ordering is
   required.

4. **Single-flight is mandatory (Claim 5):** Drop any v1 framing about
   "10-minute window means we might not need single-flight." That window
   is undocumented; treat refresh as strictly single-use.

5. **Test pattern (Claim 6):** AC-005 test uses `Mock::expect(1)` on the
   token endpoint; verification on MockServer drop. Spawn N concurrent
   `JiraClient::send()` calls via `tokio::spawn` + `join_all`.

6. **Add Risk #8 — inter-process race (Claim 8b):** Spec the failure
   mode and recommend Option C (post-hoc reconcile on `invalid_grant`)
   OR explicitly scope "intra-process single-flight only" with Option A
   documented as known limitation.

7. **Add Risk #9 — persist-failure wedge (Claim 8a):** Persist-to-keychain
   must complete BEFORE updating in-memory `RefreshState`. If persist fails,
   surface error and let next call retry refresh (which will fail with
   `invalid_grant` and trigger re-auth).

8. **Drop atlassian-mcp-server citation (Claim 7):** Source-level details
   not verified; citing it without verification would propagate uncertainty.

## Recommended v2 spec amendments (story-writer dispatch list)

- AMEND: trigger rationale — replace "blanket-401 like gh/aws-cli/gcloud"
  with "default-ON like all three; 401-reactive like gh"
- AMEND: single-flight primitive — specify
  `OnceLock<std::sync::Mutex<HashMap<String, Arc<tokio::sync::Mutex<RefreshState>>>>>`,
  no DashMap dep
- AMEND: cache-invalidation AC — waiters read in-memory `RefreshState`,
  not keychain; persist-before-publish ordering required
- ADD: Risk #8 — inter-process refresh race + Option C mitigation
  (post-hoc reconcile on invalid_grant)
- ADD: Risk #9 — keychain persist-failure handling (persist before publish;
  on persist-failure, surface error and let next call retry)
- ADD: AC-005 test pattern using `Mock::expect(1)` and tokio spawn+join
- DROP: 10-minute reuse-window framing (undocumented; treat as single-use)
- DROP: atlassian-mcp-server reference (unverified)

## Recommended next action

**AMEND-SPEC-THEN-PROCEED**

The v2 design ("Option A-fixed" with single-flight) is fundamentally
sound and verified by this research. The blanket-401 trigger, default-on,
no-flag posture, and single-flight requirement are all correct choices.
However, the spec needs the eight amendments above to ship without
factual errors and to capture two material risks (inter-process race,
persist-failure) that are not yet enumerated.

None of the findings overturn the user's choice of Option A-fixed over
Option C. The findings refine the design rather than reject it.
