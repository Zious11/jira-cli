# S-3.04 Pre-Flight Verification: Multi-cloudId Disambiguation

Verifies external factual claims in `.factory/stories/wave-3/S-3.04-multi-cloudid-disambiguation.md` against live Atlassian documentation and developer community findings before TDD begins.

## Claim 1 — Response schema of `/oauth/token/accessible-resources`

**Verdict:** VERIFIED. Confidence: HIGH.

The endpoint returns an array of objects with fields: `id` (string), `name` (string), `url` (string), `scopes` (array of strings), `avatarUrl` (string). The cloudId is exposed under the field name **`id`** — NOT `cloudId`, `resourceId`, or `siteId`. Downstream gateway calls of the form `https://api.atlassian.com/ex/jira/{cloudId}/rest/api/3/...` use this `id` value as the cloudId path segment.

This matches jr's existing `AccessibleResource.id` destructuring at `src/api/auth.rs:652+`. No struct change required.

**Citations:**
- [Atlassian: Implementing OAuth 2.0 (3LO)](https://developer.atlassian.com/cloud/oauth/getting-started/implementing-oauth-3lo/)
- [Atlassian REST v3: api-oauth-token-accessible-resources-get](https://developer.atlassian.com/cloud/jira/platform/rest/v3/api-group-oauth2-access-token/#api-oauth-token-accessible-resources-get)

## Claim 2 — Multi-site behavior (returns all? order? meaning?)

**Verdict:** PARTIALLY VERIFIED. Confidence: MEDIUM-HIGH.

- The endpoint returns **all sites the user authorized during consent**, not all sites the user technically can access. If a user with access to 5 sites selects 2 in the consent dropdown, only those 2 appear. (Atlassian community thread: "It can be multiple, but it is only those authorized by me.")
- A community-reported convention is that the array is **sorted "most recent first"** (community thread #64938), but this is NOT contractually documented by Atlassian. Treat order as **non-stable / not semantically meaningful** for correctness purposes.
- `len > 1` is reachable in normal use whenever a user authorizes more than one site at the consent screen. Disambiguation logic is justified.

**Implication:** jr must NOT rely on `resources.first()` having any deterministic meaning. The current `src/api/auth.rs` behavior is a real bug, not a theoretical one.

**Citations:**
- [Atlassian Community: accessible-resources returns sites that are not permitted](https://community.developer.atlassian.com/t/urgent-api-accessible-resources-endpoint-returns-sites-resources-that-are-not-permitted-by-the-user/66899)
- [Atlassian Community: how-to-get-site-details (#64938)](https://community.developer.atlassian.com/t/how-to-get-site-details-authorized-by-a-user-during-oauth-2-0-3lo-authorization-step/64938/4)

## Claim 3 — Empty-resource case

**Verdict:** VERIFIED. Confidence: MEDIUM-HIGH.

Empty `[]` is a documented edge case. Causes include: scopes not granting any site, app not installed on any site the user can administer, post-revocation states. The story's choice to return `JrError::NotAuthenticated` (line 127) is a defensible mapping — it matches the user-facing outcome ("token doesn't grant access to any usable site"), though strictly Atlassian does not return an explicit error here, just an empty array. Recommend the error message guide the user to re-run `jr auth login` and double-check site selection in the consent screen.

**Citations:**
- [Atlassian Community: accessible-resources returns empty list (3LO)](https://community.developer.atlassian.com/t/accessible-resources-returns-an-empty-list-oauth2-3lo/45912)

## Claim 4 — Stability of `id` across sessions

**Verdict:** VERIFIED. Confidence: HIGH.

cloudId is a permanent tenant identifier per Atlassian Cloud site. No documented rotation. Long-term caching (in profile config, as the story proposes) is the standard pattern; Atlassian itself exposes the same identifier via `https://<site>.atlassian.net/_edge/tenant_info` and admin.atlassian.com. No staleness risk under normal operation.

The only realistic invalidation events are site deletion or migration — both surface as 404s on subsequent gateway calls and are recoverable by re-running `jr auth login`. Story's storage model is sound.

**Citations:**
- [Atlassian Support: Retrieve my Atlassian site's cloud ID](https://support.atlassian.com/jira/kb/retrieve-my-atlassian-sites-cloud-id/)

## Claim 5 — Atlassian recommended UX pattern for multi-site users

**Verdict:** INCONCLUSIVE (no official pattern). Confidence: HIGH that no official guidance exists.

Atlassian publishes **no** prescriptive UX for post-token site disambiguation. Their developer guide stops at "call accessible-resources." The community-converged pattern is exactly what S-3.04 proposes:

1. Token exchange.
2. Call `/oauth/token/accessible-resources`.
3. If `len == 1`, auto-select.
4. If `len > 1`, app-side picker (interactive prompt) OR user-supplied flag (`--cloud-id`).
5. If `len == 0`, error.

This validates the story's UX design — it matches both community best practice and the only viable pattern given Atlassian's flow. The `--cloud-id` flag for non-interactive / scripted use is an additive improvement over the typical OSS implementation. No reason to pivot the UX shape.

A known design critique (community thread #91300) is that Atlassian violates RFC 6749 by not echoing the user-selected resource back in the token response. The story cannot fix that — only mitigate it, which is what disambiguation does.

**Citations:**
- [Atlassian Community: OAuth 2.0 3LO is broken by design (#91300)](https://community.developer.atlassian.com/t/atlassian-oauth-2-0-3lo-is-broken-by-design-heres-why-it-s-a-problem/91300)
- [Atlassian Community: pre-choose a site when authenticating](https://community.atlassian.com/forums/Jira-questions/How-can-I-pre-choose-a-site-when-authenticating-with-auth/qaq-p/3044540)

## Claim 6 — Fixed callback port 53682 + multi-site selection ordering

**Verdict:** VERIFIED. Confidence: HIGH.

The OAuth flow ordering is unambiguous:

1. Browser redirects to `https://auth.atlassian.com/authorize`.
2. User sees the consent screen, **selects site(s) on Atlassian's UI**.
3. Atlassian redirects to the static `redirect_uri` (jr's `http://127.0.0.1:53682/callback`) with `?code=...`.
4. jr exchanges code for access token at `/oauth/token`.
5. jr calls `/oauth/token/accessible-resources` with the access token.
6. **Site disambiguation happens here, locally, AFTER both the callback hit and the token exchange.**

The callback URL is invariant — Atlassian sends to the same configured `redirect_uri` regardless of site choice. Story AC-004's claim that `--cloud-id` does not affect callback URL is correct; `--cloud-id` is a purely client-side filter applied to the `accessible-resources` response.

**Critical confirmation:** the embedded OAuth design (ADR-0006, fixed port 53682) is fully compatible with multi-site disambiguation. No changes needed to `src/api/auth_embedded.rs` or the Atlassian Developer Console registered callback. The selection logic lives in `auth.rs` between steps 5 and the keychain write.

**Citations:**
- [Atlassian: Enabling OAuth 2.0 (3LO)](https://developer.atlassian.com/cloud/oauth/getting-started/enabling-oauth-3lo/)
- [Atlassian Community: multiple-callback-urls-for-a-single-3lo-app](https://community.developer.atlassian.com/t/multiple-callback-urls-for-a-single-outh-2-0-3lo-app/32413)

## Implications for S-3.04

1. **Schema is as assumed** (Claim 1). No struct rework. Field name is `id`; do not rename to `cloudId` or "fix" the deserializer.
2. **Disambiguation is justified** (Claim 2). The "silently mis-targets" framing in the story context is accurate — `resources.first()` has no semantic meaning. Order is not a contract.
3. **Empty-array handling is correct in shape** (Claim 3). Recommend the error message be actionable: "No Atlassian sites authorized this token. Re-run `jr auth login` and select at least one site at the consent screen."
4. **Profile-stored cloudId is safe long-term** (Claim 4). No TTL or rotation handling needed. 404 on a stored cloudId means site deletion/migration — surface as auth error pointing to `jr auth login`.
5. **UX pattern matches industry consensus** (Claim 5). `--cloud-id` flag (non-interactive) + interactive prompt (TTY) + auto-select on len==1 is the exact pattern other Atlassian OSS clients use. Add `--no-input` interaction: when stdin is not a TTY and `--cloud-id` is omitted with `len > 1`, exit with a clear error listing available cloudIds and their site URLs (matches jr's existing `--no-input` convention in `cli/mod.rs`).
6. **Embedded OAuth flow needs no changes** (Claim 6). Site selection is post-callback, post-token-exchange. The 53682 listener is unaffected.

### Additional design recommendation

Story should explicitly specify that the disambiguation prompt/error displays **both `name` and `url`** (not just `id`) — cloudIds are opaque UUIDs and unhelpful to humans. AC text should require something like: `"my-org (https://my-org.atlassian.net) [cloudId: a1b2c3...]"`. This is not in the current story AC list and would be a UX defect if shipped without it.

### Edge case to add to test plan

If `accessible-resources` is called with a token whose granted scopes don't include any Jira/Confluence product scope, the array can be non-empty but no entry's `scopes` field includes a useful product scope. jr should validate that the chosen resource's `scopes` array contains at least one expected scope (e.g., `read:jira-work`) and error clearly if not. This is not currently in the story and is a realistic failure mode for users who customized scopes via BYO OAuth client.

## Recommended next action

**PROCEED.**

All six claims are either VERIFIED or, in the one INCONCLUSIVE case (Claim 5), the lack of official guidance reinforces rather than undermines the story's chosen approach. No DEC-010/DEC-011-class pivot is warranted. The story's architecture (post-token disambiguation, profile-stored cloudId, `--cloud-id` flag, interactive prompt) maps exactly onto Atlassian's flow constraints.

Two **non-blocking refinements** for the story author to fold into the AC list before TDD:

- **Refinement A** (UX): require disambiguation output to display `name` + `url` alongside `id`.
- **Refinement B** (test plan): add coverage for non-empty-but-unscoped resource arrays and for `--no-input` behavior when `--cloud-id` is omitted and `len > 1`.

Both can be added as story-edit notes; neither is a pivot.
