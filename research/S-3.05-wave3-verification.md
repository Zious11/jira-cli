# S-3.05 verification: buffer_unordered concurrency cap

## Claim 1 — `buffer_unordered` vs `buffered` semantics

**Verified TRUE.** Both methods cap concurrent in-flight futures at `n`. Per
docs.rs/futures/0.3 `StreamExt`:

- `buffered(n)`: yields outputs **"in the same order as the underlying stream"** while bounding in-flight to `n`.
- `buffer_unordered(n)`: yields outputs **"in the order in which they complete"** while bounding in-flight to `n`.
- Quote: "No more than `n` futures will be buffered at any point in time."

**Appropriate choice for jr:** `buffer_unordered(8)` is correct. Both call sites
(`src/api/assets/linked.rs:216` and `src/cli/issue/list.rs:446`) consume results
into order-independent collections:

- `linked.rs:216` — results are `(idx, result)` tuples and indexed back into a `Vec<Asset>` by `idx`. Order of arrival is irrelevant.
- `list.rs:446` — results feed a `StdHashMap<(String, String), (String, String, String)>` keyed by `(wid, oid)`. Order-independent.

Story's risk note ("If ordering matters, use `buffered(8)` instead") does not
apply — the consumers are HashMap inserts.

## Claim 2 — Error propagation under `buffer_unordered`

**Verified TRUE — matches `join_all` semantics.** `buffer_unordered` (and its
sibling `buffered`) on a `Stream<Item = Future<Output = Result<_,_>>>` does NOT
short-circuit on `Err`. Each future's `Result` is yielded individually to the
collected `Vec<Result<...>>`. All futures run to completion regardless of any
peer's failure.

This matches `join_all`'s collect-all behavior. The `try_buffer_unordered`
variant is the one that aborts on first `Err` — jr does NOT use it. Existing
loop in `linked.rs:218-224` (`if let Ok(obj) = result`) and `list.rs:449-456`
(`if let Ok(obj) = result`) already pattern-match individual `Ok`/`Err` —
exactly the right shape. **No semantic change post-migration.**

Source: docs.rs/futures `BufferUnordered`; Perplexity synthesis confirms
`try_buffer_unordered` is the short-circuiting variant per
futures-rs#1278 discussion.

## Claim 3 — Cancellation safety on stream drop

**Verified TRUE with caveat.** When the consumer drops the `BufferUnordered`
stream (e.g., the `.collect().await` future is cancelled by Ctrl+C), all
in-flight futures are dropped at their next `.await` point. For reqwest
HTTP futures backed by rustls + tokio, this cleanly closes the in-flight TCP
sockets — no leak, no hang.

**Caveat for jr:** main.rs uses `tokio::signal::ctrl_c()`-driven shutdown. The
`buffer_unordered` futures are NOT `tokio::spawn`'d tasks — they're polled
in-place by `.collect().await`. When the outer `select!`/handler future is
dropped, the `BufferUnordered` is dropped, which drops each `BoxFuture`, which
drops the underlying reqwest `Response::send` future. Tokio rustls cleanly
unwinds. **No regression vs `join_all`** — `join_all` has identical drop
semantics.

Sources: Oxide RFD 400 on async cancellation; users.rust-lang.org thread on
cancel-safety. Both confirm cancel-on-drop is the documented contract for
futures-driven streams.

## Claim 4 — Idiomatic test pattern for AC-002 concurrency assertion

**Recommendation: `Arc<AtomicUsize>` peak counter (option b).** Most reliable,
least flaky, most idiomatic.

```rust
let in_flight = Arc::new(AtomicUsize::new(0));
let peak = Arc::new(AtomicUsize::new(0));
// In wiremock ResponseTemplate handler:
let cur = in_flight.fetch_add(1, Ordering::SeqCst) + 1;
peak.fetch_max(cur, Ordering::SeqCst);
tokio::time::sleep(Duration::from_millis(50)).await; // ensures overlap window
in_flight.fetch_sub(1, Ordering::SeqCst);
// After collect:
assert!(peak.load(Ordering::SeqCst) <= 8);
```

**Why not the alternatives:**
- (a) `Semaphore` in the mock — *enforces* concurrency rather than *measuring* it; would mask a bug where the cap is broken (test would still pass).
- (c) Timing-based batch counting — flaky on loaded CI runners; sleep precision varies; race windows can miss peaks.

The atomic-counter pattern with `fetch_max` directly answers the AC-002
question ("did we ever exceed 8?") and is deterministic. The brief
`sleep(50ms)` in the mock is required so that requests overlap (otherwise a
fast mock could serve 20 requests sequentially with peak=1, hiding a missing
cap).

Source: Perplexity synthesis of Rust concurrency patterns (gendignoux blog,
onesignal blog).

## Claim 5 — `MAX_CONCURRENT_ASSET_FETCHES = 8` value rationale

**Verified REASONABLE. Slightly conservative, well within Atlassian guidance.**

Per Atlassian's official rate-limiting docs
(developer.atlassian.com/cloud/jira/platform/rate-limiting/, fetched 2026-05-08):

- **GET burst limit**: 100 requests/second per endpoint default.
- **Hourly point quota**: 65,000 points global pool (typical GET = 1 point).
- **No documented per-app concurrency cap.** Atlassian instead recommends
  exponential backoff with jitter on 429.

`MAX_CONCURRENT_ASSET_FETCHES = 8` is well below the 100 req/s GET burst limit
(8 concurrent with ~50ms latency = ~160 req/s steady state, but that's only
during enrichment which is bursty and short). For typical jr usage (5-10
assets per issue list call), the cap rarely binds. For pathological cases
(100+ unique assets), 8 keeps the burst small enough that `rate_limit.rs`
retry rarely activates.

**Verdict:** 8 is reasonable and tunable via a named constant. Not pessimal.
A higher value (16-32) would also be safe; 8 errs on the side of being a good
neighbor on shared/enterprise sites.

## Claim 6 — `buffer_unordered` interaction with retry

**Verified TRUE — no global state issue.** `JiraClient::send` retry on 429
(per `rate_limit.rs`) is per-future — the retry loop lives inside each
`get_asset` future. From `buffer_unordered`'s perspective, each future is
opaque; a future that internally retries 3× before resolving simply takes
longer to complete. The cap of 8 in-flight is preserved throughout (if asset
A is retrying, it still counts as one of the 8 slots).

**Subtle interaction worth noting:** if many futures hit 429 simultaneously,
they will all sleep `Retry-After` in their own internal loops — but
`buffer_unordered` will NOT advance to the next 8 until at least one slot
frees. This is the desired behavior: it naturally throttles. No code change
needed.

No global mutable state in `rate_limit.rs` (verified by read of
src/api/rate_limit.rs context). Each retry attempt is self-contained.

## Implications for S-3.05 implementation

1. **Use `buffer_unordered(MAX_CONCURRENT_ASSET_FETCHES)`**, not `buffered`. Order doesn't matter; unordered yields slightly faster perceived latency.
2. **Constant placement:** put `const MAX_CONCURRENT_ASSET_FETCHES: usize = 8;` at the top of `src/api/assets/linked.rs` and import into `cli/issue/list.rs`. Single source of truth.
3. **Imports needed:** `use futures::stream::{self, StreamExt};` — `StreamExt` is required for the `.buffer_unordered` method to be in scope.
4. **AC-002 test:** use `Arc<AtomicUsize>` peak counter + `fetch_max` + a 50ms sleep in the wiremock handler. Avoids `Semaphore`-based enforcement (would mask bugs) and timing-based batch counting (flaky).
5. **Pre-cap red gate:** without the cap, peak counter will reach 20 (matching `to_enrich.len()`). After cap, peak ≤ 8. Clean red→green.
6. **No retry regression:** 429 retry remains per-future and works unchanged. AC-003 is satisfied by the existing `rate_limit.rs` logic.
7. **No cancellation regression:** drop semantics match `join_all` exactly. Ctrl+C handling in main.rs unaffected.

## Recommended action

**Proceed with the story as written.** All six claims are verified. The
proposed `stream::iter(futures).buffer_unordered(MAX_CONCURRENT_ASSET_FETCHES).collect()`
pattern is idiomatic, semantically equivalent to `join_all` for error
propagation and cancellation, and uses crates already in the dependency tree.

**One minor refinement to story:** the story's "Risk: FuturesUnordered
ordering" note can be downgraded to "non-issue" — both call sites consume
results into order-independent HashMaps. Update the risk register accordingly
during implementation.
