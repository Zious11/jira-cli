# S-3.07 External Claim Verification

Verifies factual claims in S-3.07 (Parts A, B, D) against Atlassian docs, RFCs,
and Rust language semantics. Part C (profile name validation) skipped — no
external facts.

---

## Part A — Retry-After cap (`MAX_RETRY_AFTER_SECS = 60`)

### Claim 1 — Atlassian Retry-After value distribution

**Status: REFUTED for the original cap rationale; cap is still defensible but for a different reason.**

Atlassian Jira Cloud routinely sends `Retry-After` values **far above 60 seconds**
in normal operation:

- Atlassian developer-community forum confirms typical observed values of
  **1425s, 1652s (multiple), 1847s, 2157s, 3089s** (~25–51 minutes).
- Atlassian's own documentation describes a documented ceiling of **3600s (1
  hour)** tied to the rate-limit time-window.
- Atlassian publicly acknowledges these are "incredibly long" and is exploring
  shorter alternatives, but **has not changed the ceiling** as of May 2026.

**Implication:** The story's framing — "`Retry-After: 86400` (24h)" as the
pathological case — overstates how rare the problem is. Real Atlassian instances
send values **20×–50× larger than `MAX_RETRY_AFTER_SECS = 60`** routinely under
load. With a 60s cap, jr would `abort retry → exit error` on essentially **every
real-world 429** instead of waiting it out.

This is a behavior-changing decision, not a defensive guard:
- **Before cap:** jr sleeps 30 min on a real 429 (annoying but correct).
- **After cap:** jr exits on a real 429 (user retries manually after the
  embargo lifts).

Both are reasonable for a CLI (a long sleep is worse than a clean error in an
interactive context), but the story's "surprising behavior" framing is wrong.
The correct framing is: "CLIs should fail fast on long server cooldowns; let
the user re-invoke after the embargo lifts."

### Claim 2 — Retry-After integer-only format

**Status: VERIFIED.**

Atlassian sends `Retry-After` as **delta-seconds (integer)** only. No evidence
in any Atlassian doc, forum thread, or GitHub issue of HTTP-date format. The
current `parse_retry_after` (integer-only `parse::<u64>()`) is correct;
NFR-SCA-1's "DOCUMENT-AS-IS" routing is justified.

### Claim 3 — RFC 9110 / RFC 6585 client abort permission

**Status: VERIFIED.**

RFC 9110 §10.2.3 and §15.6.5 both use **MAY** for `Retry-After` semantics —
servers MAY include the header; clients MAY honor it. There is no MUST or
SHOULD requiring the client to sleep. RFC 9110 §2.5 explicitly allows recipients
to "ignore semantics based on local configuration." Clients MAY abort retries
at any time. Story Part A's behavior (warn + abort + return error) is RFC-kosher.

### Part A verdict — **PROCEED, with reframed rationale**

The 60s cap is still useful for jr (it prevents 30-minute foreground sleeps in
an interactive CLI), but the story's rationale must be rewritten before
implementation:

- DROP "surprising behavior with `Retry-After: 86400`" framing — 86400 is not
  realistic; 1800–3600s is.
- ADD "fail-fast policy: in interactive CLIs, a 30-minute sleep is worse UX
  than an error message" framing — cite the Atlassian community thread on
  observed 1425–3089s values.
- The error message MUST tell the user *why* (e.g., "Jira returned
  Retry-After: 1847s (~31 min). Aborting retry — re-run after the cooldown.").

## Part B — `parse_duration` overflow guard

### Claim 4 — Rust release-profile overflow behavior

**Status: VERIFIED.**

Confirmed against Rust reference / Cargo book:

- Release profile defaults: `overflow-checks = false`. (jr's Cargo.toml does
  not override — checks remain off.)
- With `overflow-checks = false`, integer arithmetic **wraps two's-complement**;
  no panic occurs.
- `panic = "abort"` (in jr's release profile) is **orthogonal** — it controls
  what happens *when* a panic fires, not whether overflow panics. With
  `overflow-checks = false`, overflow does not panic, so `panic = "abort"` is
  irrelevant to overflow behavior.

The story's claim ("a release build with `panic=abort` silently wraps") is
correct. The wrap is caused by `overflow-checks = false`, not by `panic =
"abort"`.

### Claim 5 — Worst-case overflow under 64-byte input cap

**Status: VERIFIED — overflow is mathematically impossible for inputs ≤ 64 bytes.**

Analysis of `parse_duration_validate` (the WV2-SEC-01-capped validator) and
the 3-arg `parse_duration` calculator (S-3.07 Part B's target):

- Input cap: **64 bytes** (`MAX_DURATION_INPUT_LEN`, src/duration.rs:11).
- Worst-case digit string: 63 digits + 1 unit (`w`) = e.g.,
  `999...9w` (63 nines).
- Numeric value: 10⁶³ − 1 ≈ 10⁶³.
- Largest multiplier in calculator: `* days_per_week * hours_per_day * 3600`.
  With realistic ceilings (DPW=7, HPD=24): `7 × 24 × 3600 = 604800 ≈ 6×10⁵`.
- Product: ~6×10⁶⁸. **u64::MAX ≈ 1.8×10¹⁹.** So a 63-digit input does in
  fact overflow u64 by ~50 orders of magnitude.

**However** — the calculator uses `current_num.parse::<u64>()` on the digit
string, which **fails first** with `ParseIntError` (u64 overflow) for any
digit string > 20 characters (u64::MAX has 20 decimal digits). So:

- Input ≤ 20 digits + unit: numeric parse OK, multiplication may overflow.
  Worst case: `99999999999999999999w` (20 nines, value ≈ 10²⁰ — already
  overflows u64 at parse step → bail returns "Invalid number in duration").
- Input ≤ 19 digits + unit: numeric parse OK (max ≈ 10¹⁹), `× 604800` ≈
  6×10²⁴ — overflows u64.
- Input ≤ 14 digits + unit: numeric parse OK (max ≈ 10¹⁴), `× 604800` ≈
  6×10¹⁹ — still overflows u64 marginally.
- Input ≤ 13 digits + unit: max ≈ 10¹³, `× 604800` ≈ 6×10¹⁸ — fits in u64
  (u64::MAX ≈ 1.8×10¹⁹). **Safe.**

**So:** for 14–20-digit inputs in `99...9w` form (well under the 64-byte cap),
multiplication still overflows AFTER the digit parse succeeds. The
`parse::<u64>()` step is NOT a sufficient guard. The story's example
`99999999999999w` (14 nines + `w`) is exactly such an input.

**Verdict on Claim 5:** Overflow IS still reachable through `parse_duration` for
inputs of 14–20 digits + a multiplier unit (`w` or `d`), even with the 64-byte
cap. Part B's `checked_mul` would catch this.

### Part B verdict — **DROP, but for a different reason than the orchestrator's**

The orchestrator's argument was: "WV2-SEC-01's 64-byte cap eliminates overflow,
so Part B is redundant." **That argument is mathematically wrong** (Claim 5
above). Overflow IS reachable for 14–20-digit inputs.

**However**, S-3.10 deletes the 3-arg `parse_duration` entirely (zero
production callers post-S-2.06; only test/proptest callers remain after
S-2.06's worklog refactor moved to server-side parsing). If S-3.10 lands first
or in the same wave, Part B's target code disappears.

**Recommendation:** **DROP Part B**, but document the real reason in the story
amendment:
- NOT "overflow can't happen with the 64-byte cap" (false — see Claim 5).
- YES "the function `parse_duration(input, hpd, dpw)` is being deleted by
  S-3.10; adding `checked_mul` is wasted effort on dead code."

If S-3.10 is delayed or descoped, Part B becomes a real defensive fix again
and should be re-added.

## Part D — `search_issues` anti-loop guard

### Claim 6 — Is the looping `nextPageToken` "not a confirmed bug"?

**Status: REFUTED. The story's claim is OUT OF DATE.**

The repeated/looping `nextPageToken` from `/rest/api/3/search/jql` is a
**confirmed, ongoing, community-documented bug** in Jira Cloud as of May 2026:

- Atlassian developer community has multiple threads where users report
  `nextPageToken` returned every page with `isLast: false`, but the **same
  first-page issues** repeat indefinitely.
- Related JIRACLOUD bugs: JRACLOUD-94632 (initial-page null token, FIXED),
  JRACLOUD-92049 (`startAt` ignored, marked Invalid), JRACLOUD-85546
  (pagination instability).
- A separate community issue (atlassian/atlassian-mcp-server#118) documents
  the same loop in Atlassian's own MCP server implementation.
- Recommended workarounds in the wild include exactly the guard Part D
  proposes: track `seenTokens` and break on duplicate.

The line in S-3.07 that says "this is not a confirmed bug (Atlassian doesn't
return a looping cursor in practice)" is **factually incorrect** as of May 2026.

### Claim 7 — Atlassian's documented anti-loop guidance

**Status: PARTIALLY VERIFIED — no formal Atlassian guidance, but strong
community consensus.**

Atlassian's official `/search/jql` documentation does NOT prescribe an
anti-loop guard. However:

- Community-recommended patterns universally include duplicate-token
  detection or per-issue-key deduplication.
- ankitpokhrel/jira-cli (a sibling project) issue #898 documents the same
  pagination-loop class.
- Atlassian's own examples push "infinite scroll / load more" UX, which
  implicitly papers over the loop with deduplication.

### Part D verdict — **PROCEED, and ELEVATE PRIORITY**

Part D is mis-classified in S-3.07. It is described as a "defensive" guard for
an unconfirmed bug, with DOCUMENT-AS-IS routing. In reality:

- The bug IS confirmed and ongoing (May 2026).
- Other Jira clients (atlassian-mcp-server, ankitpokhrel/jira-cli) have hit
  the same defect.
- Without the guard, jr's `search_issues` can hang indefinitely on a real,
  reproducible Atlassian bug.

The current code at src/api/jira/issues.rs:59-65 already contains a
"KNOWN-GAP: NFR-R-F" comment acknowledging the gap. The guard should be
implemented as a real defensive fix, not as a DOCUMENT-AS-IS comment.

Recommended changes to the story:
- Reclassify Part D from "defensive gap" to "live bug mitigation."
- Cite the Atlassian community threads + JRACLOUD-94632 in the source comment
  (not just "see get_changelog pattern").
- Add a stderr warning when the guard fires (so users know they hit the bug
  and can report it / try the legacy `/rest/api/3/search` endpoint).
- Consider raising priority above the rest of S-3.07's bundle.
