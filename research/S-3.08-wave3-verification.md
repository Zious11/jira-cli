# S-3.08 Wave-3 External-Claim Verification

Two NFRs in S-3.08 (`tdd_mode: facade`, doc-only) make external factual claims that
must be verified before the source comments ship. This report verifies each claim
against Atlassian's official developer documentation.

## Claim 1 — NFR-O-T: Worklog endpoint default page size = 5000

**Status:** NEEDS-DESIGN-CHANGE (claim is plausible-but-unsupported; the comment
overstates certainty by saying "5000 per page (per REST API docs)")

**Planned comment text** (story line 152-158):
> Atlassian default max-results for worklogs is 5000 per page (per REST API docs).

**What the docs actually say**

The official endpoint reference
([developer.atlassian.com — Get issue worklogs](https://developer.atlassian.com/cloud/jira/platform/rest/v3/api-group-issue-worklogs/#api-rest-api-3-issue-issueidorkey-worklog-get),
retrieved 2026-05-08) documents `startAt`, `maxResults`, `expand`,
`startedAfter`, `startedBefore` query parameters and returns a `PageBean<Worklog>`
envelope (`startAt`, `maxResults`, `total`, `worklogs[]`). The endpoint **does**
support offset-based pagination — it has not been migrated to cursor-based.

However, Atlassian's policy page on `maxResults`
([JRACLOUD-67570](https://jira.atlassian.com/browse/JRACLOUD-67570)) states
verbatim that **"default and maximum sizes of paged data are not considered part
of the API and may change without deprecation notice."** The endpoint reference
page itself does not quote a numeric default for the worklog endpoint.

The "5000" figure traces to a community thread
([CDAC-78720](https://community.developer.atlassian.com/t/failed-to-get-more-than-the-maxresults-limit-for-worklogs-for-an-issue-through-rest-api/78720))
where staff/users report observed behaviour, not to canonical documentation.
The competing "1048" figure is unsubstantiated — likely a transcription error
elsewhere.

**Confidence:** medium. Behaviour is *probably* 5000 today, but Atlassian
explicitly disclaims any contractual guarantee. A source comment that asserts
"5000 per REST API docs" is **not accurate** — the docs do not state a number.

**Recommended comment rewrite:**
```rust
// NFR-O-T: The worklog endpoint returns a PageBean<Worklog> envelope with
// offset-based startAt/maxResults pagination. Atlassian does not contractually
// document a default page size (per JRACLOUD-67570: "default and maximum sizes
// of paged data are not considered part of the API"). Observed behaviour as of
// 2026-05 returns up to ~5000 worklogs per page; treat this as best-effort and
// always loop on pagination, never assume a single call returns everything.
```

## Claim 2 — NFR-O-I: ADF mention/emoji/inlineCard/media rendering hints

**Status:** NEEDS-DESIGN-CHANGE (rendering hints are partially wrong; canonical
attribute names exist and should be used).

**Planned comment text** (story line 62-65):
> render `@<displayName>` for mentions, `:emoji:` for emojis, `[<title>](url)`
> for inlineCard.

**What the ADF spec says** (retrieved 2026-05-08):

- **mention** ([nodes/mention](https://developer.atlassian.com/cloud/jira/platform/apis/document/nodes/mention/)):
  attrs are `id` (required, accountId), `text` (textual representation
  **including a leading @**), `accessLevel`, `userType`. There is no
  `displayName` attr — the canonical render source is `attrs.text`, which
  already contains `@`.
- **emoji** ([nodes/emoji](https://developer.atlassian.com/cloud/jira/platform/apis/document/nodes/emoji/)):
  attrs are `shortName` (required, e.g. `:grinning:`), `id` (optional service
  ID), `text` (optional unicode glyph). Spec: "`shortName` is rendered instead
  if [text is] omitted." Canonical preference is `attrs.text` (real glyph) with
  fallback to `attrs.shortName`.
- **inlineCard** ([nodes/inlineCard](https://developer.atlassian.com/cloud/jira/platform/apis/document/nodes/inlineCard/)):
  attrs are `url` and `data` (JSON-LD). Spec: **"Either `data` or `url` must be
  provided, but not both."** There is no guaranteed `title` to render — the
  story's `[<title>](url)` is wrong because `title` may not exist; the URL
  itself is the only universally available label.
- **media** ([nodes/media](https://developer.atlassian.com/cloud/jira/platform/apis/document/nodes/media/)):
  attrs are `id` (required, Media Services ID), `type` (`file`|`link`),
  `collection` (required), plus optional `width`/`height`/`alt`. There is no
  `fileName` attr surfaced on the node — fileName requires a Media Services
  API call. A `[image]` or `[media]` placeholder is the most that's locally
  available.

**Confidence:** high. All four node specs were retrieved from the canonical
Atlassian developer reference today.

**Recommended comment rewrite:**
```rust
// NFR-O-I: ADF inline nodes mention/emoji/inlineCard/media fall through to `_`
// here. Canonical render hints (per developer.atlassian.com/cloud/jira/platform/
// apis/document/nodes/, retrieved 2026-05):
//   mention    -> attrs.text (already includes leading "@"); fallback "@?"
//   emoji      -> attrs.text (unicode glyph) or attrs.shortName (e.g. ":smile:")
//   inlineCard -> attrs.url (title not guaranteed; either url OR data, not both)
//   media      -> "[media]" placeholder; fileName requires Media Services call
// Not implemented in v0.5; tracked under issue #202.
```
