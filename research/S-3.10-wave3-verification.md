# S-3.10 Verification Report

Five claims verified against current Rust ecosystem documentation and primary sources.

## Claim 1 — Idiomatic proptest pattern for round-trip without inverse

**Story stance:** Inline structural-token-parser in the test body (the ~12 line sketch at story lines 115-135).

**Verification (Perplexity, proptest book reference):** The community-recommended pattern is **NOT** an inline tokenizer. The idiomatic shape is:

1. Generate a **structured value** (e.g., `(h, m)` tuple or a `Duration` struct) with `prop_map` / `prop_flat_map`.
2. Format it: `let s = format_duration(seconds_from_struct);`.
3. Assert a property of the formatted output (or, if the inverse exists, parse and compare).

For S-3.10's case (no inverse parser permitted), the cleanest variant is option (a) of the four candidates: a small inline tokenizer in the test body, but it should be written as a tight loop over `(suffix, multiplier)` pairs — exactly what the story sketch already does. The sketch is acceptable.

**However**, an even more idiomatic alternative the story does not consider:
- Generate the seconds value via `(0u64..24, 0u64..60).prop_map(|(h,m)| h*3600 + m*60)`. Then `format_duration(seconds)` is asserted to equal the expected string built from `(h,m)` directly — no tokenizer needed at all. This is the pattern referenced in the proptest book's "transforming strategies" tutorial.

**Verdict:** Story's structural-tokenizer approach is **valid but not optimal**. Recommend the implementer consider the `prop_map`-from-`(h,m)` form as a simpler alternative (~6 lines vs ~12). Either is acceptable; both eliminate the inverse-call dependency and unblock calculator deletion.

Source: [proptest book — Transforming strategies](https://altsysrq.github.io/proptest-book/proptest/tutorial/transforming-strategies.html)

## Claim 2 — "Code pollution" anti-pattern citation

**Verified (WebFetch direct from enterprisecraftsmanship.com).** Article exists. Central thesis confirmed verbatim: *"Don't pollute production code base with code that exists solely to enable unit testing."* The article identifies test-only methods and test-conditional branches as the two primary manifestations.

The story's application of this principle to "preserving `parse_duration` (3-arg) only because a proptest uses it" is a faithful extension. The original article focuses on `GetByCustomerId()`-style helpers and `isTestEnvironment` branches; using it to justify removing a deprecated function whose only remaining caller is a test is a reasonable analogical application, though one step removed from Khorikov's exact examples.

**Verdict:** Citation accurate. Application is sound but slightly extended from the article's literal scope. No change needed.

## Claim 3 — Meszaros "Obsolete Test" + Fowler observable-outcome principle

**Verified (Perplexity).** Both are real, citable principles:

- **Meszaros, *xUnit Test Patterns* (2007), Addison-Wesley** — "Obsolete Test" smell is a documented test smell. Tests that pin internal implementation details rather than observable behavior become obsolete when implementations change.
- **Fowler** — observable-behavior testing principle is restated across his bliki and *Refactoring* (2nd ed., 2018). Widely echoed in Khorikov's *Unit Testing Principles* (2020).

**Verdict:** Both citations are real and well-established. No change needed.

## Claim 4 — `prop_filter` divisible-by-60 idiom (FLAG)

**Story line 117:** `(1u64..86400).prop_filter("divisible by 60", |s| s % 60 == 0)`

**Verification (Perplexity, proptest docs):** This is **borderline anti-idiomatic**. The proptest crate's own documentation explicitly states `prop_filter` should only be used when "(a) relatively few values will actually be rejected" — but here 59/60 values are rejected (~98.3% rejection rate).

- Default `LOCAL_REJECT_LIMIT` = 65,536. A 1/60 pass-rate means proptest generates ~60k values to surface ~1k passing cases. It does **not** exceed the budget, but it wastes ~98% of generation cycles and degrades shrinking quality (rejected candidates can't guide minimization).
- The idiomatic replacement is `(1u64..1440).prop_map(|m| m * 60)` — generate the minute count directly, multiply to get seconds. Same value space, zero rejection, better shrinking.

**Verdict:** Story should swap `prop_filter` for `prop_map`. This is a small but real improvement and aligns with the proptest book's filtering guidance. Recommend story AC-001 be amended to specify `prop_map` directly, since the implementer otherwise codifies the suboptimal pattern.

Source: [proptest book — Filtering](https://altsysrq.github.io/proptest-book/proptest/tutorial/filtering.html)

## Claim 5 — `cargo build` as zero-caller verification gate (PARTIAL)

**Story claim:** `cargo build` exiting 0 confirms zero callers of the deleted function.

**Verification (Perplexity + Rust knowledge):** **Plain `cargo build` is insufficient as the gate.** It compiles only lib + bins + examples — **not** tests or benches. Since the deleted function's only remaining caller is the proptest (a test target), `cargo build` would have exited 0 even *before* the proptest was rewritten. The gate as stated does not actually verify what the story claims.

The correct gate is **`cargo build --all-targets`** or, equivalently, **`cargo test --all-targets --no-run`**. These compile every target including tests, benches, and integration tests under `tests/`.

The story's Test Plan step 7 already lists `cargo test --all-targets` correctly. The narrative claim in "Risk / Dependencies" line 238 (`cargo build is the definitive zero-caller verification gate`) is technically wrong and should be updated to `cargo build --all-targets` or `cargo test --all-targets --no-run`.

Additionally: the project does not use feature-gated callers for `parse_duration` (verified by reading `src/duration.rs`), so the standard concern about feature-flagged dead code does not apply here.

**Verdict:** Substantive amendment recommended. Update line 238 to reference `--all-targets`. The remainder of the Test Plan is correct.

## Recommended Next Action

**PROCEED-WITH-AMENDMENT.** The story is approved to start with two amendments:

1. **AC-001 / Implementation Notes:** Replace `(1u64..86400).prop_filter("divisible by 60", |s| s % 60 == 0)` with `(1u64..1440).prop_map(|m| m * 60)`. Document the swap as "preferred per proptest book filtering guidance." Optionally promote the structured `(h, m)` generator form for the round-trip body if the implementer prefers it.
2. **Risk / Dependencies (line 238):** Replace `cargo build` with `cargo build --all-targets` (or `cargo test --all-targets --no-run`) as the zero-caller verification gate. Plain `cargo build` does not compile test targets and would not catch a remaining proptest caller.

These amendments are corrections, not scope changes. Story remains low-stakes and small-effort.
