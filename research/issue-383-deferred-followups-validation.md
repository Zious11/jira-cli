### Fix direction check — proposed direction is **CORRECT**

The proposed fix (rename `## BC-3.8:` → `### 3.8 <Name>`) is in the right direction.
Cross-checking other bc-N body files confirms `### N.M <Name>` is the project standard:

- `bc-1-auth-identity.md:24` — `### 1.1 OAuth Flow & Profile Resolution`
- `bc-2-issue-read.md:24` — `### 2.1 JQL Composition (the canonical build pipeline)`
- `bc-4-assets-cmdb.md:24` — `### 4.1 AQL / CMDB Field Resolution`
- `bc-7-output-render.md:23` — `### 7.1 Table / JSON Output`

Every other bounded-context file uses `### N.M <Name>`. bc-3's 3.1–3.7 follow the
standard; **3.8 is the lone outlier.** The proposed fix direction (conform 3.8 to the
others) is correct — NOT the reverse.

Decisively, `BC-INDEX.md:301` already lists this subdomain with the standard format:
`### 3.8 JSM Request Create + Platform-Path Inverse Warnings (13 BCs: BC-3.8.001..013)`.
So the canonical index and the body file currently disagree on heading style; the fix
brings the body into line with the index that derives from it.

### Tooling-impact check

The candidate's claim that `^### 3\.` filtering would miss 3.8 is correct — but note the
project's actual count guard (`check-spec-counts.sh`) filters on `^#### BC-`
(definitional headings, h4), NOT on `^### N.` (subdomain headings, h3). So no CURRENT
script breaks today. The risk is latent: any future tooling that enumerates subdomains
by `^### ` would silently miss 3.8. The "8 subdomains" preamble claim
(`bc-3-issue-write.md:21-23`) is itself correct in count but inconsistent with the
machine-greppable heading shape.

### Anchor-breakage check

The current anchor generated by `## BC-3.8: JSM Request Create + Platform-Path Inverse
Warnings` differs from what `### 3.8 JSM Request Create + Platform-Path Inverse Warnings`
would generate. I found no cross-file links targeting the `#bc-38-...` anchor — BC
cross-references in the body (e.g., "see BC-3.8.001") point to the h4 `#### BC-3.8.NNN:`
headings, not the subdomain h2/h3. The h4 BC anchors are unaffected by this rename.
**Low anchor-breakage risk**, but the issue body should still instruct the fixer to
grep for any `#bc-3` anchor links before merging.

## Candidate Issue 3 — DEFER-383-3 (process-gap) — HIGHEST PRIORITY

### Verdict: **FILE-WITH-REVISION**

### Premise check: **CONFIRMED** (with one important scope refinement)

`scripts/check-spec-counts.sh` was read in full (62 lines). Confirmed: it validates
**only** three things, and `total_bcs` / cumulative sums are NOT among them:

1. Per-`bc-*.md` file: `grep -c '^#### BC-'` vs frontmatter `definitional_count:`
   (lines 21-29).
2. `nfr-catalog.md`: `^| **NFR-` count vs `total_nfrs:` (lines 32-42).
3. `holdout-scenarios.md`: `^### H-` count vs `total_holdouts:` (lines 44-55).

It does **NOT** validate:
- The per-file `total_bcs:` frontmatter value (the cumulative claim) against anything.
- The grand-total sum across files.
- The `BC-INDEX.md` Section-N header counts (e.g., `## Section 3: ... — 91 BCs
  cumulative; 62 individually-bodied`).
- The `BC-INDEX.md` per-subdomain header counts (e.g., `### 3.8 ... (13 BCs:
  BC-3.8.001..013)`).
- `CANONICAL-COUNTS.md` table values.

`check-bc-no-numeric-test-counts.sh` was also read in full — it is a **different
concern** (PG-365-1: forbids numeric test counts in BC `Trace:`/`Source:` fields). Zero
overlap with count-reconciliation. Confirmed: no existing script covers the
cumulative/Section-header drift class.

This gap is real and is a genuine process hole. The proliferation of count claims is
substantial — `total_bcs` appears in 8 body-file frontmatters; the same numbers are
restated in `BC-INDEX.md` frontmatter `sections:`, in each `## Section N:` header, in
each `### N.M` subdomain header, in `CANONICAL-COUNTS.md`'s two tables, its grand-total
prose, and its L2-vs-L3 alignment table; plus README.md. Nothing machine-checks that
these agree.

### Scope refinement #1 — the candidate's diagnosis of the #383 F2 drift is plausible but UNVERIFIED

The candidate claims "~10 stale-count drifts survived into the #383 F2 cycle and
required 11 adversary passes to sweep" and asks to cross-check
`.factory/phase-f2-spec-evolution/issue-383/adversary-convergence-log.md`. **That
directory does not exist** — `.factory/phase-f2-spec-evolution/issue-383/` returned no
files (the Glob/listing found nothing; the path is absent in this worktree). So the
"~10 drifts / 11 passes" figure **could not be verified** and should NOT be stated as
fact in the filed issue. Recommend the issue describe the gap structurally (what the
script does not check) rather than citing an unverifiable incident count.

What CAN be verified from the current files is that the gap is real and that the count
surface is wide and interdependent — that alone justifies the issue.

### Scope refinement #2 — the proposed fix WOULD trip on intentionally-PENDING rows

This is the most important refinement. A naive cumulative-sum / cross-file equality
check **would produce false positives** against the current repo state. Verified
examples of intentional, documented mismatches a naive check would flag:

- `CANONICAL-COUNTS.md:77` — L2 `bc-02` bc_count = 92 vs L3 `bc-2` total_bcs = 93,
  explicitly marked `PENDING (L2 bc_count not yet bumped; L3 +1 BC-2.6.051 ...)`.
- `CANONICAL-COUNTS.md:78` — L2 `bc-03` bc_count = 77 vs L3 `bc-3` total_bcs = 91,
  explicitly marked `PENDING (L2 bc_count not yet bumped; ...)`.
- Within `bc-2-issue-read.md` itself there is an internal inconsistency the check must
  tolerate or flag deliberately: frontmatter `total_bcs: 93` (line 4) but the body
  preamble line 17 says *"92 behavioral contracts"*. This is itself a live drift
  (the +1 BC-2.6.051 bumped frontmatter but not the prose) — a cumulative check that
  compared frontmatter-to-prose would (correctly) catch it, but the issue author must
  decide whether prose-count lines are in scope.

A correct check must therefore:
- Validate the **L3-internal** consistency that is genuinely invariant: each
  `bc-N.md` `total_bcs:` == the `BC-INDEX.md` `## Section N:` header cumulative count
  == the `BC-INDEX.md` frontmatter `sections:` line == the `CANONICAL-COUNTS.md`
  per-file `total_bcs` table row. These four SHOULD always agree — there is no
  legitimate "pending" state among them.
- Validate the grand total: sum of the 8 `total_bcs` values ==
  `BC-INDEX.md` frontmatter `total_bcs:` == `CANONICAL-COUNTS.md` "Sum" (line 51) and
  "grand total" (line 53/55).
- **Explicitly exclude** the L2-vs-L3 alignment table (`CANONICAL-COUNTS.md` lines
  74-82): those rows carry a documented `PENDING` state and represent a deliberate,
  tracked divergence between L2 domain-spec and L3 PRD. The check must skip any table
  row whose cells contain `PENDING`, or skip that table section entirely.

So the proposed fix is **technically sound in principle but under-scoped as written** —
it needs the PENDING-tolerance carve-out and a precise definition of which count
surfaces are "must-agree invariants" vs "tracked-divergence-allowed."

### Is the fix worth filing?

Yes — this is the highest-value of the three. The count surface is wide, restated in
~6+ locations, and only the narrowest slice (`definitional_count`) is currently guarded.
The drift class is real (the `bc-2` frontmatter-93 vs prose-92 mismatch found during
this very validation is a live example). A correctly-scoped check would have caught it.

### Suggested issue

- **Title:** `ci: extend spec-count guard to validate cumulative total_bcs + BC-INDEX section headers`
- **Label:** `ci` / `process` (project's process-gap equivalent)
- **Body:**
  > `scripts/check-spec-counts.sh` currently validates only per-file `definitional_count`

  > `bc-2-issue-read.md` frontmatter says `total_bcs: 93` but its body preamble (line 17)

  > would catch.
  >
  > **Proposed fix:** extend `check-spec-counts.sh` (or add a sibling
  > `check-bc-cumulative-counts.sh`) to assert the must-agree invariant set: for each
  > bounded context N, `bc-N.md` `total_bcs:` == `BC-INDEX.md` `## Section N:` header
  > cumulative count == `BC-INDEX.md` frontmatter `sections:` line == `CANONICAL-COUNTS.md`
  > per-file `total_bcs` table row; and the grand total = sum of the 8 values =
  > `BC-INDEX.md` frontmatter `total_bcs:` = `CANONICAL-COUNTS.md` Sum/grand-total.
  >
  > **Required scope carve-out:** the check MUST skip the `CANONICAL-COUNTS.md` "L2
  > domain-spec bc_count vs L3 total_bcs alignment" table (lines ~74-82) — those rows

  > divergence. Skip any table row containing `PENDING`. Without this carve-out the
