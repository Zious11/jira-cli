# S-1.02: Enforce cargo-deny supply chain hardening

## Context

`deny.toml` currently sets `multiple-versions = "warn"` in the `[bans]` section,
meaning duplicate transitive crate versions are flagged but do not fail the build.
NFR-S-F (HIGH) identifies this as a supply-chain risk: with 332 transitive Cargo
dependencies for an OAuth-handling CLI, silent version duplication expands the attack
surface and binary size without any CI gate.

The fix has two parts:

1. **Raise `multiple-versions` from `"warn"` to `"deny"`** in `deny.toml`, or
   document all accepted duplicates via explicit `[[bans.skip]]` entries so the policy
   is stated explicitly rather than silently tolerated.

2. **Verify `unknown-registry = "warn"` and `unknown-git = "warn"` are not silently
   passing hidden supply-chain risks.** The cicd-setup.md §7 calls out these two
   settings as equivalent concerns. Raise them to `"deny"` or document as accepted
   policy with a comment in `deny.toml`.

No new CI job is required. The existing `deny` job in `ci.yml` runs
`EmbarkStudios/cargo-deny-action@v2`, which enforces `deny.toml` on every PR. Once
`deny.toml` is tightened, CI immediately enforces it.

## Behavioral Contract

No BC anchor exists for CI/CD supply-chain policy. This story is anchored to
NFR-S-F (HIGH). The ACs define the measurable target state.

## Acceptance Criteria

**AC-001** (traces to NFR-S-F): `deny.toml` `[bans]` section has `multiple-versions`
set to either `"deny"` (with zero unapproved duplicates) OR `"warn"` with a `[[bans.skip]]`
entry for every currently-present duplicate, each with a non-empty `reason` string explaining
why the duplicate is accepted.

**AC-002** (traces to NFR-S-F): `cargo deny check bans` exits 0 on the current
dependency tree. No silent warnings that fail to block.

**AC-003** (traces to NFR-S-F, cicd-setup.md §7): `[bans]` `unknown-registry` and
`unknown-git` are either set to `"deny"` or explicitly commented with an acceptance
rationale. The implementation MUST NOT leave them at `"warn"` without a comment
explaining why `"warn"` is acceptable.

**AC-004** (traces to NFR-S-F): `cargo deny check` (all categories: advisories,
licenses, bans, sources) exits 0 in CI after this story merges.

## Holdout Strategy

No Phase 4 holdout. Verification:
- `cargo deny check bans 2>&1 | grep -c "error"` returns 0 after implementation.
- CI `deny` job passes on the PR that implements this story.
- Reviewer verifies that every `[[bans.skip]]` entry has a non-empty `reason` field.

## Implementation Notes

### Step 1: Inventory current duplicates

```bash
cargo deny check bans 2>&1 | grep "duplicate"
```

This lists all currently-duplicated crates. For each:
- If the duplicate is avoidable (i.e., a dependency could be unified via `cargo update` or
  a Cargo.toml version constraint change), unify it.
- If the duplicate is unavoidable (e.g., two major-version dependencies required by different
  transitive crates), add a `[[bans.skip]]` entry:

```toml
[[bans.skip]]
name = "bitflags"
version = "1"
reason = "syn 1.x and clap 3.x require bitflags 1; bitflags 2 is incompatible; accepted as transitive-only duplicate."
```

### Step 2: Set multiple-versions policy

Option A (preferred if duplicate count is manageable):
```toml
[bans]
multiple-versions = "deny"
```
Then add explicit `[[bans.skip]]` for each unavoidable duplicate.

Option B (acceptable if duplicate count is large):
```toml
[bans]
multiple-versions = "warn"
# NOTE: We acknowledge N duplicates listed in [[bans.skip]] below. Each is documented.
```

The story requires that Option B not be used without explicit `[[bans.skip]]` entries —
silent `"warn"` with no skip list is the current unacceptable state.

### Step 3: Tighten unknown-registry and unknown-git

```toml
[sources]
unknown-registry = "deny"   # or "warn" with comment if specific non-crates.io registries are expected
unknown-git = "deny"        # or "warn" with comment if specific git deps expected
```

Check `Cargo.toml` for any `git = "..."` dependencies before raising to `"deny"`.
Current `Cargo.toml` uses only `crates.io` sources; raising to `"deny"` should be safe.

## Risk / Dependencies

- **No Wave 0 dependency.** Independent infrastructure work.
- **Risk: large skip list.** If there are many unavoidable duplicates, the `[[bans.skip]]`
  list will be long. This is expected for a project with 332 transitive deps. Each entry
  must have a `reason`.
- **Risk: advisory noise.** Running `cargo deny check` for the first time at higher
  strictness may surface existing advisories. Treat any new errors as separate issues;
  do not suppress them.
- **No breaking change** for end users. This is a CI policy tightening only.
- **Blocks:** None explicitly, but stricter bans policy will gate all future dependency
  additions at CI — a desirable side effect.

## Out of Scope

- SBOM generation (GAP-4, deferred).
- Binary signing (GAP-5, deferred).
- `cargo audit` as a standalone step (cargo-deny covers the advisories database).
- Upgrading any specific crate to remove a duplicate (unless trivially safe during
  this story's implementation).
