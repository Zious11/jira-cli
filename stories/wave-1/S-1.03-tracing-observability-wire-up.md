# S-1.03: Add tracing crate + wire structured logging to client.rs and auth.rs

## Context

`tracing` is NOT currently a dependency of `jr`. The codebase has 47 `eprintln!` and
103 `println!` calls across 24 CLI handlers (NFR-O-A, MEDIUM). This story adds `tracing`
and `tracing-subscriber` as dependencies, initializes a subscriber in `main.rs`, and
replaces the most critical `eprintln!("[verbose]...")` sites in `client.rs` and `auth.rs`
with `tracing::debug!` / `tracing::info!` calls.

Scope is deliberately limited to the API layer (`client.rs`, `auth.rs`): these are the
noisiest and most agent-integration-relevant sites. CLI handler `eprintln!` sites (hint
messages, table truncation warnings) are intentional user-facing output and are NOT
replaced in this story.

S-0.06 (Wave 0) introduced the `--verbose-bodies` flag and PII warning. This story
integrates tracing so that `--verbose` (headers only) and `--verbose-bodies` (full body)
are respected by the structured logging subscriber, rather than being hardcoded
`eprintln!` calls. SD-003 is satisfied by S-0.06; this story integrates the tracing
layer that makes SD-003's verbose behavior structurally sound going forward.

## Behavioral Contract

No standalone BC exists for observability infrastructure. This story is anchored to
NFR-O-A. The ACs below define the measurable behavior.

## Acceptance Criteria

**AC-001** (traces to NFR-O-A): `Cargo.toml` `[dependencies]` contains `tracing` and
`tracing-subscriber` with explicit version pins (not `*`). Both compile without warnings
under `cargo clippy --all -- -D warnings`.

**AC-002** (traces to NFR-O-A): `main.rs` initializes a `tracing-subscriber` with
`EnvFilter` that defaults to `WARN` level (no verbose output by default) and escalates
to `DEBUG` when `--verbose` is passed. When `--verbose-bodies` is passed, filter level
includes `TRACE` for body-logging events in `client.rs`.

**AC-003** (traces to NFR-O-A): The `eprintln!("[verbose] ...")` calls in `client.rs`
that log request/response headers are replaced by `tracing::debug!` events with structured
fields (e.g., `tracing::debug!(method = %method, url = %url, "request")`). Behavior
under `--verbose` is equivalent to current (headers shown); behavior under no `--verbose`
is equivalent (no output).

**AC-004** (traces to NFR-O-A, SD-003): `--verbose` without `--verbose-bodies` does NOT
log request or response bodies. `--verbose-bodies` enables body logging. This is the
SD-003 behavioral contract already implemented in S-0.06; this story must not regress it.
Test: invoke a command with mock server; assert body does not appear in stderr when only
`--verbose` is used.

**AC-005** (traces to NFR-O-A): `tracing::info!` or `tracing::debug!` events are
emitted at key OAuth flow entry points in `auth.rs` (e.g., "starting OAuth flow",
"exchanging code for token", "refreshing access token"). These events are visible
under `--verbose` and provide actionable context for debugging OAuth failures.

**AC-006** (traces to NFR-O-A): `cargo test --all-features` continues to pass without
test infrastructure changes. The tracing subscriber initializer must be guarded
(idempotent or only initialized in `main`, not in `lib`) to avoid double-init panics in
test mode.

## Holdout Strategy

No Phase 4 holdout is defined for observability infrastructure. Regression protection:
- The existing verbose-body tests from S-0.06 (`tests/verbose_bodies.rs` or equivalent)
  MUST still pass after this story.
- `cargo clippy --all -- -D warnings` MUST pass (tracing macros generate warnings if
  imported but unused).

## Implementation Notes

### Dependency versions to add

```toml
[dependencies]
tracing = "0.1"
tracing-subscriber = { version = "0.3", features = ["env-filter"] }
```

Version pins (use exact versions from the Rust ecosystem at implementation time; do not
use `*`). The `env-filter` feature enables `RUST_LOG`-style filter syntax.

### Subscriber initialization (main.rs)

```rust
// After parsing Cli args, before any command dispatch:
let log_level = if cli.verbose_bodies {
    tracing::Level::TRACE
} else if cli.verbose {
    tracing::Level::DEBUG
} else {
    tracing::Level::WARN
};
tracing_subscriber::fmt()
    .with_max_level(log_level)
    .with_writer(std::io::stderr)  // tracing output to stderr; stdout reserved for data
    .init();
```

### client.rs: replace eprintln! verbose sites

Current pattern (approximate):
```rust
if verbose {
    eprintln!("[verbose] --> {method} {url}");
    eprintln!("[verbose] --> headers: {headers:?}");
}
```

Target pattern:
```rust
tracing::debug!(method = %method, url = %url, "request");
// headers only if TRACE-level
tracing::trace!(headers = ?headers, "request headers");
```

Body logging (already gated by `--verbose-bodies` per S-0.06):
```rust
if verbose_bodies {
    tracing::trace!(body = %body_str, "request body");
}
```

### auth.rs: add key event traces

Add `tracing::info!` at:
- OAuth flow start (`build_authorize_url` entry)
- Token exchange (`exchange_code_for_token` entry and exit)
- Token refresh (`refresh_oauth_token` entry)

These should NOT log client_id/client_secret values. Use structured fields for profile
name and flow type only.

### Test infrastructure guard

The `src/lib.rs` or any test helper that constructs `JiraClient` directly must NOT
call `tracing_subscriber::fmt().init()`. The subscriber is initialized ONLY in
`main.rs`. Tests that want to observe tracing output can set `RUST_LOG=debug` or
call `tracing_subscriber::fmt().try_init().ok()` (ignores double-init error).

## Previous Story Intelligence

Key lesson: the verbose body flag is threaded through `JiraClient` as a field, not as
an env var. When replacing `eprintln!` with `tracing::trace!` for bodies, check that

## Architecture Compliance Rules

- `tracing` subscriber is initialized ONLY in `main.rs`. NOT in `lib.rs` or any test
  helper. Double-init panics are a common mistake with `tracing-subscriber`.
- `tracing` output goes to `stderr`. `println!` for data output is unchanged.
- Structured field names use `%` formatter for Display, `?` formatter for Debug.
  Do not log client_secret, access_token, or refresh_token values under any log level.
- Reference: `architecture/module-decomposition.md` (observability module boundaries).

## Risk / Dependencies

**Note on priority vs. wave placement:** `priority: medium` reflects NFR-O-A's MEDIUM severity
in the NFR catalog. However, this story is placed in Wave 1 (not Wave 2) because S-0.06
(`--verbose-bodies`) depends on a structured logging layer; without `tracing`, SD-003's PII
redaction contract is enforced only by ad-hoc `eprintln!` at scattered sites in `client.rs`.
Placing `tracing` wiring in Wave 2 would leave SD-003's behavioral promise structurally unsound
for the entirety of Wave 1 delivery.

- **Depends on S-0.06**: S-0.06 introduced `--verbose-bodies`; this story replaces the
  underlying `eprintln!` with `tracing`. If S-0.06 has not merged, implement it first.
- **Risk: subscriber double-init.** `tracing_subscriber::fmt().init()` panics if called
  twice in the same process. Guard with `try_init().ok()` in test contexts.
- **Risk: tracing macro expansion.** `tracing` macros expand to inline code; clippy
  may warn on unused imports if macros are the only callers. Ensure `use tracing::{debug, info, trace};`
  imports are present.
- **No breaking change** for end users. `--verbose` behavior is equivalent; output format
  may change slightly (tracing adds timestamp/level prefix by default). Acceptable;
  verbose output is not a stable contract.

## Out of Scope

- Replacing all 47 `eprintln!` sites across all CLI handlers — that is a separate Wave 2
  cleanup story.
- `tracing-opentelemetry` or OTLP export — deferred.
- Metrics or spans (only events/logging in this story).
- `RUST_LOG` env var support — tracing-subscriber's `EnvFilter` enables this automatically
  once the subscriber is initialized; document it but no specific story requirement.
