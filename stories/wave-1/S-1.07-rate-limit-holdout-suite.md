# S-1.07: Rate-limit holdout suite (H-013, H-027)

## Context

The rate-limit retry logic (`src/api/rate_limit.rs` and `src/api/client.rs`) was
not changed by Wave 0 or ADR-0006 embedded OAuth, but it IS invoked by every API
call regardless of auth method. Two holdout scenarios from `holdout-scenarios.md`
pin critical behavior:

- **H-013**: `send_raw` returns 429 to the CALLER (not raising) after `MAX_RETRIES=3`.
  The `send_raw` path must NOT convert 429 into an error — it returns the raw 429
  response. The STANDARD `send` path converts 429 to an error after retries. The
  behavioral contract distinguishes the two paths.

- **H-027**: `RateLimitInfo::from_headers` with `Retry-After: 86400` preserves the
  literal value 86400 without any cap. This pins the KNOWN-GAP behavior: NFR-R-NEW-1
  identifies that a future fix will add `MAX_RETRY_AFTER_SECS = 60`, and when that
  fix lands, H-027 MUST become a MUST-FAIL. Until then, H-027 is MUST-PASS to
  document the current (no-cap) behavior.

These tests are CRITICAL because:
1. ADR-0006 embedded OAuth changes the auth header construction but NOT the retry
   plumbing. A refactor that accidentally changes retry semantics would be invisible
   without these pins.
2. NFR-R-NEW-1 explicitly states that when `MAX_RETRY_AFTER_SECS = 60` is implemented
   in Phase 3, H-027 must be updated to assert `retry_after_secs == 60`. This story
   establishes H-027 as `retry_after_secs == 86400` (current no-cap behavior), creating
   a documented flip point for Phase 3.

## Behavioral Contract

**BC-X.1.005** (`bc-1-auth-identity.md` extended / `bc-2-issue-read.md` cross-ref):
`send_raw` returns 429 status to caller after MAX_RETRIES=3 without raising a Rust error.
Exactly N+1 HTTP calls are made for N retries. Stderr contains the rate-limit warning.

**BC-X.4.002** (`rate_limit.rs` invariant, per holdout-scenarios.md H-027):
`RateLimitInfo::from_headers` with `Retry-After: 86400` returns
`retry_after_secs == 86400`. No clamping. This is the current (KNOWN-GAP) behavior.
When NFR-R-NEW-1 is implemented, this contract changes to BC-X.4.009 (capped at 60).

## Acceptance Criteria

**AC-001** (traces to BC-X.1.005 postcondition): Given a wiremock server that returns
429 with `Retry-After: 0` for ALL requests (no eventual success), when `client.send_raw(GET /myself)`
is called, then:
- The return value is an `Ok(Response)` with status 429 (NOT an `Err`).
- Exactly 4 HTTP calls are made (1 initial + 3 retries = MAX_RETRIES=3).
- Stderr contains the literal text `rate limited by Jira — gave up after 3 retries`.
(Pins H-013.)

**AC-002** (traces to BC-X.4.002 postcondition): Given an `http::HeaderMap` containing
`Retry-After: 86400`, when `RateLimitInfo::from_headers(&headers)` is called, then
`rate_limit_info.retry_after_secs == 86400`. The value is NOT clamped to 60 or any
other maximum. (Pins H-027. KNOWN-GAP: this assertion MUST flip to `== 60` when
NFR-R-NEW-1 is implemented.)

> **Flip point:** When NFR-R-NEW-1 is implemented in S-3.07 (Wave 3), update H-027 from
> KNOWN-GAP to MUST-PASS-AFTER-FIX. The test assertion `retry_after_secs == 86400` flips to
> `retry_after_secs == 60`. S-1.07 only pins the pre-fix KNOWN-GAP behavior; S-3.07 owns the
> implementation and the NFR-R-NEW-1 anchor.

**AC-003** (traces to BC-X.1.005 postcondition): Given a wiremock server that returns
429 for the first 2 requests and then 200, when `client.send(GET /myself)` is called
(NOT `send_raw`), then:
- The return value is an `Ok(Response)` with status 200.
- Exactly 3 HTTP calls are made.
- Stderr MAY contain rate-limit messages.
(Regression guard: verify retry DOES succeed on the standard `send` path when the
server eventually responds 200.)

**AC-004** (traces to BC-X.4.002 invariant): `RateLimitInfo::from_headers` with a
`Retry-After: 0` header returns `retry_after_secs == 0`. Zero is a valid value (no
delay before retry). (Ensures the parser accepts boundary value 0.)

**AC-005** (traces to BC-X.4.002 invariant): `RateLimitInfo::from_headers` with a
`Retry-After: abc` (non-integer) header falls back to `DEFAULT_RETRY_SECS = 1`.
(Per NFR-SCA-1: HTTP-date format silently falls through to default; any non-integer
falls through.)

## Holdout Strategy

- **H-013**: MUST-PASS against activation HEAD dea1664. If it fails, there is a
  pre-existing bug in `send_raw` semantics. Fix the bug before marking this story done.
- **H-027**: MUST-PASS against activation HEAD dea1664 with assertion
  `retry_after_secs == 86400`. Add a prominent comment:
  ```rust
  // KNOWN-GAP: H-027. When NFR-R-NEW-1 (MAX_RETRY_AFTER_SECS=60) is implemented,
  // change this assertion to: assert_eq!(rate_limit_info.retry_after_secs, 60);
  // and update the holdout status from MUST-PASS to MUST-FAIL-BEFORE-FIX / MUST-PASS-AFTER-FIX.
  ```
- Tests are unit tests (H-027) and integration tests with wiremock (H-013, AC-003).
- NOT `#[ignore]`-gated. Both must run in standard `cargo test`.

## Implementation Notes

### H-013 test structure (wiremock)

```rust
#[tokio::test]
async fn test_send_raw_returns_429_after_max_retries() {
    let server = MockServer::start().await;
    // Mount: GET /rest/api/3/myself → 429 with Retry-After: 0, expect(4)
    Mock::given(method("GET"))
        .and(path("/rest/api/3/myself"))
        .respond_with(ResponseTemplate::new(429).insert_header("Retry-After", "0"))
        .expect(4)  // 1 initial + 3 retries
        .mount(&server)
        .await;
    
    let client = JiraClient::new_for_test(&server.uri(), "Bearer test");
    let response = client.send_raw(/* GET /myself */).await;
    
    assert!(response.is_ok(), "send_raw must return Ok, not Err");
    assert_eq!(response.unwrap().status(), 429);
    server.verify().await;  // confirms exactly 4 calls
}
```

### H-027 test structure (unit test, no wiremock)

```rust
#[test]
fn test_rate_limit_from_headers_preserves_86400_no_cap() {
    use http::HeaderMap;
    let mut headers = HeaderMap::new();
    headers.insert("Retry-After", "86400".parse().unwrap());
    
    let info = RateLimitInfo::from_headers(&headers);
    
    // KNOWN-GAP: H-027. When NFR-R-NEW-1 cap is implemented, change to: assert_eq!(info.retry_after_secs, 60);
    assert_eq!(info.retry_after_secs, 86400,
        "H-027: RateLimitInfo must preserve Retry-After value without upper-bound cap (KNOWN-GAP until NFR-R-NEW-1 lands)");
}
```

### Import path for RateLimitInfo

`RateLimitInfo::from_headers` is in `src/api/rate_limit.rs`. Check if it is `pub` or
`pub(crate)`. If `pub(crate)`, the test must be an `src/api/rate_limit.rs` inline test
(within the module), not an external integration test. Move the H-027 test inline if needed.

Alternative: if `RateLimitInfo` and `from_headers` are re-exported via `src/lib.rs`,
the test can be in `tests/rate_limit_holdouts.rs` without change. Verify before writing.

## Architecture Compliance Rules

- `RateLimitInfo` is an internal type in `src/api/rate_limit.rs`. If it is `pub(crate)`,
  the H-027 unit test MUST be an inline test in `rate_limit.rs`, not in `tests/`.
- Do not change visibility of `RateLimitInfo` or `from_headers` for this story. If a
  visibility change is needed, flag it as a separate refactor.
- `send_raw` semantics: returns raw response on ANY status code, including 429. Do NOT
  change `send_raw` to raise on 429 — that would break the `jr api` passthrough command.
- Reference architecture: `architecture/module-decomposition.md` (api/rate_limit.rs
  and api/client.rs components).

## Risk / Dependencies

- **No Wave 0 dependencies.** Independent of S-0.01..S-0.07.
- **Risk: `RateLimitInfo` visibility.** If `pub(crate)`, H-027 must be an inline test.
  This is a mechanical adaptation, not a scope change.
- **Risk: MAX_RETRIES value.** H-013 asserts exactly 4 calls. Verify `MAX_RETRIES = 3`
  in `client.rs` before writing the test. If the constant changes, the test must change.
- **Known flip point.** When NFR-R-NEW-1 is implemented (Phase 3 or Wave 2), H-027
  MUST be updated. The comment in the test serves as the flip-point contract.
- **No breaking change** for end users.
