# S-1.06: OAuth flow regression holdout suite (H-001..H-008 consolidated)

## Context

The OAuth authentication flow (ADR-0006, ADR-0013) is the most security-critical code
path in `jr`. It was significantly reworked to embed the `jr` OAuth app credentials.
Phase 1d convergence (Pass 28) identified that H-001 through H-008 represent
MUST-PASS behavioral contracts for the auth flow, but no consolidated integration test
file exercises all of them together.

This story writes a single test file `tests/oauth_flow_holdouts.rs` that pins the
following holdout scenarios as MUST-PASS tests:

- **H-001**: `auth status` on fresh install exits 0 with "No profiles configured"
- **H-002**: `auth list --output json` on fresh install returns `[]`
- **H-003**: Profile precedence — flag > env > config > "default"
- **H-004**: `auth refresh --no-input` on unconfigured profile exits 64 with actionable error
- **H-005**: Malformed config TOML → exit 78, file NOT overwritten
- **H-006**: `issue move` idempotent when already in target state (uses auth fixture; OAuth client dispatches correctly)
- **H-022**: 401 scope-mismatch dispatch boundary (case sensitivity, status gate, substring)
- **H-029**: BYO OAuth uses dynamic port; embedded uses fixed port 53682

These scenarios are chosen because ADR-0006 (embedded OAuth) and SD-002 (`JR_AUTH_HEADER`
`#[cfg(test)]` gate from S-0.05) created behavioral boundaries that are easy to silently
regress. A future refactor of `auth.rs` or `config.rs` that breaks any of these would
cause silent regressions in the onboarding flow and multi-profile setup.

H-006 is included to verify that the auth client (now using embedded OAuth) dispatches
correctly to the idempotent `issue move` path.

## Behavioral Contract

**BC-1.1.001**: When no `~/.config/jr/config.toml` exists, `jr auth list --output json`
exits 0 and stdout is `[]`.

**BC-1.1.002**: `jr auth status` against an uninitialized config exits 0 and prints
`No profiles configured` to stderr.

Additional BCs exercised (not enumerated in frontmatter per story constraint of ≤2 BCs):
- BC-1.1.003 (`auth switch <unknown>` exits 64)
- BC-1.1.007 (precedence chain)
- BC-1.1.011 (`auth refresh --no-input` no-URL exits 64 with actionable message)
- BC-1.1.012 (malformed config → exit 78, no overwrite)
- BC-1.6.043 / BC-1.6.044 / BC-1.6.045 (401 dispatch boundaries)
- BC-1.5.031 / BC-1.5.034 (embedded vs BYO callback URL)

## Acceptance Criteria

**AC-001** (traces to BC-1.1.001, BC-1.1.002 postcondition): Given an empty
`XDG_CONFIG_HOME`, when `jr auth list --output json` is invoked, then exit code is 0
and stdout equals `[]`. When `jr auth status` is invoked, then exit code is 0 and
stderr contains `No profiles configured`. (Pins H-001, H-002.)

**AC-002** (traces to BC-1.1.007 postcondition): Given config with three profiles
`from-config / from-env / from-flag` and `default_profile = "from-config"`, when
`JR_PROFILE=from-env` is set and `jr --profile from-flag auth list --output json`
is invoked, then exactly one element has `"active": true` with `"name": "from-flag"`.
(Pins H-003.)

**AC-003** (traces to BC-1.1.011 postcondition): Given no profile URL configured and
`--no-input`, when `jr auth refresh` is invoked, then exit code is 64 and stderr
contains `no URL configured`, `jr auth login`, and `--url`. Stderr does NOT contain
`panic`. (Pins H-004.)

**AC-004** (traces to BC-1.1.012 postcondition): Given a malformed TOML at
`XDG_CONFIG_HOME/jr/config.toml`, when `jr auth login --oauth --client-id X --client-secret Y --no-input`
is invoked, then exit code is 78, stderr contains `toml` or `parse`, and the on-disk
file bytes are unchanged. (Pins H-005.)

**AC-005** (traces to BC-1.6.043, BC-1.6.044, BC-1.6.045 postconditions): Given four
wiremock fixtures returning (a) 401 + `scope does not match`, (b) 401 + `Scope Does Not Match`,
(c) 401 + `Session expired`, (d) 403 + `scope does not match policy`, when each is
invoked via a command that triggers an API call, then:
- (a) and (b) → `InsufficientScope` exit code 2 with `Insufficient token scope` in stderr
- (c) → `Not authenticated` in stderr (NOT `Insufficient token scope`)
- (d) → `API error (403)` exit code 1 (NOT `InsufficientScope`)
(Pins H-022.)

**AC-006** (traces to BC-1.5.031, BC-1.5.034 postconditions): Given embedded OAuth app
present in the binary, `build_authorize_url` produces a redirect_uri of exactly
`http://127.0.0.1:53682/callback`. Given BYO credentials (`--client-id X --client-secret Y`),
`build_authorize_url` produces a redirect_uri of `http://localhost:<dynamic_port>/callback`
where `<dynamic_port>` is NOT 53682. (Pins H-029.)

## Holdout Strategy

All holdouts in this story are MUST-PASS from activation HEAD. They do not require
code changes — they are behavioral pins on existing correct behavior. If any test in
this suite fails against the current codebase, that indicates a pre-existing regression
that must be fixed before this story is marked done.

Verification:
- `cargo test --test oauth_flow_holdouts` passes with zero failures.
- Tests are NOT `#[ignore]`-gated (they use `JR_BASE_URL` + wiremock, not real keychain).
- Exception: tests that exercise real keychain (H-005 malformed config) use `temp_dir()`
  for `XDG_CONFIG_HOME` isolation and do NOT touch the developer's actual keychain.

## Implementation Notes

### Test infrastructure pattern

All tests in this file use:
- `tempfile::TempDir` for `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` isolation.
- `assert_cmd::Command::cargo_bin("jr")` for process-spawn style tests (most holdouts).
- `wiremock::MockServer` + `JR_BASE_URL` for HTTP-dependent tests.
- `JR_SERVICE_NAME=jr-jira-cli-test` for keychain isolation (where applicable).

### H-029 (embedded vs BYO callback URL)

H-029 cannot be tested via process-spawn without a real browser interaction. Instead:
- Test via unit/library-level call to `build_authorize_url` with a mock
  `EmbeddedOAuthApp` present vs. a BYO `client_id`/`client_secret`.
- Assert the `redirect_uri` field in the returned authorize URL string.
- No actual OAuth flow is exercised (no browser, no token exchange).

This is a `JiraClient::new_for_test`-style test at the library level, not a full
process-spawn test.

### H-022 (401 dispatch)

Use `wiremock::MockServer` and the `JiaClient::new_for_test` constructor directly:
```rust
let server = MockServer::start().await;
// Mount 401 with scope-mismatch body
// Call client.send(GET /myself).await
// Assert Err(JrError::InsufficientScope)
```

No process-spawn needed; test at the `JiraClient` level directly.

### Grouping

Tests for H-001/H-002 are simple process-spawn tests. Tests for H-003/H-007 require
config manipulation. Tests for H-022 require wiremock. Tests for H-029 require
library-level access. These can be in different `#[tokio::test]` functions within
the same file.

## Architecture Compliance Rules

- Tests use `assert_cmd` for process-level tests and `wiremock` for HTTP mocking.
- `JR_SERVICE_NAME=jr-jira-cli-test` is set for all keychain-adjacent tests to prevent
  keychain pollution in the developer's environment.
- `JR_BASE_URL` points to the local mock server; no real Jira instance is contacted.
- Tests must NOT depend on the developer's `~/.config/jr/` directory. All config is
  in temp dirs.
- Test file imports: `use assert_cmd::prelude::*; use wiremock::*;` — no new crates
  beyond what is already in `[dev-dependencies]`.
- Reference: `architecture/module-decomposition.md` for auth module boundaries.

## Risk / Dependencies

- **Depends on S-0.05**: `JR_AUTH_HEADER` is `#[cfg(test)]` gated in S-0.05. Tests
  in this suite use `JR_BASE_URL` + wiremock, which is the correct test pattern.
  No `JR_AUTH_HEADER` usage in this test file.
- **Risk: H-029 embedded OAuth app absent in dev builds.** `embedded_oauth_app()` returns
  `None` when `JR_BUILD_OAUTH_CLIENT_ID` is not set at build time (BC-1.3.020). The
  H-029 test for the embedded callback URL must check `embedded_oauth_app().is_some()`
  first; if None, skip the embedded-URL assertion and only test the BYO path.
  Use `#[cfg(all(...))]` or a runtime skip with `println!("skip: no embedded app")`.
- **Risk: test isolation.** If any test modifies `XDG_CONFIG_HOME` without cleanup,
  subsequent tests in the same binary may see stale state. Mitigate with `TempDir`
  per test function.
- **No breaking change** for end users.

## Out of Scope

  not an auth flow test; it belongs in `tests/issue_workflow_holdouts.rs` (a separate story).

  gate (BC-1.3.024).
