# S-1.08: Keychain per-profile layout holdout suite (H-016 + per-profile key invariants)

## Context

CLAUDE.md "Gotchas" identifies the multi-profile boundary as one of the most fragile
correctness invariants in `jr`:

> Per-profile vs shared OAuth keys: `email`, `api-token`, `oauth_client_id`,
> `oauth_client_secret` live under flat keychain keys. `oauth-access-token` /
> `oauth-refresh-token` are namespaced as `<profile>:oauth-*` because they're
> cloudId-scoped. The `"default"` profile lazy-migrates legacy flat keys on first read;
> other profiles do not.

This story pins the keychain layout contracts as MUST-PASS holdout tests:

- **H-016** (`auth remove <active>` rejected): Destructive operation safety. Removing the
  active profile must exit 64 with `cannot remove active` and leave the config file unchanged.

Additionally, this story writes **non-keyring tests** that pin the behavioral logic of:
- **BC-1.4.025** (lazy migration is `default`-only; non-default profiles never inherit legacy keys)
- **BC-1.4.027** (namespaced key format: `<profile>:oauth-access-token`)
- **BC-1.4.028** (partial state — one token present, other missing — returns error)
- **BC-1.4.029** (`load_oauth_tokens("sandbox")` does NOT inherit legacy flat keys)
- **BC-1.4.030** (`resolve_refresh_app_credentials` prefers KEYCHAIN over EMBEDDED)

These are NOT full keychain round-trip tests (those require `JR_RUN_KEYRING_TESTS=1`
and a real secret-service daemon). Instead, this story tests the KEY NAMING LOGIC
and PROFILE BOUNDARY LOGIC at the unit level, using mock keychain backends where
the keychain trait allows injection, or by testing the string constants and branching
logic directly.

The goal is to pin the invariants that prevent cross-profile credential leakage
(sandbox vs prod custom-field IDs can differ — a correctness bug, per CLAUDE.md).

## Behavioral Contract

**BC-1.4.027**: All OAuth token storage/retrieval uses namespaced keys of the form
`<profile>:oauth-access-token` and `<profile>:oauth-refresh-token`. Shared keys
(`email`, `api-token`, `oauth_client_id`, `oauth_client_secret`) are NOT namespaced.

**BC-1.4.025**: `load_oauth_tokens(profile)` lazy-migrates legacy flat keys ONLY when
`profile == "default"`. For `profile == "sandbox"` or any other non-default profile,
legacy migration is skipped. Two `if profile == "default"` guards enforce this at
`src/api/auth.rs:124` and `src/api/auth.rs:151`.

## Acceptance Criteria

**AC-001** (traces to BC-1.4.027 postcondition): The keychain key names for OAuth
tokens follow the pattern `<profile>:oauth-access-token` and `<profile>:oauth-refresh-token`.
For profile `"default"`: keys are `default:oauth-access-token` and `default:oauth-refresh-token`.
For profile `"sandbox"`: keys are `sandbox:oauth-access-token` and `sandbox:oauth-refresh-token`.
Verified by inspecting the key-name constants/functions in `src/api/auth.rs`.

**AC-002** (traces to BC-1.4.025 postcondition): Given a config with profile `"sandbox"`,
when `load_oauth_tokens("sandbox")` is called with NO namespaced keys present AND
legacy flat `oauth-access-token` key present (simulating old format), then the function
returns an `Err` (not the legacy flat-key value). The `if profile == "default"` guard
prevents non-default profiles from reading legacy keys.

**AC-003** (traces to BC-1.4.028 postcondition): Given that `<profile>:oauth-access-token`
is present but `<profile>:oauth-refresh-token` is absent, when `load_oauth_tokens(profile)`
is called, then an `Err` is returned. Partial credential state is never silently used.

**AC-004** (traces to BC-1.4.029 postcondition): Given profile `"sandbox"`,
`load_oauth_tokens("sandbox")` reads only `sandbox:oauth-access-token` and
`sandbox:oauth-refresh-token`. It does NOT read legacy flat `oauth-access-token`
(the default profile's legacy key). Verified via mock keychain or code inspection test.

**AC-005** (traces to BC-1.4.030 postcondition): `resolve_refresh_app_credentials`
checks keychain FIRST. If keychain has `oauth_client_id` / `oauth_client_secret`
(BYO credentials), it returns those. The embedded OAuth app is NOT used when BYO
credentials are present in keychain. Keychain priority over embedded is enforced.

**AC-006** (traces to BC-1.1.006 postcondition, H-016): Given config with
`default_profile = "default"` and `[profiles.default]`, when `jr --no-input auth remove default`
is invoked via process-spawn, then exit code is 64, stderr contains `cannot remove active`,
and the config file on disk is byte-identical to before the command. (Pins H-016.)

## Holdout Strategy

- **H-016** is MUST-PASS against activation HEAD dea1664. It is a safety check on
  destructive operation guard logic.
- BC-1.4.025 through BC-1.4.030 unit tests are MUST-PASS (pin existing correct behavior).
- None of these tests use a real keychain service. They use:
  - Process-spawn + `XDG_CONFIG_HOME` temp dir (H-016 / AC-006)
  - Unit tests on the key-naming constants and branching logic (AC-001..AC-005)
- Tests gated behind `JR_RUN_KEYRING_TESTS=1` are NOT added in this story. Those
  test the OS keychain integration layer; the tests here are at the logic layer.

## Implementation Notes

### H-016 test (process-spawn)

```rust
#[test]
fn test_auth_remove_active_profile_rejected() {
    let config_dir = tempfile::tempdir().unwrap();
    // Write config with [profiles.default] and default_profile = "default"
    let config_content = r#"
        default_profile = "default"
        [profiles.default]
        url = "https://example.atlassian.net"
        auth_method = "token"
    "#;
    let config_path = config_dir.path().join("jr/config.toml");
    std::fs::create_dir_all(config_path.parent().unwrap()).unwrap();
    std::fs::write(&config_path, config_content).unwrap();
    let original_bytes = std::fs::read(&config_path).unwrap();
    
    let output = assert_cmd::Command::cargo_bin("jr")
        .unwrap()
        .env("XDG_CONFIG_HOME", config_dir.path())
        .env("JR_SERVICE_NAME", "jr-jira-cli-test")
        .args(["--no-input", "auth", "remove", "default"])
        .output()
        .unwrap();
    
    assert_eq!(output.status.code(), Some(64));
    let stderr = String::from_utf8_lossy(&output.stderr);
    assert!(stderr.contains("cannot remove active"), "stderr: {}", stderr);
    // File bytes unchanged
    assert_eq!(std::fs::read(&config_path).unwrap(), original_bytes);
}
```

### AC-001..AC-005 (unit tests for key naming logic)

These tests inspect `src/api/auth.rs` constants and function logic. If `auth.rs`
exports key-name functions (e.g., `fn oauth_access_token_key(profile: &str) -> String`),
test them directly:

```rust
#[cfg(test)]
mod tests {
    use super::*;
    
    #[test]
    fn test_oauth_key_names_are_namespaced() {
        // Adjust to actual function names in auth.rs
        assert_eq!(oauth_token_key("default", "access"), "default:oauth-access-token");
        assert_eq!(oauth_token_key("sandbox", "access"), "sandbox:oauth-access-token");
        assert_eq!(oauth_token_key("default", "refresh"), "default:oauth-refresh-token");
    }
    
    #[test]
    fn test_shared_keys_are_not_namespaced() {
        // email, api-token, oauth_client_id, oauth_client_secret are flat
        // Verify by reading the relevant constants in auth.rs
        assert_eq!(SHARED_EMAIL_KEY, "email");  // or whatever the constant is
        assert_ne!(SHARED_EMAIL_KEY, "default:email");
    }
}
```

If the key names are constructed inline (no dedicated function), add a helper function
`fn token_key(profile: &str, kind: &str) -> String` and test it. The implementer
should read `src/api/auth.rs:24-32` (per BC-1.4.027 source annotation) to find the
exact construction site.

### AC-002 / AC-003 (partial state and non-default guard)

These test `load_oauth_tokens` behavior which requires keychain reads. If
`load_oauth_tokens` uses a trait-injectable keychain, inject a mock. If it uses
the OS keychain directly, these tests may need to be inline `#[cfg(test)]` tests
with `JR_SERVICE_NAME` isolation.

Given the complexity of injecting keychain mocks without a trait seam, an acceptable
alternative for AC-002 through AC-005 is:
- **Code-path inspection tests**: Test that the `if profile == "default"` guard exists
  and that the branching logic produces the correct key names without actually invoking
  the OS keychain. These can be tests on private functions within the auth module using
  `#[cfg(test)] pub` visibility promotion.

Document any such visibility promotion clearly: `// test-only pub: see keychain_layout_holdouts.rs`.

## Test Plan

**Red gate**: `cargo test --test keychain_layout_holdouts` fails (file doesn't exist).

**Implementation order**:
1. Write H-016 test (AC-006) — simplest, process-spawn with temp dir.
2. Write key-name unit tests (AC-001) — inspect constants/functions in auth.rs.
3. Write guard-logic tests (AC-002, AC-004) — may require inline tests or test-pub.
4. Write partial-state test (AC-003) — depends on keychain abstraction.
5. Write resolve-refresh-app test (AC-005) — at the `auth_embedded.rs` function level.
6. Run full suite: `cargo test --test keychain_layout_holdouts && cargo test --lib auth`.

**Green gate**: All tests pass without code changes (pinning existing correct behavior).

If any test FAILS, it indicates a pre-existing bug in the keychain layout logic.
Investigate and fix before marking done.

## Architecture Compliance Rules

- Tests use `JR_SERVICE_NAME=jr-jira-cli-test` for ALL process-spawn tests to prevent
  keychain pollution in the developer's OS keychain.
- No test touches `~/.config/jr/` or `~/.cache/jr/` — all config/cache is in temp dirs.
- `#[cfg(test)] pub` visibility promotions in `src/api/auth.rs` are the ONLY acceptable
  source changes in this story. If more invasive changes are needed, escalate.
- Key naming in `auth.rs` must NOT change as a side effect of this story. Tests verify
  existing behavior; implementation adapts tests to find the correct internal functions.
- Reference: `architecture/module-decomposition.md` (auth module boundaries).

## Risk / Dependencies

- **Depends on S-0.05**: `JR_AUTH_HEADER` is test-only. Tests use `JR_BASE_URL` +
  wiremock, not `JR_AUTH_HEADER`.
- **Risk: keychain trait seam absent.** If `load_oauth_tokens` cannot be tested without
  a real OS keychain, the AC-002/AC-003/AC-004 tests may need to be placed inline in
  the `auth.rs` `#[cfg(test)]` module rather than in `tests/`. This is acceptable.
- **Risk: `JR_SERVICE_NAME` not implemented.** If this env var doesn't exist or is
  not honored, H-016 tests could pollute the developer's keychain. Verify the env var
  is respected before writing keychain-touching tests. If not, confine to logic-only
  tests (key-name and guard-logic tests, not actual write/read round-trips).
- **No breaking change** for end users.

## Out of Scope

  secret-service; they are already `#[ignore]`-gated in the existing test suite).
- Testing `auth login` flow end-to-end (that is H-009 / H-017 territory; separate story).
- Multi-profile config migration (covered by S-0.04 / BC-6.3.001).
