# S-1.05: Enable GitHub secret scanning and add gitleaks CI step

## Context

No secrets scanning is configured for the repository (cicd-setup.md GAP-3). The project
handles OAuth tokens (`JR_BUILD_OAUTH_CLIENT_ID`, `JR_BUILD_OAUTH_CLIENT_SECRET`) and
API tokens that could accidentally be committed. NFR-S-B (HIGH) identifies
`JR_AUTH_HEADER` as a pattern that could accidentally leak credentials if a developer
commits a `.env` file or shell history.

This story addresses the gap with two complementary mechanisms:

1. **GitHub repository-level secret scanning** (Settings > Security > Secret scanning):
   GitHub scans all commits for known credential patterns (OAuth tokens, API keys, etc.)
   and alerts the repository owner. This is a repository settings toggle — no code change.

2. **gitleaks CI step**: Add `gitleaks/gitleaks-action` as a step in a new `security` job
   in `ci.yml`. Gitleaks scans the repository history on every PR for leaked secrets,
   catching patterns that GitHub's built-in scanning may miss (e.g., custom token formats).

The goal is defense-in-depth: GitHub's native scanning catches known patterns immediately
at push; gitleaks provides a CI gate that blocks PRs containing newly-committed secrets.

## Behavioral Contract

No BC anchor. Anchored to cicd-setup.md GAP-3 / R-L13 / NFR-S-B. The ACs define
measurable target state.

## Acceptance Criteria

**AC-001** (traces to cicd-setup.md GAP-3, NFR-S-B): GitHub repository-level secret
scanning is enabled. Verification: `gh api repos/Zious11/jira-cli/vulnerability-alerts`
returns HTTP 204 (enabled) or equivalent. Instructions for enabling via UI are
documented in this story's implementation notes (settings toggle requires maintainer
access; the CI step alone is not sufficient).

**AC-002** (traces to cicd-setup.md GAP-3): `ci.yml` contains a `security` job that
runs `gitleaks/gitleaks-action` (pinned to SHA per S-1.01 policy). This job runs on
`pull_request` events only (no need to run on `push` to `develop`/`main` as GitHub
native scanning covers post-merge).

**AC-003** (traces to cicd-setup.md GAP-3): The `security` job has a
`timeout-minutes: 10` (per S-1.04 policy). It does NOT block other CI jobs — it runs
in parallel with `fmt`, `clippy`, etc.

**AC-004** (traces to cicd-setup.md GAP-3): A `.gitleaks.toml` configuration file is
added to the repository root. It defines at minimum:
- Title and version fields (required by gitleaks format).
- An `[[allowlist.regexes]]` entry for the embedded OAuth XOR-obfuscated byte arrays
  in `src/api/auth_embedded.rs` (if gitleaks flags those as secrets — they are not real
  credentials but the pattern may match). This prevents false positives from blocking CI.

**AC-005** (traces to NFR-S-B, SD-002): The `gitleaks` step does NOT require or read
`GITHUB_TOKEN` beyond the minimal permissions needed for PR checkout. The job permissions
block is set to `contents: read` (read-only for PR checkout).

## Implementation Notes

### Step 1: Enable GitHub secret scanning (repository settings)

This requires maintainer access to the repository and cannot be done via workflow files.

Instructions:
1. Go to `https://github.com/Zious11/jira-cli/settings/security_analysis`
2. Enable "Secret scanning" (the toggle should be in the "Security" section).
3. Optionally enable "Push protection" to block pushes containing known secrets.

Document completion by adding a comment to the story or a commit note.

### Step 2: Add gitleaks CI job

```yaml
# In .github/workflows/ci.yml, add as a new job:
  security:
    name: Secret Scanning (gitleaks)
    runs-on: ubuntu-latest
    timeout-minutes: 10
    permissions:
      contents: read
    steps:
      - uses: actions/checkout@<sha>  # v4.2.2 (use SHA per S-1.01)
        with:
          fetch-depth: 0  # full history for gitleaks scan
      - uses: gitleaks/gitleaks-action@<sha>  # v2.x (pin SHA)
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

Note: `fetch-depth: 0` is required for gitleaks to scan history. The default
shallow clone (`fetch-depth: 1`) only scans the latest commit.

### Step 3: Add .gitleaks.toml

```toml
title = "jr gitleaks config"
version = 8

[[allowlist.regexes]]
description = "Embedded OAuth XOR-obfuscated byte array — not a real credential"
regex = '''EMBEDDED_ID|EMBEDDED_SECRET|XOR_KEY'''

[[allowlist.paths]]
description = "Test fixtures — mock credentials only"
regex = '''tests/common/fixtures\.rs'''
```

Adjust regex to match the actual obfuscated variable names in `src/api/auth_embedded.rs`.

### Gitleaks action SHA

Look up the current SHA for `gitleaks/gitleaks-action@v2` from
`https://github.com/gitleaks/gitleaks-action/tags`. Pin it per S-1.01 policy.

## Architecture Compliance Rules

- The `security` job runs in parallel with other CI jobs; it is NOT a gate for other
  jobs (no `needs:` pointing to it from `test` or `clippy`).
- Exception: if the team decides gitleaks should be a hard PR gate, add it to the
  branch protection required status checks via repository settings. Document this decision.
- No source code changes. No Cargo.toml changes.

## Library and Framework Requirements

No Rust dependencies. Gitleaks is a standalone Go binary executed via the GitHub Action.
`gitleaks` can optionally be installed locally for pre-push validation:
```bash
brew install gitleaks
gitleaks detect --source . --config .gitleaks.toml
```

## Risk / Dependencies

- **No dependencies** on other Wave 1 stories (though S-1.01 SHA pinning policy applies
  to the gitleaks action reference).
- **Risk: false positives.** Gitleaks may flag test fixtures, embedded OAuth byte arrays,
  or other non-secret patterns. Mitigate with `.gitleaks.toml` allowlist entries.
- **Risk: GitHub secret scanning toggle.** Enabling via repository settings requires
  maintainer access and cannot be automated via workflow files. If the maintainer cannot
  enable it, this story is partially complete (gitleaks CI step covers the same content
  but without push-protection).
- **No breaking change** for end users.
- **Risk: scan duration.** With `fetch-depth: 0`, gitleaks scans full history.
  For a repository with many commits, this could be slow. The 10-minute timeout
  (AC-003) provides a safety net. If scans consistently take >5 minutes, consider
  caching the gitleaks database.

## Out of Scope

- CodeQL or Semgrep static analysis — separate security concern, not secret scanning.
- Trivy or Snyk container scanning — not applicable (no Docker image produced).
- Signing the gitleaks action itself — the SHA pin (per S-1.01) provides integrity.
- Enabling Dependabot security alerts (likely already on by default for public repos).
