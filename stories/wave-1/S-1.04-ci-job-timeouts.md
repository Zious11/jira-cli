# S-1.04: Add `timeout-minutes` to all GitHub Actions jobs

## Context

No `timeout-minutes:` is set on any job in `ci.yml` or `release.yml` (cicd-setup.md
GAP-2). Without timeouts, a hung build — e.g., a wiremock integration test waiting
for a port that never opens, or a cross-compilation that deadlocks — will consume
the full GitHub Actions default billing window (6 hours) before failing.

The release workflow is particularly exposed because it runs a 4-target cross-compilation
matrix. A single timeout-runaway in release builds costs 24 runner-hours per incident.
Additionally, `cargo install cross --git https://...` in `release.yml` involves a
network fetch with no timeout guard.

This is classified R-L12 (LOW severity) in the risk register but is a zero-effort
security-posture and cost-control improvement. It is included in Wave 1 because it
takes under 30 minutes to implement and has no risk of regression.

## Behavioral Contract

No BC anchor. Anchored to cicd-setup.md GAP-2 / R-L12. The ACs define target state.

## Acceptance Criteria

**AC-001** (traces to cicd-setup.md GAP-2): Every job in `ci.yml` has a
`timeout-minutes:` field. Recommended values:
- `fmt`: 5 minutes (format check is fast)
- `clippy`: 15 minutes (lint; incremental is fast, cold is slower)
- `test`: 30 minutes (unit + integration on both ubuntu and macos)
- `msrv`: 15 minutes (cargo check only, no linking)
- `deny`: 10 minutes (license + advisory check)
- `coverage`: 30 minutes (full instrumented build + upload)

**AC-002** (traces to cicd-setup.md GAP-2): Every job in `release.yml` has a
`timeout-minutes:` field. Recommended values:
- `build` jobs: 60 minutes (cross-compilation for aarch64 targets is slow)
- `release` job: 30 minutes (artifact download + GitHub release creation)

**AC-003** (traces to cicd-setup.md GAP-2): No job has `timeout-minutes: 0`
(GitHub's unlimited). All timeouts are positive integers within the recommended
ranges. The specific values may be adjusted based on observed build times, but
MUST be set (not absent).

**AC-004** (traces to cicd-setup.md GAP-2): After this story merges, `grep -L
'timeout-minutes' .github/workflows/*.yml` returns no files — every workflow file
contains at least one `timeout-minutes` reference.

## Holdout Strategy

No Phase 4 holdout. Verification:
- `grep -c 'timeout-minutes' .github/workflows/ci.yml` returns N (equal to job count).
- `grep -c 'timeout-minutes' .github/workflows/release.yml` returns M (equal to job count).
- CI continues to pass after adding timeouts (no job exceeds its timeout in normal operation).

## Implementation Notes

### ci.yml job-level timeout format

```yaml
jobs:
  fmt:
    name: Format
    runs-on: ubuntu-latest
    timeout-minutes: 5
    steps:
      - uses: actions/checkout@<sha>  # v4.2.2
      - run: cargo fmt --all -- --check
```

`timeout-minutes` is placed immediately after `runs-on` at the job level, not the step
level. Step-level timeouts are also available but job-level is sufficient here.

### release.yml: adjust for cross-compilation

`aarch64-unknown-linux-gnu` builds via `cross` are typically 2-3x slower than native.
Set `build` matrix job timeout at 60 minutes to accommodate the slowest target without
false-positive timeouts on a warm CI machine.

### Choosing timeout values

If build times are unknown, start with the conservative values in AC-001/AC-002.
After the first few CI runs, adjust downward to 2x the observed P95 build time.
Over-cautious timeouts (e.g., 60 min for fmt) are fine initially; the important
property is that the field is SET.

## Risk / Dependencies

- **No dependencies.** Independent of all other Wave 1 stories (though it naturally
  pairs with S-1.01 since both touch the same YAML files).
- **Risk: false-positive timeout.** If a CI machine is slow (e.g., GitHub-hosted runner
  under load), a tight timeout could fail a legitimate run. Set conservative values
  initially; tighten after observing actual runtimes.
- **No breaking change** for end users.

## Out of Scope

- Step-level timeouts within jobs.
- Adding `cancel-in-progress: true` concurrency settings (separate concern).
- Changing any job's runner, matrix, or steps.
