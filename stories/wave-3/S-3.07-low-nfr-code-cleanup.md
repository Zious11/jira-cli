# S-3.07: Consolidated LOW NFR code fixes: Retry-After cap, parse_duration overflow, profile name error, anti-loop guard

> **Sizing justification**: This story bundles 4 unrelated ≤10-LOC fixes (Parts A/B/C/D).
> Bundling is deliberate — per-story overhead (wave slot, PR, review cycle) would exceed the work
> for any individual part. If parallelization is needed at implementation time, the implementer
> may split into S-3.07a/b/c/d sub-shards with explicit coordination; otherwise deliver as one PR.

## Context

This story consolidates four LOW-severity NFRs that each require a small code change (≤10 LOC each)
but are too small to warrant individual stories. Grouping them achieves closure without story sprawl.

### Part A — NFR-R-NEW-1: Retry-After cap (BC-X.4.009)

`src/api/rate_limit.rs:14-19` honors any integer `Retry-After` value without an upper bound.
`Retry-After: 86400` (24 hours) causes the retry loop to sleep for 24 hours. While the user can
Ctrl+C, this is a surprising behavior. BC-X.4.009 proposes `MAX_RETRY_AFTER_SECS = 60` cap.

**Fix**: Add `const MAX_RETRY_AFTER_SECS: u64 = 60;`. If the parsed value exceeds the cap,
print a warning to stderr and abort retry (return error, not sleep).

H-027 currently pins the gap (KNOWN-GAP: `Retry-After: 86400` sleeps for 24 hours). After this
fix, H-027 must transition to MUST-PASS.

### Part B — NFR-R-NEW-2: parse_duration overflow guard

`src/duration.rs:29-32` uses arithmetic multiplication without overflow checks. For pathological
inputs like `99999999999999w`, a release build (with `panic=abort`) silently wraps. Fix: use
`checked_mul` with a human-readable error ("duration too large").

This is ~5 LOC. No new behavioral contract needed — an overflow produces an immediate `JrError`
(exit 64) rather than a silently-wrong API call.

### Part C — NFR-S-D: Profile name error message precision

`src/config.rs:113-140` emits the same error message for both profile name length violations
(>64 chars) and charset violations. A 2-LOC fix distinguishes them with more precise messages.

Example:
- Current: `"Invalid profile name"` for both cases
- After: `"Profile name too long (max 64 characters)"` vs `"Profile name contains invalid characters (use a-z, 0-9, _-)"`.

### Part D — NFR-R-F: search_issues anti-loop guard

`src/api/jira/issues.rs:222-230` has a `get_changelog` loop with a `break if nextPage == current`
anti-loop guard. `search_issues` has a cursor loop with no analogous guard. This is not a confirmed
bug (Atlassian doesn't return a looping cursor in practice) but it is a defensive gap.

**Fix**: Add the same anti-loop guard to `search_issues` cursor loop. The guard is ~3 LOC copied
from `get_changelog`. Add a source comment pointing to the `get_changelog` pattern as the template.
This is DOCUMENT-AS-IS in NFR-R-F, but the DOCUMENT-AS-IS routing means "add source comment AND
optionally add the guard" — the 3-LOC guard is the better outcome.

## Behavioral Contracts

**BC-X.4.009** (rate limit cap): When `Retry-After` value exceeds `MAX_RETRY_AFTER_SECS = 60`,
the retry loop aborts and returns an error to the caller instead of sleeping for the full duration.
A warning is printed to stderr.

**BC-X.9.002** (duration validation): When `parse_duration` receives a pathological input that
would overflow `u64`, it returns `JrError::UsageError("duration too large")` with exit 64.
No silent wrap occurs in release builds.

**BC-6.4.001** (profile name validation): `validate_profile_name` rejects names >64 characters
with a message containing "too long" or "max 64", and rejects names with invalid charset with
a message containing "invalid characters" or "a-z, 0-9". The two error cases produce distinct
messages. (Part C of this story implements this precision.)

**BC-2.1.001** (no infinite loop in search_issues): The `search_issues` cursor loop detects a
repeated cursor token (next == prev) and terminates, returning accumulated results. No infinite
loop occurs on a looping-cursor server response. (Part D of this story implements this guard.)

## Acceptance Criteria

### Part A — Retry-After cap

**AC-001** (traces to BC-X.4.009 postcondition): Given a wiremock server that responds with
`Retry-After: 3600` (1 hour, exceeds cap), when `JiraClient::send` receives a 429, then it
aborts the retry loop immediately (does NOT sleep 3600 seconds) and returns an error. The test
must complete in <1 second wall time.

**AC-002** (traces to BC-X.4.009 postcondition): Given a wiremock server that responds with
`Retry-After: 30` (within cap), when `JiraClient::send` receives a 429, then it sleeps the
`Retry-After` value (≤30s capped in test to 0 via mock) and retries. Existing H-027 retry
behavior for valid values is preserved.

**AC-003** (traces to BC-X.4.009 invariant): Given `MAX_RETRY_AFTER_SECS = 60` is defined in
`src/api/rate_limit.rs`, when a reader searches the file, the constant is visible at the top of
the file with a comment explaining the cap rationale.

### Part B — parse_duration overflow

**AC-004** (traces to BC-X.9.002 postcondition): Given `parse_duration("99999999999999w", 8, 5)`
is called, then it returns `Err(JrError::UsageError("...duration too large..."))`. NOT a silent
wrap or panic.

**AC-005** (traces to BC-X.9.002 postcondition): Given `parse_duration("1w", 8, 5)` is called,
then it returns the correct value `1 * 5 * 8 * 3600 = 144000` seconds. The overflow check does
NOT affect valid inputs.

### Part C — Profile name error precision

**AC-006** (traces to BC-6.4.001 invariant — profile name validation): Given a profile name of 65 chars
(one over the limit), when `validate_profile_name` is called, then the error message contains
`"too long"` or `"max 64"`. NOT a generic `"Invalid profile name"`.

**AC-007** (traces to BC-6.4.001 invariant — profile name validation): Given a profile name of 5 chars
containing a space (invalid charset), when `validate_profile_name` is called, then the error message
contains `"invalid characters"` or `"a-z, 0-9"`. Distinct from the length error message.

### Part D — search_issues anti-loop

**AC-008** (traces to BC-2.1.001 invariant — no infinite loop): Given `search_issues` receives a
wiremock cursor loop (the next-cursor response body returns the same cursor as the current cursor),
when `search_issues` is called, then the loop terminates after detecting the repeated cursor and
returns the accumulated results so far. No infinite loop.

## Implementation Notes

### Part A: rate_limit.rs

```rust
const MAX_RETRY_AFTER_SECS: u64 = 60;

pub fn parse_retry_after(header: Option<&HeaderValue>) -> Duration {
    let secs = header
        .and_then(|v| v.to_str().ok())
        .and_then(|s| s.parse::<u64>().ok())
        .unwrap_or(DEFAULT_RETRY_SECS);

    if secs > MAX_RETRY_AFTER_SECS {
        eprintln!(
            "[jr] Retry-After: {} exceeds cap of {}s — aborting retry.",
            secs, MAX_RETRY_AFTER_SECS
        );
        // Returning u64::MAX causes the caller to return an error (not sleep)
        // The caller checks: if retry_secs > MAX_RETRY_AFTER_SECS { return Err(...) }
        return Duration::from_secs(u64::MAX);
    }
    Duration::from_secs(secs)
}
```

Alternatively, return `Option<Duration>` where `None` signals "abort retry". The approach must
not break existing `MAX_RETRIES=3` behavior for values within the cap.

### Part B: duration.rs

```rust
// Before:
let weeks = w_count * days_per_week as u64 * hours_per_day as u64 * 3600;

// After:
let weeks = w_count
    .checked_mul(days_per_week as u64)
    .and_then(|v| v.checked_mul(hours_per_day as u64))
    .and_then(|v| v.checked_mul(3600))
    .ok_or_else(|| JrError::UsageError("duration too large (overflow)".into()))?;
```

Apply `checked_mul` at each multiplication step. Return `JrError::UsageError` (exit 64).

### Part C: config.rs

```rust
if name.len() > 64 {
    return Err(JrError::config_error(
        "Profile name too long (max 64 characters)"
    ));
}
if !name.chars().all(|c| c.is_ascii_alphanumeric() || c == '-' || c == '_') {
    return Err(JrError::config_error(
        "Profile name contains invalid characters (use a-z, 0-9, -, _)"
    ));
}
```

Order: check length first (so the message is unambiguous if both conditions are true).

### Part D: issues.rs search_issues cursor loop

```rust
let mut prev_cursor: Option<String> = None;
loop {
    // ... fetch page ...
    let next_cursor = response.next_page_token.clone();
    if next_cursor == prev_cursor {
        // Anti-loop guard: cursor did not advance — abort
        break;
    }
    prev_cursor = next_cursor;
    // ... continue loop ...
}
```

Mirrors the pattern in `get_changelog` (lines 222-230).

## Risk / Dependencies

- **Risk: rate_limit.rs interface change.** `parse_retry_after` may be called from `client.rs`
  in a way that expects `Duration` not `Option<Duration>`. Audit the call sites before changing
  the return type. An alternative: return `Duration::from_secs(0)` for cap-exceeded (skip retry)
  and let the caller handle it as a zero-sleep with a separate "retry count exceeded" check.
- **Risk: tokio::time::pause in tests.** Tokio time control requires `tokio::test` with the
  `start_paused = true` parameter. Verify this is compatible with existing test infrastructure.
- **No cross-story dependencies.** Parts A-D are independent of all Wave 3 stories.
