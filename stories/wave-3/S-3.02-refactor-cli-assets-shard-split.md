# S-3.02: Shard-split src/cli/assets.rs (1,055 LOC) by command theme

## Context

`src/cli/assets.rs` is 1,055 LOC — just over the implicit ~1,000 LOC shard threshold (NFR-O-D).
The file handles 6 distinct asset subcommands: `search`, `view`, `tickets`, `schemas`, `types`,
`schema`. These map naturally to 3-4 thematic shards following the `cli/issue/` split pattern.

Unlike the `api/auth.rs` split (S-3.01 — nearly 2,000 LOC), this split is smaller and lower risk.
The `--open` filter in `assets.rs:303-321` uses a client-side mechanism (`status.colorName != "green"`)
and must remain functionally unchanged (NFR-O-M is DOCUMENT-AS-IS, covered in S-2.05).

**This is a pure refactor.** No behavioral changes. All existing tests must pass unchanged.

## Behavioral Contracts

**BC-4.2.001** (workspace ID discovery): After the split, workspace ID discovery and caching behavior
remains unchanged — `search.rs` shard invokes the workspace discovery path identically to the
pre-split `assets.rs`.

**BC-4.2.006** (AQL search): After the split, `jr assets search <AQL>` returns the same paginated
results and renders the same table/JSON output as pre-split.

## Acceptance Criteria

**AC-001** (traces to BC-4.2.001 postcondition): Given the split is complete, when `cargo test --lib`
runs, then ALL existing assets tests pass with zero modifications. H-037 and H-038 holdout tests
(written in S-2.03) continue to pass unchanged.

**AC-002** (traces to BC-4.2.006 postcondition): Given the module split, when `cargo build --release`
is run, then `jr assets search`, `jr assets view`, `jr assets tickets`, `jr assets schemas` all
produce output identical to pre-split (verified by existing integration tests).

**AC-003** (traces to BC-4.2.001 invariant): Given the split, when `wc -l src/cli/assets/*.rs` is
run, then NO single shard file exceeds 600 LOC (the split has achieved genuine decomposition).

## Implementation Notes

### Proposed module layout

```
src/cli/assets/
├── mod.rs      # dispatch match + re-exports; keeps the existing clap enum
├── search.rs   # handle_search (AQL search + enrichment + table/JSON output)
├── view.rs     # handle_view (single object lookup + attribute display)
├── tickets.rs  # handle_tickets (connected tickets for an asset key)
└── schemas.rs  # handle_schemas, handle_types, handle_schema (schema discovery)
```

### Migration plan

1. Create `src/cli/assets/` directory; move `src/cli/assets.rs` to `src/cli/assets/mod.rs`.
2. Extract `handle_search` and its helpers into `search.rs`.
3. Extract `handle_view` into `view.rs`.
4. Extract `handle_tickets` into `tickets.rs`.
5. Extract `handle_schemas` / `handle_types` / `handle_schema` into `schemas.rs`.
6. Verify `cargo test` green after each step.

The `--open` filter logic at `assets.rs:303-321` (client-side `colorName != "green"` comparison)
moves with `search.rs` unchanged. No behavioral modification.

## Test Plan

- Multi-profile boundary: cache calls within each shard continue to pass `profile: &str`.
- Reference: `architecture/module-decomposition.md` for the cli/issue/ canonical pattern.

## Risk / Dependencies

- **Risk: Shard boundary for helpers.** Private helper functions used by multiple handle_* fns
  must stay in `mod.rs` or move to a `helpers.rs` shard rather than being duplicated.
- **Risk: `--open` filter extraction.** The client-side `colorName != "green"` filter is
  co-located with `search.rs` logic. Verify it is NOT accidentally left in `mod.rs`.
- **Independent of S-3.01.** The cli/assets.rs and api/auth.rs splits can proceed in parallel.

## Out of Scope

- Any new behavior for assets commands.
