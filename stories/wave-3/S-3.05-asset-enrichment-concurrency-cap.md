# S-3.05: Cap asset enrichment join_all concurrency with buffer_unordered(8)

## Context

`src/cli/issue/list.rs:445` and `src/api/assets/linked.rs:216` use `futures::future::join_all`
to resolve CMDB asset objects concurrently. `join_all` starts ALL futures simultaneously — for an
issue list with K unique assets, K simultaneous HTTP calls fire at once.

For typical issue lists (5-10 assets) this is fine. For large project views (100+ issues with
many CMDB assets), it creates a burst of 100+ simultaneous HTTP calls. The `MAX_RETRIES=3`
retry cap in `rate_limit.rs` mitigates 429 storms but does not bound the burst size.

**Fix**: Replace `join_all(futures)` with a `FuturesUnordered`-style bounded approach.
The recommended pattern using the `futures` crate already in deps is `stream::iter(futures).buffer_unordered(8).collect()`.

The cap of 8 concurrent enrichment calls is a constant `MAX_CONCURRENT_ASSET_FETCHES = 8` that
must be easy to tune. It should be a named constant in `api/assets/linked.rs`, not a magic literal.

## Behavioral Contracts

**BC-4.3.002** (asset enrichment): After the cap is applied, enrichment produces the same results
as `join_all` — all K assets are resolved; only the concurrency (not correctness) changes.
H-038 pins this: `enrich_assets` resolves id-only assets via GET. The test must continue to pass.

**BC-X.1.005** (rate-limit retry): The cap reduces the 429-storm risk but does NOT remove the
existing retry logic. Both mechanisms operate independently. Capping concurrency reduces the
number of simultaneous requests; retry handles the remaining 429 responses.

## Acceptance Criteria

**AC-001** (traces to BC-4.3.002 postcondition): Given a wiremock server that serves 10 distinct
asset GET endpoints, when `enrich_assets` is called with 10 id-only assets, then all 10 are
resolved correctly — same as the pre-cap `join_all` behavior. H-038 holdout continues to pass.

**AC-002** (traces to BC-4.3.002 invariant): Given a wiremock server that accepts at most 8
concurrent connections simultaneously (enforced via a semaphore in the test), when `enrich_assets`
is called with 20 assets, then NO more than 8 concurrent in-flight HTTP calls are observed at
any moment. The test uses a `tokio::sync::Semaphore` or a wiremock delay + timing assertion.

**AC-003** (traces to BC-X.1.005 invariant): Given the cap is applied, when a wiremock server
returns 429 for some asset GETs, then the existing retry logic in `JiraClient::send` still retries
those calls (cap does not bypass retry). All assets are eventually resolved (or fail with
`NetworkError` after MAX_RETRIES).

**AC-004** (traces to BC-4.3.002 postcondition): Given `MAX_CONCURRENT_ASSET_FETCHES = 8` constant
is defined in `src/api/assets/linked.rs`, when a reviewer reads the code, the constant is visible
at the top of the file (not a magic literal buried in the `buffer_unordered` call site).

## Implementation Notes

### Code change

```rust
// Before (src/api/assets/linked.rs or src/cli/issue/list.rs):
let results = futures::future::join_all(enrich_futures).await;

// After:
use futures::stream::{self, StreamExt};

const MAX_CONCURRENT_ASSET_FETCHES: usize = 8;

let results: Vec<_> = stream::iter(enrich_futures)
    .buffer_unordered(MAX_CONCURRENT_ASSET_FETCHES)
    .collect()
    .await;
```

### Test strategy for AC-002 (verified pattern from research)

Use an `Arc<AtomicUsize>` peak-counter pattern with `fetch_max`:

```rust
let peak = Arc::new(AtomicUsize::new(0));
let in_flight = Arc::new(AtomicUsize::new(0));

// In the wiremock responder:
let n = in_flight.fetch_add(1, Ordering::SeqCst) + 1;
peak.fetch_max(n, Ordering::SeqCst);
tokio::time::sleep(Duration::from_millis(50)).await;  // forces overlap window
in_flight.fetch_sub(1, Ordering::SeqCst);
// ... return mock response ...

// After the test runs:
assert!(peak.load(Ordering::SeqCst) <= 8);
```

Do NOT use:
- `tokio::sync::Semaphore` in the mock (this *enforces* the cap rather than *measuring* it — would mask bugs in the cap implementation under test).
- Timing-based batch counting (e.g., "20 assets / 8 = 3 batches × 50ms = 150ms total"). This is flaky on CI under load.

The 50ms sleep in the wiremock handler is required to ensure overlap windows actually exist; without it, `buffer_unordered` may complete each future before the next starts.

## Architecture Compliance Rules

- `MAX_CONCURRENT_ASSET_FETCHES = 8` is a named constant at the top of `src/api/assets/linked.rs`.
  It is NOT an env-var or config option in v0.5 (too niche to expose as user-facing config).
  A comment `// Tune via MAX_CONCURRENT_ASSET_FETCHES for higher-volume environments` is acceptable.
- The `futures` crate is already in `[dependencies]`. No new dependency needed for `stream::iter`
  and `buffer_unordered`.
- The cap applies to the ENRICHMENT phase only (resolving id-only assets via GET). It does NOT
  apply to the initial AQL search or workspace discovery paths.
- `MAX_CONCURRENT_ASSET_FETCHES = 8` is conservative against Atlassian's documented 100 req/s GET burst with no per-app concurrency cap. 8 is a "good neighbor" default; not pessimal. Verified 2026-05-08.
- Reference: `architecture/module-decomposition.md` §assets for the enrichment pipeline.

- **Risk: `join_all` location.** NFR-P-NEW-1 lists two possible sites. Read both before changing.
  The primary fix is at the outermost call; the inner join_all (if any) in `linked.rs` is the
  per-asset resolution loop.
- **Non-issue: FuturesUnordered ordering.** Pre-flight verification confirmed both consumers (`linked.rs:218` `if let Ok(obj) = result` loops, `list.rs:449` HashMap inserts) are order-independent. `buffer_unordered` is the correct choice over `buffered`; no consumer change needed.
- **Small risk, small fix.** This is a ~5 LOC code change + test. Scope is intentionally tight.
