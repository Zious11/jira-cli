# S-3.01: Shard-split src/cli/auth.rs (~2,245 LOC at 2026-05-08 develop HEAD) into cli/auth sub-modules

**Verified LOC at activation HEAD dea1664: `wc -l src/cli/auth.rs` = 1,998.** (growth to ~2,245 LOC by 2026-05-08 develop HEAD due to S-2.07 auth JSON output paths and BC-7.4.013–016 sub-BCs).

## Context

`src/cli/auth.rs` is 1,998 LOC — nearly double the implicit ~1,000 LOC shard threshold that the
`cli/issue/` split already established as project convention. This violates NFR-O-D. The file
contains the following separable concerns:

- **OAuth flow entry points** (login handler, OAuth flag dispatch, token exchange orchestration)
- **Keychain layout** (per-profile key namespacing, legacy migration glue)
- **Profile management helpers** (switch, list, status, refresh, logout, remove — CLI command handlers)
- **Interactive prompts** (email/URL/client-id prompts for login, prompt branching)

The split mirrors `cli/issue/` which was decomposed from a single large file into `list.rs`,
`create.rs`, `workflow.rs`, `links.rs`, `helpers.rs`, `format.rs`, `assets.rs`.

Note: `src/api/auth_embedded.rs` is a sibling module (thin XOR-obfuscation plumbing for the
embedded OAuth app) and is out of scope — it stays as-is per ADR-0006. The `src/api/auth.rs`
file (OAuth flow + keychain backend) is a SEPARATE shard-split candidate addressed in a future
story; this story covers ONLY the CLI handler layer.

**This is a pure refactor.** No behavioral changes. All existing tests must pass after the split
with zero modifications to test bodies.

## Behavioral Contracts

**BC-1.1.001** (auth list): After the split, `jr auth list --output json` returns a JSON array
of profile objects with `name`, `active`, `url` fields — identical to pre-split behavior.

**BC-1.4.027** (keychain layout): Per-profile OAuth key namespacing (`<profile>:oauth-access-token`,
`<profile>:oauth-refresh-token`) must be preserved exactly after the split. The keychain glue in
`src/cli/auth/keychain.rs` only calls through to `src/api/auth.rs` helpers — it does NOT introduce
new keychain write paths.

**BC-7.4.013–016** (auth subcommand JSON shapes — added by S-2.07): The split must preserve the
JSON output paths for `auth list/switch/status/refresh/logout/remove`. Each shard handling these
subcommands continues to emit identical JSON shapes per the existing 11-test insta snapshot suite at
`src/cli/issue/json_output.rs:84-149` (and the equivalent auth snapshots added by S-2.07).

## Acceptance Criteria

**AC-001** (traces to BC-1.1.001 postcondition): Given the split is complete, when `cargo test --lib`
runs, then ALL existing auth tests pass with zero modifications to test assertions or fixture setup.
No tests are added or removed to accommodate the refactor.

**AC-002** (traces to BC-1.4.027 invariant): Given the keychain module is extracted to
`src/api/auth/keychain.rs`, when any function that reads or writes keychain credentials is called
(login, refresh, logout, remove), then the call goes through the keychain module's exported fns —
not direct `keyring::Entry` calls scattered elsewhere in the split modules.

**AC-003** (traces to BC-1.1.001 postcondition): Given the module split, when `cargo build --release`
runs, then the binary produces identical output for `jr auth list`, `jr auth status`, `jr auth
switch`, `jr auth logout` — verified by running the existing integration tests unchanged.

**AC-004** (traces to BC-1.1.001 invariant): Given the split, when `wc -l src/cli/auth/*.rs` is run,
then NO single shard file exceeds 800 LOC (ensuring the split creates genuine decomposition, not a
trivial rename).

## Implementation Notes

### Proposed module layout

```
src/cli/auth/
├── mod.rs          # Re-exports; public API surface unchanged (all pub items re-exported)
├── login.rs        # handle_login: OAuth flag dispatch, interactive email/URL prompts
├── switch.rs       # handle_switch: profile switching + active-profile update
├── list.rs         # handle_list: profile listing, JSON output
├── status.rs       # handle_status: auth state inspection + expiry hint
├── refresh.rs      # handle_refresh: token refresh (calls api/auth.rs refresh path)
├── logout.rs       # handle_logout: token revocation + keychain cleanup
├── remove.rs       # handle_remove: profile deletion guard (rejects active profile)
└── keychain.rs     # keychain glue constants for CLI layer (delegates to api/auth.rs)
```

### Migration plan

1. Create `src/cli/auth/` directory; move `src/cli/auth.rs` to `src/cli/auth/mod.rs`.
2. Extract login handler (largest function block) into `login.rs`.
3. Extract remaining handlers into their respective shards.
4. Extract any keychain-adjacent helpers into `keychain.rs` (or leave them in the calling handler).
5. Ensure `mod.rs` re-exports all previously-public items with the same visibility.
6. Verify `cargo test --all-features` green at each step (incremental, not big-bang).

### DO NOT change

- Function signatures
- Error types returned
- Keychain key strings (these are live keychain paths for existing users)
- The behavior of `refresh_oauth_token` (even though it has zero production callers — see S-3.03)
- `src/api/auth_embedded.rs` (sibling module stays as-is)

## Architecture Compliance Rules

- Follow the same pattern as `cli/issue/` decomposition: each shard exports only what it needs;
  `mod.rs` re-exports the public API surface unchanged.
- `src/cli/auth/keychain.rs` contains only CLI-layer keychain glue; it delegates to `src/api/auth.rs`
  for actual keyring operations. It does NOT call `keyring::Entry` directly.
- No new `pub` items may be introduced; only reorganization of existing items.
- Multi-profile boundary rule: every function that touches the keychain or cache must take
  `profile: &str` as its first argument (100% conformance pre-split; must remain 100% post-split).
- Reference: `architecture/module-decomposition.md` for the cli/issue/ split as the canonical pattern.
- NFR-O-D and R-M6 confirm `src/cli/auth.rs` is the target file (1,998 LOC), NOT `src/api/auth.rs`
  (which is a separate backend module with its own shard-split candidacy).

## Risk / Dependencies

- **Risk: Re-export surface.** If any internal function is used across two shards without being
  re-exported from `mod.rs`, compilation fails. Strategy: compile after EACH extraction step.
- **Risk: Test visibility.** Some `#[cfg(test)]` items in auth.rs may use `super::*` — these
  must move with their extraction target, not stay in `mod.rs`.
- **Risk: Circular references.** If `oauth.rs` calls `keychain.rs` and `keychain.rs` imports
  `oauth.rs`, the module graph is cyclic. Solution: `keychain.rs` has no imports from sibling
  shards; it is a pure leaf module.
- **No behavioral risk.** This is structural only.
