# S-3.03: Auto-refresh OAuth access tokens on 401 with per-profile single-flight coordination

## Context

This is a v2.0.0 rewrite of S-3.03. The v1.0.0 design was an investigation/decision-matrix
story (Options A/B/C) that proposed wiring `refresh_oauth_token` into `JiraClient::send`
but left the exact implementation path open to the implementer.

### v1 defects found by Perplexity verification (2026-05-08)

Two material spec defects were identified in v1.0.0 Option A via pre-flight verification
documented in `.factory/research/S-3.03-wave3-verification.md`:

1. **Trigger condition does not exist (Claim 2, REFUTED):** v1 Option A triggered on
   `{"code": "EXPIRED"}` in the 401 body. The real Jira REST v3 shape is
   `{"errorMessages": ["The access token provided is expired, revoked, malformed, or invalid
   for other reasons."]}` — there is NO machine-readable `code` field and NO RFC-6750-compliant
   `WWW-Authenticate: Bearer error="invalid_token"` header. The v1 trigger condition would
   never fire; the story was unimplementable as written.

2. **Concurrency coordination omitted (Claim 4, VERIFIED risk):** v1 did not address the case
   where multiple concurrent `JiraClient::send()` calls all see 401 simultaneously and all
   attempt `refresh_oauth_token`. Because Atlassian enforces strict single-use rotation (Claim 5,
   REFUTED: no 10-minute grace window), concurrent refresh attempts cause the second-through-Nth
   callers to see `invalid_grant` (the first refresh rotated the token out from under them).

### v2 design pre-flight verification (2026-05-08)

Six additional refinements were verified in `.factory/research/S-3.03-v2-design-verification.md`
before this spec was written:

1. **No DashMap dependency needed:** `OnceLock<std::sync::Mutex<HashMap<String, Arc<tokio::sync::Mutex<RefreshState>>>>>` — outer `std::sync::Mutex` held only briefly for HashMap lookup, no new dependency.
2. **Inner mutex MUST be `tokio::sync::Mutex`:** NOT `std::sync::Mutex` — tokio's does not poison on panic, which is the correct semantic for refresh coordination.
3. **Drop 10-minute reuse window claim:** Atlassian's stance is single-use rotation per RFC 6749; no grace window is documented. Treat refresh as strictly single-use (Claim 5, REFUTED).
4. **Inter-process race is a real risk:** Two concurrent `jr` processes sharing a profile can both see expired tokens, both refresh, and the second gets `invalid_grant`. Mitigated by AC-010 (post-hoc reconcile).
5. **Persist-failure wedge is a real risk:** If `store_oauth_tokens` fails after a successful refresh exchange, in-memory state must NOT be updated. Mitigated by AC-011 (persist-before-publish).
6. **Cite `gh` CLI alone for blanket-401 precedent:** aws-cli and gcloud use proactive expiry-time refresh (different pattern). `gh` CLI uses reactive 401 trigger like v2. Do not conflate the trigger patterns.

### Why this story now

NFR-O-B (MEDIUM) requires resolution. Manual `jr auth refresh` UX is acceptable but
auto-refresh is strictly better when correctly implemented. The v2 design has been
pre-flight-verified against current Atlassian behavior before TDD begins, following the
same pattern that produced DEC-010 (S-2.06 v2 pivot) and DEC-011 (S-2.07 v2 pivot).

This pivot is recorded as DEC-013 in STATE.md.

## Behavioral Contracts

**BC-1.1.002** (auth status): `jr auth status` shows the correct auth state regardless
of whether the current in-memory access token is the original or a refreshed one. A
transparently-retried 401 → refresh → retry cycle surfaces no state degradation to the
user; the command output is identical to a non-expired token flow.

**BC-1.4.027** (keychain layout): All refresh-rotation persistence goes through the
existing `store_oauth_tokens` helper in `src/api/auth.rs`, which uses per-profile
namespacing (`<profile>:oauth-access-token`, `<profile>:oauth-refresh-token`). The
new `src/api/refresh_coordinator.rs` module MUST NOT contain any raw `keyring::Entry::new()`
calls — it calls `store_oauth_tokens(profile, ...)` exclusively.

## Acceptance Criteria

**AC-001** (traces to BC-1.1.002 postcondition): Given a wiremock server that returns
HTTP 401 on the first request and HTTP 200 on the second, when `JiraClient::send()` is
invoked once by a caller that has previously authenticated in this process, then the
client automatically calls `refresh_with_single_flight`, retries the original request
once with the new access token, and returns the successful response to the caller. The
caller sees success — no explicit retry needed at call sites. Verified via wiremock.

**AC-002** (traces to BC-1.4.027 invariant): Given a successful refresh exchange (wiremock
`POST /oauth/token` returns new access and refresh tokens), when auto-refresh completes,
then the new `<profile>:oauth-access-token` and `<profile>:oauth-refresh-token` are written
to keychain using `store_oauth_tokens(profile, ...)` from `src/api/auth.rs` — NOT raw
`keyring::Entry` calls. Verified via keychain inspection in test.

**AC-003** (traces to BC-1.1.002 postcondition): Given the refresh token is expired or
revoked (wiremock `POST /oauth/token` returns `{"error": "invalid_grant", ...}` with any
4xx status), when auto-refresh is attempted, then `JiaClient::send()` surfaces
`JrError::NotAuthenticated` with hint `"run 'jr auth refresh' to re-authenticate"`.
No retry loop. No silent failure. One wasted HTTP call maximum.

**AC-004** (traces to BC-1.1.002 invariant): Auto-refresh is capped at exactly 1 attempt
per request. If the original request returns 401 and the refresh itself fails (any 4xx),
`send()` surfaces `NotAuthenticated`. If the refresh succeeds but the retry also returns
401, `send()` surfaces `NotAuthenticated` without a second refresh attempt. No recursion.
No loop. Verified via wiremock.

**AC-005** (traces to BC-1.4.027 postcondition) [CONCURRENCY]: Given N concurrent
`JiraClient::send()` calls (N=10) all hitting wiremock 401 on a shared profile, when
the single-flight coordinator is active, then exactly 1 HTTP `POST` request is made to
the refresh endpoint (`https://auth.atlassian.com/oauth/token`). Verified via
`Mock::expect(1)` on the refresh endpoint; MockServer drop verifies the expectation.
The remaining N-1 callers wait on the inner `tokio::sync::Mutex<RefreshState>` and
use the resulting access token from `RefreshState.last_access_token` without making
additional refresh requests.

**AC-006** (traces to BC-1.4.027 postcondition) [CONCURRENCY-EDGE]: Given N concurrent
`JiaClient::send()` calls all hitting 401 on a shared profile AND the refresh endpoint
returns `invalid_grant`, when the single-flight coordinator is active, then exactly 1
refresh HTTP request is made, the `RefreshState` caches the failure, and all N callers
surface `NotAuthenticated` without a thundering-herd retry storm. No caller retries the
refresh independently. Verified via wiremock.

**AC-007** (traces to BC-1.1.002 invariant): The existing `jr auth refresh` CLI command
(manual recovery path) continues to work unchanged. The new auto-refresh wiring does not
modify `src/cli/auth.rs` or the `refresh_credentials` / `refresh_success_payload` handler
functions. Verified by running the existing auth holdout suite (H-001..H-008, H-022, H-029)
with no regressions.

**AC-008** (traces to BC-1.4.027 postcondition): A wiremock contract test pins the
following invariants together: (a) refresh endpoint URL is `https://auth.atlassian.com/oauth/token`,
(b) request body includes `grant_type=refresh_token`, (c) rotation — both new access
token AND new refresh token — is persisted to keychain via `store_oauth_tokens`,
(d) RFC-6749 `{"error": "invalid_grant"}` body on any 4xx is surfaced cleanly as
`NotAuthenticated`. All four assertions fail before implementation (red gate).

**AC-009** (traces to BC-1.4.027 invariant) [STALE-CACHE]: After a refresh succeeds,
queued waiters (N-1 callers) retry with the NEW access token from the in-memory
`RefreshState.last_access_token`. They do NOT re-read keychain to get the token —
macOS Security framework keychain access is 5-50ms synchronous and blocks the tokio
executor. The keychain write (via `store_oauth_tokens`) happens once in the refresher
path. Waiters read from in-memory state. Verified via wiremock + Mutex drop ordering.

**AC-010** (traces to BC-1.4.027 postcondition) [INTER-PROCESS-RECONCILE]: Given two
concurrent `jr` processes sharing a profile (Process A and Process B both see expired
access token), when Process A refreshes successfully (rotating the refresh token) and
Process B subsequently attempts refresh (with now-stale refresh token), then Process B
receives `invalid_grant` from Atlassian. Before surfacing `NotAuthenticated`, Process B
re-reads the keychain's `<profile>:oauth-refresh-token`. If that value differs from the
refresh token Process B just sent (meaning Process A refreshed first), Process B retries
the ORIGINAL API call with the keychain's new `<profile>:oauth-access-token` instead of
surfacing the error. If that retry also returns 401, THEN surface `NotAuthenticated`.
Verified via wiremock + concurrent-process simulation using two `JiraClient` instances.

**AC-011** (traces to BC-1.4.027 postcondition) [PERSIST-BEFORE-PUBLISH]: Given a
successful refresh exchange from Atlassian AND a `store_oauth_tokens` failure (disk-full,
keychain timeout — fault-injected in test), when the persist step fails, then the
in-memory `RefreshState` is NOT updated with the new access token. The error propagates
to the caller; the next call will retry the refresh (which will fail with `invalid_grant`
because the rotation already happened server-side), and the user receives a clear error
prompting re-authentication. No in-memory/on-disk divergence is introduced. Verified via
fault injection in test.

## Implementation Notes

### Trigger: blanket-401

Auto-refresh fires on any HTTP 401 received from a request that used OAuth credentials
(i.e., not API-token auth). This is the `gh` CLI pattern. Cost in error paths: one wasted
HTTP round-trip when 401 means "wrong scope", "user deleted", "2FA step-up required", or
any other non-expiry cause. Acceptable trade-off: Jira REST v3 does not differentiate 401
sub-cases in the response body or headers; substring-match on `errorMessages[0]` is
locale-fragile (Atlassian can reword the message without warning) and provides no
meaningful additional signal. The blanket-401 approach matches industry convention.

Note: proactive expiry-time refresh (aws-cli / gcloud pattern) is a future optimization.
It requires persisting `expires_at` in keychain or config and is deferred to a follow-up
story. v2.0.0 is reactive-401 only.

### Single-flight primitive: `src/api/refresh_coordinator.rs`

```rust
use std::collections::HashMap;
use std::sync::{Arc, Mutex as StdMutex, OnceLock};
use tokio::sync::Mutex as TokioMutex;

// RefreshState holds the result of the most recent refresh attempt for this
// profile in this process:
// - None: no refresh has been attempted yet
// - Some(Ok(access_token)): last refresh succeeded; waiters use this token
// - Some(Err(JrError)): last refresh failed; waiters surface this error
pub struct RefreshState {
    last_result: Option<Result<String, JrError>>,
}

impl RefreshState {
    pub fn idle() -> Self {
        Self { last_result: None }
    }
}

// Outer std::sync::Mutex: held ONLY briefly to look up / insert the per-profile
// inner mutex. Released BEFORE any .await. NEVER held across an .await.
//
// Inner tokio::sync::Mutex: held across the .await for the actual refresh HTTP
// call. MUST be tokio::sync::Mutex (NOT std::sync::Mutex) — tokio's variant does
// NOT poison on panic, which is the correct semantic: a panicked refresh should
// not permanently wedge the coordinator for subsequent callers.
static COORDINATORS: OnceLock<StdMutex<HashMap<String, Arc<TokioMutex<RefreshState>>>>> =
    OnceLock::new();

pub async fn refresh_with_single_flight(profile: &str) -> Result<String, JrError> {
    let inner = {
        // Outer lock: brief HashMap lookup/insert only. Released before .await.
        let mut map = COORDINATORS
            .get_or_init(|| StdMutex::new(HashMap::new()))
            .lock()
            .expect("refresh coordinator map poisoned");
        map.entry(profile.to_string())
            .or_insert_with(|| Arc::new(TokioMutex::new(RefreshState::idle())))
            .clone()
    }; // outer StdMutex dropped here, BEFORE .await

    let mut guard = inner.lock().await; // holds inner TokioMutex across refresh

    // Single-flight: if a prior caller already refreshed (or failed), short-circuit.
    if let Some(ref cached) = guard.last_result {
        return cached.clone();
    }

    // This caller is the refresher. Call the existing refresh_oauth_token.
    // PERSIST-BEFORE-PUBLISH: store_oauth_tokens is called inside refresh_oauth_token
    // before we update RefreshState. If persist fails, refresh_oauth_token surfaces
    // an error and we set last_result to Err — RefreshState is not polluted with a
    // token that isn't on disk.
    let result = crate::api::auth::refresh_oauth_token(profile).await;
    guard.last_result = Some(result.clone());
    result
}
```

### Wire-in: `src/api/client.rs`

In `JiraClient::send`, after receiving `StatusCode::UNAUTHORIZED` from a request that
used OAuth credentials:

```rust
// Blanket-401 → one-shot auto-refresh (capped at 1 attempt; no recursion)
if response.status() == StatusCode::UNAUTHORIZED && !already_retried_refresh {
    match refresh_coordinator::refresh_with_single_flight(&self.profile).await {
        Ok(new_token) => {
            // Retry the original request once with the new access token.
            // already_retried_refresh = true prevents recursion.
            return self.send_with_token_once(req_clone, &new_token).await;
        }
        Err(_) => {
            return Err(JrError::NotAuthenticated {
                hint: "run 'jr auth refresh' to re-authenticate".into(),
            });
        }
    }
}
```

The `already_retried_refresh` flag is NOT a module-level state — it is a parameter
threaded through the call (or a local bool in the send function). This prevents a
refreshed-but-still-401 response from triggering a second refresh loop.

### Post-hoc reconcile: AC-010

On `invalid_grant` failure path inside `refresh_with_single_flight` (before setting
`last_result`):

1. Re-read `<profile>:oauth-refresh-token` from keychain.
2. Compare to the refresh token that was just sent in the failed request.
3. If they differ (a sibling process refreshed first): read `<profile>:oauth-access-token`
   from keychain and return `Ok(new_access_token)` — let the caller retry the original
   API call with this token.
4. If they are the same (genuine expiry): set `last_result = Some(Err(NotAuthenticated))`
   and surface the error.

Approximately 15 LOC. No new dependency.

### Persist-before-publish: AC-011

The existing `refresh_oauth_token` in `src/api/auth.rs` already calls `store_oauth_tokens`
before returning the new access token. The ordering in `refresh_with_single_flight` MUST be:

```
1. Call refresh_oauth_token(profile).await
   (which internally does: HTTP exchange → store_oauth_tokens → return Ok(access_token))
2. Only if (1) returns Ok: set guard.last_result = Some(Ok(access_token))
3. If (1) returns Err: set guard.last_result = Some(Err(...))
```

Because `store_oauth_tokens` is called INSIDE `refresh_oauth_token` before it returns,
if persist fails, `refresh_oauth_token` surfaces `Err` and `RefreshState` is set to
`Err` — in-memory state is never diverged from on-disk state.

### Keychain latency note

macOS Security framework `keyring::Entry` access costs 5-50ms synchronous. DO NOT call
keychain helpers inside the inner `tokio::sync::Mutex<RefreshState>` `.await` path more
than necessary:

- One keychain WRITE per successful refresh (via `store_oauth_tokens` inside
  `refresh_oauth_token`): acceptable.
- The ~15 LOC post-hoc reconcile keychain READ on `invalid_grant`: acceptable (rare path).
- Waiters reading the cached access token MUST read from in-memory `RefreshState`,
  NOT from keychain. A keychain re-read for every waiter would block the executor.

### BC-1.4.027 keychain enforcement

Every refresh-rotation persistence goes through `src/api/auth.rs::store_oauth_tokens`
(which takes `profile: &str` and uses per-profile namespaced keys). The new
`src/api/refresh_coordinator.rs` file MUST NOT contain any raw `keyring::Entry::new()`
calls. If one appears, it is a bug — remove it.

## Architecture Compliance Rules

- **No new production dependencies.** No DashMap, no moka, no singleflight-async. Use
  `OnceLock + std::sync::Mutex<HashMap> + tokio::sync::Mutex<RefreshState>` — all from
  the standard library and existing tokio dep.
- **Inner mutex MUST be `tokio::sync::Mutex`**, NOT `std::sync::Mutex`. Rationale:
  `tokio::sync::Mutex` does not poison on panic; a panicked refresh should not permanently
  wedge the coordinator for subsequent callers. A future contributor must not "fix" this
  to `std::sync::Mutex`.
- **Outer `std::sync::Mutex<HashMap>` MUST NOT be held across any `.await`.** The outer
  lock is held only for HashMap `entry()` lookup and `Arc::clone`. Release BEFORE the
  `inner.lock().await` call.
- **Retry cap is 1.** `JiaClient::send()` may invoke auto-refresh at most once per call.
  No recursive `send()` calls. No loop over refresh attempts. The `already_retried_refresh`
  flag (local bool, not global state) enforces this cap.
- **`src/api/refresh_coordinator.rs` contains NO raw `keyring::Entry::new()` calls.**
  All keychain access goes through `src/api/auth.rs`'s exported helpers. If a raw
  `keyring::Entry` appears in `refresh_coordinator.rs`, it is a correctness bug.
- **Multi-profile boundary:** `refresh_with_single_flight` takes `profile: &str` as its
  first argument. The per-profile HashMap key ensures separate coordinators per profile.
  Cross-profile state leakage is a correctness bug.
- **`src/api/mod.rs`:** Add `pub mod refresh_coordinator;` to expose the module.
