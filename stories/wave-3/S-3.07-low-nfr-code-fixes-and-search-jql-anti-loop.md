## Context

This story bundles three LOW-severity NFR fixes that each require ≤10 LOC. Grouping closes them
without story sprawl.

### Part A — NFR-R-NEW-1: Retry-After cap (BC-X.4.009)

**Reality-aligned framing (corrected from v1.0.0):** Atlassian's `/rest/api/3/...` rate-limited
responses typically send `Retry-After` values of **1425–3089 seconds** (24–50 minutes), with a
documented ceiling of **3600 seconds**. The v1.0.0 story used `Retry-After: 86400` (24h) as the
threat framing — that is a hypothetical extreme, not actual Atlassian behavior. With
`MAX_RETRY_AFTER_SECS = 60`, jr will abort essentially every real-world 429 — which IS the
intended behavior for an interactive CLI (foreground 30-min sleep is poor UX), but the story
rationale and user-facing error message must reflect reality.

**Trade-off documented**: jr is interactive-CLI-first. RFC 9110 §10.2.3 confirms the client MAY
abort instead of honoring Retry-After. Users running batch operations should wrap jr in a
shell-level retry/cron job — jr won't sleep for them.

**Fix**: Add `const MAX_RETRY_AFTER_SECS: u64 = 60;`. If the parsed value exceeds the cap, print
a reality-aligned warning to stderr and abort retry (return error, not sleep).

H-027 currently pins the gap (KNOWN-GAP: `Retry-After: 86400` sleeps for 24 hours — the phrasing
is hypothetical; actual gap is "any value >60s sleeps that long"). After this fix, H-027 transitions
to MUST-PASS with the test using `Retry-After: 2400` (matches typical Atlassian).

### Part B — DROPPED (conditional on S-3.10)

The v1.0.0 story included Part B (parse_duration overflow guard via `checked_mul` chain at the
3-arg calculator). That part is **dropped from v2** because S-3.10 deletes the entire 3-arg
`parse_duration` function. Adding `checked_mul` to a function being removed is wasted churn.

**Conditional reinstatement**: If S-3.10 slips out of Wave 3 OR is rejected, Part B becomes a
real fix again — the orchestrator's earlier claim that "WV2-SEC-01's 64-byte cap eliminates
overflow" was mathematically false. With `parse::<u64>()` accepting up to 20 digits, then
`× DPW × HPD × 3600`, inputs of 14–20 digits + `w` (e.g., the v1 story's own example
`99999999999999w`) **still overflow u64 within the 64-byte cap**. The validator
(`parse_duration_validate`, post-WV2-SEC-01) is safe because it doesn't multiply, but the
calculator is genuinely at risk. AC-NEW-B (sequencing guard) enforces verifying S-3.10 has
shipped before this story merges.

### Part C — NFR-S-D: Profile name error message precision

`src/config.rs:113-140` emits the same error message for both profile name length violations
(>64 chars) and charset violations. A 2-LOC fix distinguishes them with more precise messages.

Example:
- Current: `"Invalid profile name"` for both cases
- After: `"Profile name too long (max 64 characters)"` vs `"Profile name contains invalid characters (use a-z, 0-9, -, _)"`.

Order: check length first (so the message is unambiguous if both conditions are true — length
check fires before charset check for a 65-char name with a space).

### Part D — NFR-R-F: search_issues anti-loop guard (ELEVATED — confirmed JRACLOUD-94632 bug)

**Corrected framing (from v1.0.0):** v1 framed the cursor-loop guard as "not a confirmed bug ...
defensive gap." **This is incorrect as of May 2026.** `/rest/api/3/search/jql` returning the same
`nextPageToken` twice is a **confirmed, ongoing, multi-source-documented Jira Cloud bug**:

- Atlassian internal tickets: **JRACLOUD-94632, JRACLOUD-92049, JRACLOUD-85546**
- Atlassian Community thread: "REST: The new /rest/api/3/search/jql endpoint is a complete..."
  (forum bug report)
- Atlassian's own MCP server hits it: `atlassian/atlassian-mcp-server#118`
- Sibling CLI hits it: `ankitpokhrel/jira-cli#898`

**Fix (elevated from "defensive" to "real bug response"):** Replace the existing
`// KNOWN-GAP: NFR-R-F` source comment at `src/api/jira/issues.rs:59-65` with a real defensive
guard PLUS a user-facing stderr warning citing JRACLOUD-94632 so users have a search term for
the upstream bug tracker.

The guard mirrors `get_changelog`'s anti-loop pattern (lines 222-230 at activation HEAD
`dea1664`): if the next-cursor response equals the current cursor, abort the loop AND emit a
stderr warning. Use `next == prev` as the trigger.

NFR-R-F's nfr-catalog routing column updates from `DOCUMENT-AS-IS` to `DOCUMENT-AS-IS-FIXED`
— the comment is improved AND the actual guard is delivered, not just documented.

## Implementation Notes

### Part A: rate_limit.rs

```rust
const MAX_RETRY_AFTER_SECS: u64 = 60;
// Cap rationale: jr is interactive-CLI-first. Atlassian's typical Retry-After
// values are 1425-3089s (24-50 min) per Atlassian community forum reports;
// foreground 30-min sleep is poor UX. RFC 9110 §10.2.3 confirms the client
// MAY abort instead of honoring Retry-After. Users running batch operations
// should wrap jr in a shell-level retry/cron job.
pub fn parse_retry_after(header: Option<&HeaderValue>) -> Option<Duration> {
    let secs = header
        .and_then(|v| v.to_str().ok())
        .and_then(|s| s.parse::<u64>().ok())
        .unwrap_or(DEFAULT_RETRY_SECS);
    if secs > MAX_RETRY_AFTER_SECS {
        eprintln!(
            "[jr] Atlassian requested {}s wait — exceeds {}s cap for interactive CLI.\n\
             Aborting retry; rerun later or wrap in a shell-level retry/cron job.",
            secs, MAX_RETRY_AFTER_SECS
        );
        return None;  // None signals "abort retry" to caller
    }
    Some(Duration::from_secs(secs))
}
```

### Part D: issues.rs search_issues cursor loop

Replace the existing `// KNOWN-GAP: NFR-R-F` source comment block at lines 59-65 with:

```rust
// Anti-loop guard: Jira Cloud /rest/api/3/search/jql intermittently returns
// the same nextPageToken twice, causing infinite pagination loops. This is a
// confirmed upstream bug — JRACLOUD-94632, JRACLOUD-92049, JRACLOUD-85546
// (also reported in atlassian/atlassian-mcp-server#118 and
// ankitpokhrel/jira-cli#898). Mirrors anti-loop pattern from get_changelog
// (lines 222-230 at activation HEAD dea1664). When the guard fires, we emit
// a stderr warning citing JRACLOUD-94632 so users have a search term.
let mut prev_cursor: Option<String> = None;
loop {
    // ... fetch page ...
    let next_cursor = response.next_page_token.clone();
    if next_cursor == prev_cursor {
        eprintln!(
            "[jr] WARNING: Atlassian /rest/api/3/search/jql returned the same \
             nextPageToken twice — aborting pagination loop. Some results may \
             be missing. Upstream bug: JRACLOUD-94632."
        );
        break;
    }
    prev_cursor = next_cursor;
    // ... continue loop ...
}
```
