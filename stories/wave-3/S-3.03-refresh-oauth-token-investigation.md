# S-3.03: Investigate and wire (or document removal of) refresh_oauth_token

## Context

`refresh_oauth_token` exists at `src/api/auth.rs:704-770` and has **zero production callers**.
It was written for a future 401 auto-refresh integration per CLAUDE.md notes (gap G-A1 in
edge-case-catalog.md). It is public but unreachable in production flows.

This creates two risks:
1. Dead code accrues bit-rot — the refresh token endpoint, parameter shape, or keychain key
   path may diverge from the live refresh path used by `jr auth refresh`.
2. If it IS wired in, a broken refresh path silently fails API calls rather than prompting re-auth.

NFR-O-B (MEDIUM, DEFER) requires this to be resolved. This story makes the decision and acts on it.

### Decision tree

**Option A — Wire into `JiraClient::send`**: When `send` receives a 401 response AND the error
body contains `"code": "EXPIRED"` (or the Atlassian `expired_token` error code), attempt ONE
refresh via `refresh_oauth_token`. If refresh succeeds, retry the original request. If refresh
fails, surface `NotAuthenticated` with a hint to `jr auth refresh`. This closes the auto-refresh
gap.

**Option B — Remove as dead code**: If investigation reveals the refresh token Atlassian API path
is the same as the `jr auth refresh` CLI command (i.e., no new HTTP path needed), document the
existing `jr auth refresh` command as the recommended recovery and remove the unused function.
Add a `// REMOVED: see S-3.03 ADR note` comment.

**Option C — Keep dormant with test coverage**: If the token refresh HTTP path is valid but wiring
into `send` is risky for v0.5, add a unit test that exercises `refresh_oauth_token` directly against
a wiremock server (verifying the function itself works), and document explicitly in CLAUDE.md that
this path is NOT auto-wired (manual `jr auth refresh` remains the flow).

The implementer MUST read `refresh_oauth_token` (lines 704-770) and the `jr auth refresh`
handler before choosing an option. The decision must be documented in a `// S-3.03 DECISION:` comment.

## Behavioral Contracts

**BC-1.1.002** (auth status): After this story, `jr auth status` shows the correct auth state
regardless of which option is chosen. If Option A is chosen, a 401+expired_token causes a transparent
retry — auth status is not degraded.

**BC-1.4.027** (keychain layout): If `refresh_oauth_token` is wired (Option A), the refresh path
reads `<profile>:oauth-refresh-token` from keychain and writes back the new `<profile>:oauth-access-token`.
Keychain key naming is unchanged.

## Acceptance Criteria

**AC-001** (traces to BC-1.1.002 postcondition): Given Option A is chosen, when `JiraClient::send`
receives a 401 with `{"code": "EXPIRED"}` (or Atlassian's `expired_token` error), then the client
automatically calls `refresh_oauth_token`, updates the keychain with the new token, and retries the
original request once. The caller sees the successful response — no explicit retry needed at call sites.

**AC-002** (traces to BC-1.4.027 invariant): Given Option A is chosen, when the refresh token
exchange succeeds (mock `/oauth/token` returns new access/refresh tokens), then the new
`<profile>:oauth-access-token` and `<profile>:oauth-refresh-token` are written to keychain using
the existing `keychain.rs` helper (NOT raw `keyring::Entry` calls).

**AC-003** (traces to BC-1.1.002 postcondition): Given Option B or C is chosen, when
`cargo clippy -- -D warnings` runs, then NO `dead_code` warning is emitted for `refresh_oauth_token`
(it is either used, removed, or suppressed with a documented justification comment).

**AC-004** (traces to BC-1.1.002 invariant): Given any option chosen, when this story is complete,
a `// S-3.03 DECISION: [Option A|B|C] chosen because <reason>` comment appears at the original
function site or its removal location in git history. The decision is not left implicit.

**AC-005** (traces to BC-1.4.027 postcondition): Given Option A is chosen, when the refresh
attempt itself returns 401 (the refresh token is also expired), then `send` surfaces a
`JrError::NotAuthenticated` with hint `run 'jr auth refresh' to re-authenticate` — NOT an infinite
retry loop. The retry counter for auto-refresh is capped at 1 attempt.

## Implementation Notes

### Investigation checklist (do BEFORE choosing option)

1. Read `refresh_oauth_token` at lines 704-770. Note: what endpoint does it call?
   (`POST /oauth/token` with `grant_type=refresh_token`)
2. Read the `jr auth refresh` CLI handler. Does it call `refresh_oauth_token` already?
   If yes, the function IS called — just not from `send`. Option B (remove) would be wrong.
3. Check `src/api/client.rs::send`. Where does 401 currently surface?
   (Currently: raw `StatusCode::UNAUTHORIZED` → `JrError::NotAuthenticated`)
4. Verify the Atlassian expired-token response body format. Does it actually include
   `"code": "EXPIRED"` or `"error": "expired_token"`? Check wiremock fixtures or API docs.

### If Option A — wire into send

```rust
// In client.rs::send, after receiving StatusCode::UNAUTHORIZED:
if let Some(expired_token_code) = extract_expired_token_code(&body) {
    // One-shot refresh attempt (not recursive)
    // Per CLAUDE.md gotcha: pass ONLY profile — refresh_oauth_token resolves credentials internally.
    if let Ok(new_token) = refresh_oauth_token(&self.profile).await {
        // Update in-memory auth header for retry
        let retry_response = self.send_with_token(req, &new_token).await?;
        return Ok(retry_response);
    }
}
return Err(JrError::NotAuthenticated { hint: "run 'jr auth refresh'".into() });
```

## Architecture Compliance Rules

- If Option A: the refresh retry in `send` is capped at ONE attempt. No recursion. No loop.
  This prevents infinite refresh loops if the refresh token endpoint itself fails.
- `refresh_oauth_token` reads/writes keychain using the `keychain.rs` helper (NOT raw
  `keyring::Entry` calls after S-3.01 split; or existing keychain helpers if S-3.01 not yet merged).
- Multi-profile boundary: `refresh_oauth_token` already takes `profile: &str` — preserve this.
- **CRITICAL per CLAUDE.md gotcha**: `refresh_oauth_token` resolves credentials internally
  (keychain → embedded). Callers pass ONLY `profile: &str`. Do NOT re-introduce
  `client_id`, `client_secret`, or `config` parameters — they short-circuit the internal resolver.
  The pseudo-code above reflects the correct call signature: `refresh_oauth_token(&self.profile).await`.
- Reference: `architecture/module-decomposition.md` §auth for the refresh path architecture.
