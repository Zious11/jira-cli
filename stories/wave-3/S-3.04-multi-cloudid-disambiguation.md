# S-3.04: Multi-cloudId disambiguation: --cloud-id flag + prompt for ambiguous orgs

## Context

`accessible_resources` at `src/api/auth.rs:666-668` uses `resources.first()` — the first result
returned by `GET /oauth/token/accessible-resources` is used, and all other resources are silently
ignored. For users with a single Atlassian Cloud org, this is correct. For users who have access
to multiple orgs (e.g., employee + contractor Jira instances, or a developer with a personal + work
Atlassian account), the "first result wins" behavior causes silent mis-targeting — commands run
against the wrong org with no indication.

H-047 is registered as `KNOWN-GAP` for this issue. This story resolves the known gap.

### The fix

When `accessible_resources` returns >1 result:
- If `--cloud-id <id>` is provided, use it directly (non-interactive, for scripts).
- If `--no-input` is NOT set, prompt the user to select from the list interactively.
- If `--no-input` IS set and no `--cloud-id` provided, emit a clear error: exit 64 with
  `"Multiple Atlassian orgs found. Use --cloud-id <id> to disambiguate."`.

When `accessible_resources` returns exactly 1 result, behavior is unchanged (no prompt, no flag needed).

## Behavioral Contracts

**BC-1.5.038** (multi-cloud disambiguation — primary): When `accessible_resources` returns
multiple resources and no `--cloud-id` is provided and `--no-input` is set, the command exits 64
with an actionable error message identifying available cloud IDs. This closes H-047's KNOWN-GAP.
After this story ships, H-047 transitions from `KNOWN-GAP` to `MUST-PASS`.

**BC-1.1.007** (profile precedence): When `--cloud-id` is provided alongside `--profile`, the
cloud-id selection applies to the specified profile only. The precedence chain for other flags
is unchanged.

**BC-1.5.031** (OAuth callback): The `--cloud-id` flag does NOT change the callback URL or port.
It only filters the `accessible_resources` response AFTER the OAuth exchange completes. The
`http://127.0.0.1:53682/callback` callback URL remains fixed.

## Acceptance Criteria

**AC-001** (traces to BC-1.1.007 postcondition): Given a mock `accessible_resources` response
returns two resources (`cloudId-A`, `cloudId-B`), when `jr auth login --oauth --cloud-id cloudId-A`
is invoked, then the stored profile `cloud_id` is `cloudId-A` — NOT `cloudId-B` (which would be
the first-result-wins outcome). The mock for the auth completion endpoint receives a request
scoped to `cloudId-A`.

**AC-002** (traces to BC-1.1.007 postcondition): Given a mock `accessible_resources` response
returns two resources AND `--no-input` is set AND `--cloud-id` is NOT provided, when `jr --no-input
auth login --oauth` is invoked, then exit 64 with stderr containing `"Multiple Atlassian orgs"` and
`"--cloud-id"`.

**AC-003** (traces to BC-1.1.007 invariant): Given a mock `accessible_resources` response returns
exactly one resource, when `jr auth login --oauth` is invoked (no `--cloud-id`), then behavior is
identical to pre-fix (first-and-only result used, no prompt, no error). This is the common case;
it must not regress.

**AC-004** (traces to BC-1.5.031 invariant): Given any invocation, the callback URL bound in the
OAuth PKCE/authorization flow remains `http://127.0.0.1:53682/callback`. `--cloud-id` is a
post-exchange filter on `accessible_resources`, not a change to the authorization URL.

**AC-005** (traces to BC-1.1.007 postcondition): Given the interactive prompt path (multi-resource,
`--no-input` not set), when a test invokes via piped stdin with a selection index, then the correct
`cloud_id` is stored in the profile. (Interactive test uses `assert_cmd` with `write_stdin`.)

**AC-006** (traces to H-047, BC-1.5.038 postcondition): Given a mock `accessible_resources`
response returns two resources `[{id: "cloud-A", name: "Company A"}, {id: "cloud-B", name: "Company B"}]`
and `--no-input` is set and `--cloud-id` is NOT provided, when `jr --no-input auth login --oauth`
is invoked, then exit code is 64 and stderr contains both `"Multiple Atlassian orgs"` and `"--cloud-id"`.
**H-047 status flips from KNOWN-GAP to MUST-PASS once this story merges** (update
`.factory/specs/prd/holdout-scenarios.md` H-047 status field as part of this story).

## Implementation Notes

### `accessible_resources` signature change

```rust
// Before:
pub async fn accessible_resources(&self) -> Result<CloudId> {
    let resources: Vec<AccessibleResource> = ...;
    resources.first().map(|r| r.id.clone()).ok_or(...)
}

// After:
pub async fn accessible_resources(
    &self,
    cloud_id_override: Option<&str>,
    no_input: bool,
) -> Result<CloudId> {
    let resources: Vec<AccessibleResource> = ...;
    match resources.len() {
        0 => Err(JrError::NotAuthenticated { ... }),
        1 => Ok(resources[0].id.clone()),
        _ => {
            if let Some(id) = cloud_id_override {
                resources.iter().find(|r| r.id == id)
                    .map(|r| r.id.clone())
                    .ok_or(JrError::config_error("cloud-id not found in accessible resources"))
            } else if no_input {
                Err(JrError::UsageError("Multiple Atlassian orgs found. Use --cloud-id <id>".into()))
            } else {
                // Interactive prompt — render list, get selection
                prompt_cloud_id_selection(&resources)
            }
        }
    }
}
```

### `--cloud-id` flag

Add to `jr auth login` subcommand (and potentially `jr auth refresh`):
```rust
/// Cloud ID to use when multiple Atlassian orgs are accessible (disambiguates first-result-wins).
#[arg(long)]
cloud_id: Option<String>,
```

### Error exit code

`--cloud-id <id>` provided but not in the list: exit 64 (`JrError::UsageError`).
`--no-input` + multi-resource + no `--cloud-id`: exit 64.

## Test Plan

## Architecture Compliance Rules

- `--cloud-id` is a flag on `jr auth login` (and `jr auth refresh` if applicable). It is NOT
  a global flag — cloud-id selection is only meaningful during OAuth flow, not on every command.
- Exit code for disambiguation errors: 64 (`JrError::UsageError`) per error taxonomy.
- Interactive prompt (when `--no-input` not set) uses the same `dialoguer::Select` pattern
  as other interactive prompts in the codebase.
- Multi-profile boundary: `accessible_resources` takes `profile: &str` — unchanged.
- Reference: `architecture/module-decomposition.md` §auth for OAuth flow architecture.

## Risk / Dependencies

- **Risk: `accessible_resources` call sites.** If called from multiple places (e.g., `auth login`
  AND `auth refresh`), both sites must pass `cloud_id_override` and `no_input`. Audit all callers
  before changing the signature.
- **Risk: Interactive path in CI.** The interactive prompt cannot run in CI without `write_stdin`.
  Gate interactive-path tests behind a flag or use `assert_cmd::Command::write_stdin` to simulate
  selection.
- **No dependency on S-3.01.** This story modifies `accessible_resources` — if S-3.01 has
  already split auth.rs, update the file path; otherwise modify `auth.rs` directly.
