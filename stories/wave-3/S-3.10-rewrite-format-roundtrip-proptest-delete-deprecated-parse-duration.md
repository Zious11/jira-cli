# S-3.10: Rewrite format_roundtrip proptest + delete deprecated 3-arg parse_duration calculator + retire H-018

## Context

This story is the Option 4 follow-up identified in `.factory/research/H-018-holdout-strategy-research.md`
after S-2.06 v2.0.0 pivoted away from the client-side calculator. The full lineage:

1. **S-2.06 v1.0.0** (original spec) intended to fetch Jira's configured working-hours-per-day from the
   `/configuration/timetracking` endpoint and use a 3-arg `parse_duration(input, hours_per_day, days_per_week)`
   calculator to produce a seconds total for the worklog POST.
2. **S-2.06 v2.0.0** (delivered, PR #308) pivoted to `timeSpent` string passthrough — the string
   `"1w2d3h30m"` is forwarded verbatim to Jira's API, which computes the seconds total server-side using
   its own configured working-hours. This eliminated the admin endpoint dependency and the hardcoded `(8, 5)`
   assumption.
3. The 3-arg `parse_duration` calculator was preserved in `src/duration.rs` with a `SUPERSEDED-BY` comment
   because one caller remained: the `format_roundtrip` proptest (lines 226-233 at time of S-2.06 merge).
4. **H-018** was replaced in place (Option 2) per the research-agent recommendation: its Expected clause
   now tests `parse_duration_validate("1w2d3h30m") -> Ok(())` rather than the deprecated calculator's
   seconds arithmetic. The research doc committed alongside H-018's replacement explicitly schedules this
   follow-up (S-3.10) to complete the cleanup.
5. **This story** rewrites the `format_roundtrip` proptest with a structural assertion (no inverse),
   deletes the now-callerless calculator, and deletes H-018 entirely (its user-observable contract is
   already covered by `parse_duration_validate` unit tests added by S-2.06).

## Acceptance Criteria

**AC-001** (traces to BC-X.5.005 postcondition): `src/duration.rs::format_roundtrip` proptest is
rewritten to use a structural assertion — parse the formatted output inline in the test by splitting
on `h`/`m`/`d`/`w` tokens, sum the components, and compare to input. Use `prop_map` (generate
minutes 1..1440 → multiply by 60) instead of `prop_filter` divisible-by-60 — `prop_filter` has a
~98.3% rejection rate on the original `1u64..86400` range, exceeding proptest's implicit
max-rejection budget and producing flakier shrinking. Same value space, zero rejection.

## Implementation Notes

### Structural-Assertion Proptest Sketch

Replace the `format_roundtrip` proptest body with a structural assertion (approximately 12 lines):

```rust
proptest! {
    #[test]
    fn format_roundtrip(minutes in (1u64..1440)) {
        let seconds = minutes * 60;
        let formatted = format_duration(seconds);
        // Parse formatted string back to seconds structurally (no inverse function).
        // format_duration outputs: "Xm", "Xh", "XhYm", "Xd", "XdYh", "XdYhZm", etc.
        let mut reconstructed = 0u64;
        // Walk tokens: w, d, h, m
        for (suffix, multiplier) in &[("w", 7*24*3600), ("d", 24*3600), ("h", 3600), ("m", 60u64)] {
            if let Some(pos) = formatted.find(suffix) {
                let start = formatted[..pos].rfind(|c: char| !c.is_ascii_digit()).map(|i| i + 1).unwrap_or(0);
                let n: u64 = formatted[start..pos].parse().unwrap_or(0);
                reconstructed += n * multiplier;
            }
        }
        prop_assert_eq!(reconstructed, seconds,
            "format_duration({}) = {:?} did not round-trip to same seconds", seconds, formatted);
    }
}
```

Note: the exact token-parsing logic should be adapted to match how `format_duration` actually produces
its output. Read `src/duration.rs::format_duration` at HEAD before writing the test.

### The "Code Pollution" Anti-Pattern

Preserving `parse_duration` (3-arg calculator) only because a single proptest uses it as an inverse is
the "code pollution" anti-pattern documented by Enterprise Craftsmanship
(https://enterprisecraftsmanship.com/posts/code-pollution/): production code maintained for test
infrastructure rather than for user-observable contracts. The fix is always to restructure the test,
not to preserve the production artifact.

Verify there is no other location in `.factory/` that hardcodes the holdout count 51 (run
