# S-2.01: BC-2 issue-read regression holdout suite (H-021, H-030..H-035)

## Context

The issue-read bounded context (`src/cli/issue/list.rs`, `src/cli/issue/view.rs`,
`src/api/jira/users.rs`, `src/jql.rs`) implements several behavioral contracts
that are at risk of regression during Phase 3 code changes (particularly the
Wave 0 multi-workspace fix S-0.03, and any future list.rs refactoring).

The seven holdout scenarios in this group test distinct behavioral boundaries:

- **H-021**: `issue list --status prog` with a single-substring match that maps to
  multiple statuses rejects with "Ambiguous status" and does NOT fire JQL search.
  Pre-existing coverage: `tests/issue_list_errors.rs:369`. Formally anchored here.
- **H-030**: `extract_error_message` with empty body returns the literal string
  `"<empty response body>"` (NOT a status-derived substitution). CONV-ABS-004 fix.
- **H-031**: `user search --all` continues past short non-empty pages per JRACLOUD-71293
  workaround — `start_at` advances by `USER_PAGE_SIZE` (100), NOT by returned count.
- **H-032**: `user search --all` hits `USER_PAGINATION_SAFETY_CAP` (1500) and emits
  a warning to stderr.
- **H-033**: `jr issue remote-link --url ftp://...` rejected pre-HTTP by scheme allowlist.
- **H-034**: `jr issue remote-link` URL normalization adds trailing slash via `url::Url::parse`.
- **H-035**: Combined `issue list` with all filters active — no panic, correct JQL composition.

All seven are MUST-PASS from activation HEAD dea1664. They pin correct existing behavior.

## Behavioral Contracts

**BC-7.3.001**: `extract_error_message` with a zero-length response body returns
the literal string `"<empty response body>"`. There is no status-code-derived substitution.
Empty body takes priority over all other paths (FIRST, not LAST).

**BC-2.1.012**: `--asset KEY` with multiple AQL matches rejects with "Multiple assets match"
and does NOT fire a JQL issue search.

**BC-X.7.006**: User search all-pages iterator advances `start_at` by `USER_PAGE_SIZE`
constant, not by returned-count, to work around JRACLOUD-71293.

**BC-X.2.005**: Pagination terminates on empty page, not on short page.

**BC-X.2.006**: User search --all has a `USER_PAGINATION_SAFETY_CAP` (1500); when
reached, iteration stops and a warning is emitted to stderr.

**BC-3.7.004**: Remote link URL with non-HTTP/HTTPS scheme (`ftp://`, `file://`, etc.)
is rejected with exit 64 before any HTTP calls are made.

**BC-3.7.001**: Remote link URL is normalized by `url::Url::parse` — trailing slash is
added to bare host URLs (e.g., `https://example.com` → `https://example.com/`).

**BC-2.1.007**: `--status <substring>` that matches multiple statuses rejects with
"Ambiguous status" message and does NOT fire a JQL search request.

**BC-2.1.001**: Issue list with combined filters composes correct JQL without panic.

## Acceptance Criteria

**AC-001** (traces to BC-7.3.001 postcondition): Given a wiremock server that returns
400 with an empty response body (byte length == 0) on any request, when a command that
triggers a 400 is invoked, then stderr contains the literal string `"<empty response body>"`.
No status-code text appears in place of this string. (Pins H-030.)

**AC-002** (traces to BC-X.7.006 postcondition, BC-X.2.005 postcondition): Given a
wiremock user-search endpoint that returns 100 users on page 1, 35 users on page 2,
and 100 users on page 3, and an empty page on page 4 (where page offsets are 0, 100,
200, 300), when `jr user search u --all --output json` is invoked, then the JSON
array length is 235 and no duplicates or missing entries exist. (Pins H-031.)

**AC-003** (traces to BC-X.2.006 postcondition): Given a wiremock user-search endpoint
that returns 100 users per page indefinitely, when `jr user search u --all --output json`
is invoked, then exit code is 0, the JSON array length is 1500, and stderr contains
`"hit pagination safety cap"` or equivalent warning. (Pins H-032.)

**AC-004** (traces to BC-3.7.004 postcondition): Given no wiremock (or wiremock with
`expect(0)` on all endpoints), when `jr issue remote-link FOO-1 --url ftp://example.com`
is invoked, then exit code is 64, stderr contains both `"http or https"` and `"ftp"`,
and zero HTTP calls are made. (Pins H-033.)

**AC-005** (traces to BC-3.7.001 postcondition): Given a wiremock POST `/rest/api/3/issue/PROJ-123/remotelink`,
when `jr issue remote-link PROJ-123 --url https://example.com --title "Example"` is
invoked, then the wiremock receives a request body where the `url` field is
`"https://example.com/"` (trailing slash present). (Pins H-034.)

**AC-006** (traces to BC-2.1.001 postcondition): Given a wiremock that returns project
statuses, a team list, a CMDB workspace, and 5 issues in a JQL search response, when
`jr issue list --open --assignee "Jane" --created-after "2026-01-01" --status "In Progress" --team "engineering" --output json`
is invoked, then exit code is 0, stdout is a JSON array with 5 elements, and no panic
occurs. (Pins H-035.)

**AC-007** (traces to BC-2.1.007 postcondition): Given a project with statuses
`["To Do", "In Progress", "Done"]` and a wiremock JQL search mock with `expect(0)`,
when `jr issue list --status prog` is invoked in a project context with no `--no-input`
flag (default interactive path), then exit code is 64, stderr contains `"Ambiguous
status"` and both `"In Progress"` and `"Done"` (all matching statuses), and the JQL
search mock is NOT invoked. (Pins H-021. Pre-existing coverage confirmed at
`tests/issue_list_errors.rs:369` (`issue_list_status_single_substring_rejected`),
activation HEAD dea1664.)

## Implementation Notes

### H-030 (empty body extract_error_message)

The `extract_error_message` function is in `src/api/client.rs` (or similar; verify actual
location). The test invokes a command against a wiremock returning 400 + empty body.

```rust
#[tokio::test]
async fn test_extract_error_message_empty_body() {
    let server = MockServer::start().await;
    Mock::given(method("POST"))
        .and(path("/rest/api/3/issue"))
        .respond_with(ResponseTemplate::new(400).set_body_bytes(b""))
        .mount(&server)
        .await;
    // Spawn: jr issue create --summary "x" ... --no-input
    // Assert stderr contains "<empty response body>"
}
```

### H-031 / H-032 (user search pagination)

Use library-level test via `JiraClient::new_for_test` if `list_users` is accessible,
or process-spawn with mock server. The key invariant: page 2 has 35 users (short page),
but the NEXT request still starts at `200` (not `135`). This is the JRACLOUD-71293
workaround.

For H-032: use `wiremock::Respond` with a responder that always returns 100 users
regardless of offset. After `USER_PAGINATION_SAFETY_CAP / USER_PAGE_SIZE` iterations,
the client should stop. Assert total = 1500 and stderr contains the safety-cap warning.

### H-035 (combined filter)

This test requires multiple mocks mounted simultaneously:
1. `GET /rest/api/3/project/PROJ/statuses` — return statuses including "In Progress"
2. Some endpoint for team list (check `src/cli/issue/list.rs` for exact endpoint)
3. Assets workspace discovery endpoint
4. `POST /rest/api/3/search/jql` — return 5 issues

Mount all mocks, then spawn the `jr issue list --open --assignee ...` command.
Assert exit 0 and JSON array length 5.

  the source constant.

  unbounded `wiremock` responder that returns 100 users regardless of offset. Confirm
  `wiremock` supports stateless repeated responses (it does via `Respond` trait).
