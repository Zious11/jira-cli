# S-2.02: BC-3 issue-write regression holdout suite (H-006, H-007, H-008, H-014)

## Context

The issue-write bounded context (`src/cli/issue/workflow.rs`, `src/cli/issue/create.rs`,
`src/cli/issue/helpers.rs`) implements several state-changing behavioral contracts
that are at risk of regression during Phase 3 code changes (particularly the
multi-profile fields fix from Wave 0 S-0.04, and any future refactoring of
`workflow.rs`).

The four holdout scenarios in this group cover critical idempotency, error messaging,
JQL short-circuit, and disambiguation behaviors:

- **H-006**: `issue move FOO-1 "In Progress"` is idempotent — if already in target
  state, exits 0 with `"transitioned": false` and makes NO POST to transitions.
- **H-007**: `issue move FOO-1 Done` when transition returns 400 with resolution
  required — stderr must contain both `--resolution` AND `jr issue resolutions`.
- **H-008**: `issue list --status prog` with a single-substring ambiguous match
  errors before firing JQL search (short-circuit validation).
- **H-014**: `assign --to <name>` with duplicate display names + `--no-input` errors
  with both emails AND both accountIds in stderr.

All four are MUST-PASS from activation HEAD dea1664. They pin correct existing behavior.

## Behavioral Contracts

**BC-3.2.001**: `issue move <key> <status>` is idempotent. If the issue is already
in the target status, exit 0, return `{"transitioned": false}`, and make ZERO POST
requests to the transitions endpoint.

**BC-3.2.009**: When a transition fails with 400 because a `resolution` field is required,
the error message in stderr contains both `--resolution` (the flag to use) and
`jr issue resolutions` (the command to list valid values).

**BC-2.1.013**: `--status <substring>` that matches multiple statuses (ambiguous) rejects
with "Ambiguous status" and does NOT make a JQL search call.

**BC-X.7.004**: When user search returns multiple users with the same `displayName`
and `--no-input` is set, the error message contains both emails AND both accountIds
to enable the AI-agent caller to retry with an unambiguous identifier.

## Acceptance Criteria

**AC-001** (traces to BC-3.2.001 postcondition): Given a wiremock that returns
`GET /rest/api/3/issue/FOO-1` with `status.name = "In Progress"`, and a transitions
mock with `expect(0)` on POST, when `jr issue move FOO-1 "In Progress" --output json`
is invoked, then exit code is 0, stdout JSON contains `"transitioned": false`, and the
POST transitions mock is NOT invoked. (Pins H-006.)

**AC-002** (traces to BC-3.2.009 postcondition): Given a wiremock that (a) returns a
transitions list including "Done" and (b) returns 400 with body
`{"errors": {"resolution": "Field 'resolution' is required"}}` on POST to transitions,
when `jr --no-input issue move FOO-1 Done` is invoked, then exit code is non-zero and
stderr contains both `--resolution` and `jr issue resolutions`. (Pins H-007.)

**AC-003** (traces to BC-2.1.013 postcondition): Given a project with statuses
`["To Do", "In Progress", "Done"]` and a wiremock JQL search mock with `expect(0)`,
when `jr --no-input issue list --status prog` is invoked in a project context, then
exit code is 64, stderr contains `"Ambiguous status"` and `"In Progress"`, and the
JQL search mock is NOT invoked. (Pins H-008, note: H-008 is identical in behavior to
H-021 — both pin the same BC-2.1.013 short-circuit; H-021 is covered in S-1.06.)

**AC-004** (traces to BC-X.7.004 postcondition): Given a wiremock assignable-user-search
endpoint returning two users with identical `displayName = "John Smith"` but different
`emailAddress` and `accountId` values, when `jr issue assign FOO-1 --to "John Smith" --no-input`
is invoked, then exit code is non-zero and stderr contains both email addresses AND both
accountIds. (Pins H-014.)

## Holdout Strategy

All four holdouts are MUST-PASS at activation HEAD dea1664. These are behavioral-pinning
tests on existing correct code. If any test fails, a pre-existing regression exists.

Verification:
- `cargo test --test issue_write_holdouts` passes with zero failures.
- Tests are NOT `#[ignore]`-gated.
- AC-001: wiremock with `expect(0)` on the POST endpoint — verifies no HTTP call.
- AC-003: wiremock with `expect(0)` on JQL search — verifies short-circuit.

## Implementation Notes

### H-006 (idempotent move)

Critical detail: wiremock `expect(0)` on the POST transitions endpoint enforces that
NO HTTP call is made after the idempotency check.

```rust
#[tokio::test]
async fn test_issue_move_idempotent_when_already_in_target() {
    let server = MockServer::start().await;
    // GET /issue/FOO-1 → status.name = "In Progress"
    // POST /issue/FOO-1/transitions → expect(0)
    let response = Command::cargo_bin("jr")
        .unwrap()
        .env("JR_BASE_URL", server.uri())
        .args(["issue", "move", "FOO-1", "In Progress", "--output", "json"])
        .assert()
        .success();
    // Parse stdout JSON, assert "transitioned": false
    server.verify().await;  // confirms POST was NOT called
}
```

### H-007 (resolution required error)

The wiremock must mock two endpoints:
1. `GET /rest/api/3/issue/FOO-1/transitions` → list including "Done"
2. `POST /rest/api/3/issue/FOO-1/transitions` → 400 with resolution error body

The test asserts stderr contains both strings. Check exact CLI flag name:
`--resolution` and exact command name: `jr issue resolutions`.

### H-008 (ambiguous status short-circuit)

Requires a mock for the statuses endpoint to provide the project status list.
The JQL search mock with `expect(0)` is the critical assertion — it proves the
short-circuit fired before the search.

Also requires a `.jr.toml` in the test working directory or a `--project PROJ` flag.
Use a temp directory with a `.jr.toml` file containing `project = "PROJ"`.

### H-014 (duplicate display names)

The user disambiguation error message format must contain:
- `email1` AND `email2`
- `accountId1` AND `accountId2`

Use the `assert_cmd` output with `contains()` on the stderr bytes.
