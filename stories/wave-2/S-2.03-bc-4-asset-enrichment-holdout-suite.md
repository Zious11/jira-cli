# S-2.03: BC-4 assets/CMDB regression holdout suite (H-037, H-038, H-039)

## Context

The assets/CMDB bounded context (`src/api/assets/`, `src/cli/assets.rs`,
`src/cli/issue/list.rs` enrichment pipeline) implements caching and enrichment
behaviors that are easy to regress silently. The S-0.03 Wave 0 fix (multi-workspace
HashMap composite key) addressed the MUST-FIX bug in H-036. This story pins the
three MEDIUM-priority asset holdouts that are separate from the S-0.03 fix:

- **H-037**: `assets search` workspace discovery is cached — a second call fires NO
  HTTP calls to the workspace discovery endpoint. Pin the cache-hit path.
- **H-038**: `enrich_assets` skips GET for assets that already have `id + key + name`
  populated. Only id-only assets are fetched. Pin the skip-already-resolved invariant.
- **H-039**: `assets tickets --status PROG` with two matching statuses ("In Progress"
  and "Progressing") errors with "Ambiguous status" and both candidates in stderr.

All three are MUST-PASS from activation HEAD dea1664. They pin correct existing behavior
distinct from the H-036 MUST-FIX.

## Behavioral Contracts

**BC-4.2.001**: The workspace ID is cached after the first discovery call. Subsequent
calls within the 7-day TTL window return the cached ID without making an HTTP request
to the workspace discovery endpoint.

**BC-4.3.002**: In `enrich_assets`, assets that already have both `key` and `name`
populated skip the GET fetch step. Only assets with id-only data are enriched via HTTP.

**BC-4.2.006**: `assets tickets --status <substring>` that matches multiple status
strings rejects with "Ambiguous status" and lists all candidates in stderr.

## Acceptance Criteria

**AC-001** (traces to BC-4.2.001 postcondition): Given a wiremock workspace discovery
endpoint that is mounted only during the first invocation (removed before the second),
when `jr assets search "Key = X"` is invoked a second time, then exit code is 0 and
no HTTP call reaches the workspace discovery endpoint. The result is served from cache.
(Pins H-037.)

**AC-001-alternate** (traces to BC-4.2.001 postcondition): Given a wiremock workspace
discovery endpoint with `expect(1)`, when `jr assets search "Key = X"` is invoked
TWICE, then the workspace discovery endpoint receives exactly ONE call total (first
invocation hits HTTP; second hits cache). Both invocations exit 0. (Pins H-037,
alternate strategy using `expect(1)` across two process-spawn calls.)

**AC-002** (traces to BC-4.3.002 postcondition): Given a `LinkedAsset` list containing:
(a) an asset with only `id` set, and (b) an asset with `id + key + name` already set,
and a wiremock GET on the asset endpoint with `expect(1)`, when the enrichment pipeline
is invoked, then only asset (a) is fetched, and asset (b)'s `key` and `name` are
unchanged. The GET endpoint is called exactly once. (Pins H-038.)

**AC-003** (traces to BC-4.2.006 postcondition): Given connected tickets for an asset
with statuses `["In Progress", "Progressing"]`, when
`jr assets tickets OBJ-1 --status PROG` is invoked, then exit code is 64, stderr
contains `"Ambiguous status"`, and stderr contains both `"In Progress"` and
`"Progressing"`. (Pins H-039.)

## Holdout Strategy

All three holdouts are MUST-PASS at activation HEAD dea1664.

- **H-037**: Tests the cache layer. Requires two consecutive invocations OR using
  `expect(1)` across both. The cache files are written to `XDG_CACHE_HOME` (temp dir
  per-test). The second invocation must reuse the cache file.
- **H-038**: Tests the enrichment pipeline at the library level if accessible, or via
  a combined `issue list` invocation where some assets are pre-enriched in the
  fixture. If library-level access is not available, use a contrived issue list
  scenario.
- **H-039**: Tests the `assets tickets` subcommand status disambiguation.

Verification:
- `cargo test --test asset_holdouts` passes with zero failures.
- Tests are NOT `#[ignore]`-gated.

## Implementation Notes

### H-037 (workspace cache)

**Strategy A** (preferred if `XDG_CACHE_HOME` is isolated per test):
1. Set `XDG_CACHE_HOME` to a temp dir.
2. Mount workspace discovery endpoint on wiremock with `expect(1)`.
3. Spawn `jr assets search "Key = X"` twice in the same temp cache dir.
4. Call `server.verify().await` — confirms exactly 1 HTTP call despite 2 invocations.

**Strategy B** (if multi-invocation is complex):
Write the cache file to the temp `XDG_CACHE_HOME` manually before the second invocation,
then confirm the second invocation does NOT call the workspace endpoint.

Note: the workspace cache file lives at `~/.cache/jr/v1/<profile>/workspace_id.json`
(per `src/api/assets/workspace.rs` and `cache.rs`). In the test, use profile "default".

### H-038 (skip-already-resolved)

The `enrich_assets` function is in `src/api/assets/linked.rs`. If it is accessible as
a library-level function (via `src/lib.rs` re-export), test it directly:

```rust
#[tokio::test]
async fn test_enrich_assets_skips_already_resolved() {
    let server = MockServer::start().await;
    // Mount GET /assets/object/<id-only-asset> with expect(1)
    // Mount GET /assets/object/<id-key-name-asset> with expect(0)
    let assets = vec![
        LinkedAsset { id: "id-A".to_string(), key: None, name: None },
        LinkedAsset { id: "id-B".to_string(), key: Some("B-001".to_string()), name: Some("Asset B".to_string()) },
    ];
    let result = enrich_assets(&client, assets).await.unwrap();
    server.verify().await;  // only id-A fetched
    // Assert: id-A now has key/name; id-B unchanged
}
```

If `enrich_assets` is `pub(crate)`, this test must be in `src/api/assets/linked.rs`
inline (`#[cfg(test)]` module), not in `tests/`.

### H-039 (ambiguous status in assets tickets)

