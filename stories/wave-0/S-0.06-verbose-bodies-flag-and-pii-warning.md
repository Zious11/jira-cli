# S-0.06: Add `--verbose-bodies` flag with PII warning; make `--verbose` header-only by default

## Context

`--verbose` currently prints the full HTTP request/response body to stderr. Bodies contain PII: ADF text content, account IDs, email addresses, and any issue field value. This is a privacy risk in:

1. CI/CD shared terminal sessions
2. AI-agent contexts (this session) where stderr is captured in transcripts
3. Debug log files piped from shared terminals

Security Decision SD-003 was resolved at the Phase 1→2 gate (2026-05-04) with Option B: header-only default + opt-in body logging via `--verbose-bodies`.

This is a **breaking change**: developers relying on `--verbose` to see request bodies must migrate to `--verbose-bodies`.

The fix is a two-site modification in `src/api/client.rs` plus a new clap flag in `src/cli/mod.rs`.

## Behavioral Contract

**BC-X.1.001** (cross-cutting.md) + **SD-003 Resolution**:
> Default `--verbose` shows: `[verbose] {METHOD} {URL}` + response status only (no body).
> `--verbose-bodies` enables body logging (explicit opt-in) with a PII warning to stderr.

## Acceptance Criteria

**AC-001** (traces to SD-003 resolution postcondition): When `--verbose` is set without `--verbose-bodies`, stderr contains `[verbose] {METHOD} {URL}` lines but does NOT contain `[verbose] body:` lines.

**AC-002** (traces to SD-003 resolution postcondition): When `--verbose` is set without `--verbose-bodies`, stderr DOES contain a suppression hint: `[verbose] body suppressed (use --verbose-bodies to inspect, will print PII)`.

**AC-003** (traces to SD-003 resolution postcondition): When `--verbose-bodies` is set, stderr contains `[jr] WARNING: --verbose-bodies prints request/response bodies to stderr.` before any body content.

**AC-004** (traces to SD-003 resolution postcondition): When `--verbose-bodies` is set, stderr contains `[verbose] body: <content>` for each request/response body.

**AC-005** (traces to SD-003 resolution postcondition): `--verbose-bodies` without `--verbose` is a no-op on the method/URL logging (only bodies are printed, with the warning). That is, `--verbose-bodies` implies body printing; combined with `--verbose` it prints method+URL+bodies+warning.

**AC-006** (traces to SD-003 breaking change): Release notes entry is included in `CHANGELOG.md` or equivalent documenting the breaking change and migration path.

## Holdout Strategy

Per SD-003 resolution:
- **MUST-PASS (new holdout 1)**: `--verbose-bodies` emits the PII warning to stderr
- **MUST-PASS (new holdout 2)**: `--verbose` alone does NOT print body content
- **MUST-FAIL (regression)**: if a future change inadvertently prints bodies under `--verbose` only, the MUST-PASS holdouts catch it

These holdouts are defined inline in the test plan; they do not require a separate S-0.07-style spec story.

## Implementation Notes

### New clap flag in `src/cli/mod.rs`

In the global CLI struct (where `--verbose` is already defined), add:

```rust
/// Log request and response bodies to stderr. Implies --verbose-level body output.
/// WARNING: bodies contain PII (accountId, emailAddress, ADF content).
/// Do not use in shared terminals or AI-agent contexts.
#[arg(long, global = true)]
pub verbose_bodies: bool,
```

Update `--verbose` help text:
```rust
/// Verbose mode: log HTTP method, URL, and response status to stderr.
/// Does NOT log request/response bodies (use --verbose-bodies for that).
#[arg(short = 'v', long, global = true)]
pub verbose: bool,
```

### Two body-printing sites in `src/api/client.rs`

**Site 1** (`send` method, lines 197-201):

Current:
```rust
if self.verbose {
    eprintln!("[verbose] {} {}", r.method(), r.url());
    if let Some(bytes) = r.body().and_then(|b| b.as_bytes()) {
        eprintln!("[verbose] body: {}", String::from_utf8_lossy(bytes));
    }
}
```

Target:
```rust
if self.verbose || self.verbose_bodies {
    eprintln!("[verbose] {} {}", r.method(), r.url());
}
if self.verbose_bodies {
    if let Some(bytes) = r.body().and_then(|b| b.as_bytes()) {
        eprintln!("[verbose] body: {}", String::from_utf8_lossy(bytes));
    }
} else if self.verbose {
    eprintln!("[verbose] body suppressed (use --verbose-bodies to inspect, will print PII)");
}
```

**Site 2** (`send_raw` method, lines 274-278): Same pattern as site 1.

### PII warning placement

Emit the warning once at client construction or at the first `verbose_bodies` log, not on every request. Add to `JiraClient::from_config`:

```rust
if verbose_bodies {
    eprintln!("[jr] WARNING: --verbose-bodies prints request/response bodies to stderr.");
    eprintln!("[jr] These bodies contain PII (accountId, emailAddress, ADF text content).");
    eprintln!("[jr] Do not pipe to AI-agent contexts or shared logs without consent.");
}
```

### `JiraClient` struct update

Add `verbose_bodies: bool` field to `JiraClient` (alongside existing `verbose: bool`). Pass it through `from_config` and `new_for_test` (default `false` in `new_for_test`).

### CHANGELOG entry

Add to `CHANGELOG.md` (or create if absent):
```
## BREAKING CHANGE (v0.6)

**--verbose no longer prints HTTP request/response bodies by default.**

Use `--verbose-bodies` for full body inspection. The new flag emits a PII warning.

Rationale: prevents accidental PII leakage in shared terminals, debug log files,
and AI-agent context windows. See SD-003 for details.

Migration: replace `--verbose` with `--verbose-bodies` in any script that relied
on body output.
```

## Architecture Compliance Rules

- `--verbose` and `--verbose-bodies` are both `global = true` flags in the clap CLI struct.
- The `JiraClient` struct must carry both `verbose: bool` and `verbose_bodies: bool`.
- `new_for_test` defaults `verbose_bodies: false` — no test should see unexpected PII warnings.
- Body printing is gated on `verbose_bodies` only (not on `verbose` alone). This is the invariant; do not invert it.
- Reference: `architecture/security-decisions/SD-003-verbose-pii-redaction.md`.

## Risk / Dependencies

- **Breaking change.** Users/scripts using `--verbose` for body inspection must add `--verbose-bodies`. The suppression hint in stderr (AC-002) makes the migration path discoverable.
- **No other Wave 0 stories depend on this.** Independent.
- **Risk (LOW):** Any existing test that asserts on `[verbose] body:` in stderr will fail after this change. Update those tests to use `verbose_bodies=true`.
- **Risk (LOW):** If `JiraClient::new_for_test` does not currently accept a `verbose` parameter, `verbose_bodies` should default to `false` in that constructor. No behavior change for existing tests.
