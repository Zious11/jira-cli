# S-0.05: Gate `JR_AUTH_HEADER` env var read behind `#[cfg(test)]`

## Context

`JiraClient::from_config` reads the `JR_AUTH_HEADER` environment variable unconditionally at `src/api/client.rs:64-66`. This means the env var is active in production release binaries. Any process that inherits `JR_AUTH_HEADER` from its parent (CI runner, shell script, test harness) can bypass keychain authentication entirely.

Security Decision SD-002 was resolved at the Phase 1→2 gate (2026-05-04) with Option A: wrap the env-var read in `#[cfg(test)]`. The code excluded from release binaries eliminates the runtime exploitation vector categorically.

**Important:** `JiraClient::new_for_test(base_url, auth_header)` — the primary integration test constructor — passes the auth header directly as a constructor argument and does NOT use `JR_AUTH_HEADER`. Most integration tests already use `new_for_test`. The `#[cfg(test)]` gate affects only the narrow code path in `from_config` that reads the env var.

This story also sets up the holdout H-NEW-AUTH-002 (formalized in S-0.07) by verifying the `#[cfg(test)]` gate works correctly.

## Behavioral Contract

**SD-002 Resolution** (primary anchor for this story):
> Wrap `src/api/client.rs:64-66` env-var read in `#[cfg(test)]`. `JR_AUTH_HEADER` is a compile-time test-only mechanism excluded from release binaries.

**BC-X.1.001** (`cross-cutting.md`) — context only, not primary anchor:
> Auth header is injected on every API call via `req.header("Authorization", &self.auth_header)` at line 195. Header value is verbatim auth string.
>
> Note: BC-X.1.001 does not itself mandate keychain-only in release mode — that constraint
> comes from SD-002's resolution. The ACs below trace to SD-002, not to BC-X.1.001 postcondition.

## Acceptance Criteria

**AC-001** (traces to SD-002 resolution): In `#[cfg(test)]` code (integration tests compiled with `cargo test`), setting `JR_AUTH_HEADER=fake` causes `JiraClient::from_config` to use `fake` as the auth header value. Existing integration tests using `JR_AUTH_HEADER` directly (if any) continue to work.

**AC-002** (traces to SD-002 resolution — H-NEW-AUTH-002 setup): In `#[cfg(not(test))]` context (release build, `cargo build --release`), the `JR_AUTH_HEADER` env var read block does NOT exist in the compiled binary. This is verified by inspecting `cfg!(test)` == false in a test compiled in release mode, or by confirming via a test that the code path is unreachable.

**AC-003** (traces to SD-002 resolution): After the fix, `JiaClient::new_for_test(base_url, auth_header)` continues to work identically. No change to the test constructor.

**AC-004** (traces to SD-002 precondition): Audit `tests/` for any test that sets `JR_AUTH_HEADER` as a bare env var (not via `new_for_test`). If found, migrate to `new_for_test`. Document the result in this story's implementation notes.

## Holdout Strategy

- **H-NEW-AUTH-002** (formalized in S-0.07) targets this story's implementation.
- H-NEW-AUTH-002 MUST-FAIL pre-fix (current code reads `JR_AUTH_HEADER` in release)
- H-NEW-AUTH-002 MUST-PASS post-fix (`JR_AUTH_HEADER` not active in release build)
- The holdout test: build `jr` with `cargo build --release`; set `JR_AUTH_HEADER=fake`; run a `jr` command that requires auth; assert it does NOT use `fake` (returns a keychain error, not a fake-auth API call)

## Implementation Notes

**Step 1: Audit `JR_AUTH_HEADER` usage in tests**

Run:
```
grep -r "JR_AUTH_HEADER" tests/ src/
```

Expected result: Only `src/api/client.rs:64-66` references it in non-test code. Test files that use it directly (not via `new_for_test`) must be migrated first.

**Step 2: Wrap the env-var read**

Current code (`src/api/client.rs:64-66`):
```rust
// JR_AUTH_HEADER env var overrides keychain auth (used by tests to inject mock auth)
let auth_header = if let Ok(header) = std::env::var("JR_AUTH_HEADER") {
    header
} else {
    match auth_method {
        // ... keychain lookup
    }
};
```

Target code:
```rust
// JR_AUTH_HEADER is test-only: excluded from release builds via #[cfg(test)] gate.
let auth_header = {
    #[cfg(test)]
    if let Ok(header) = std::env::var("JR_AUTH_HEADER") {
        header
    } else {
        Self::load_auth_from_keychain(auth_method, &config.active_profile_name)?
    }
    #[cfg(not(test))]
    Self::load_auth_from_keychain(auth_method, &config.active_profile_name)?
};
```

Alternatively, use a simpler conditional expression:
```rust
#[cfg(test)]
let auth_header = if let Ok(header) = std::env::var("JR_AUTH_HEADER") {
    header
} else {
    // keychain lookup inline
    match auth_method { ... }
};
#[cfg(not(test))]
let auth_header = match auth_method { ... };
```

Choose the implementation that minimizes code duplication. The keychain lookup block can be extracted to a helper `fn build_auth_header(auth_method: &str, profile_name: &str) -> anyhow::Result<String>` to avoid duplicating the oauth/api_token match arms.

**Rust `check-cfg` note (Rust 1.80+):** If the build emits `unexpected_cfgs` warnings after this change, add `cargo::rustc-check-cfg=cfg(test)` to `build.rs` or check if it is already handled. `#[cfg(test)]` is a built-in Rust attribute and does not need `check-cfg` registration — this note is only relevant if a future linter adds warnings for it.

## Architecture Compliance Rules

- `#[cfg(test)]` is the ONLY acceptable gate. Do not use `#[cfg(debug_assertions)]` or a runtime env var check — only `#[cfg(test)]` excludes the code from release binaries (SD-002 rationale).
- `JiraClient::new_for_test` must not be modified. It is the canonical integration test constructor.
- The `JR_BASE_URL` override (separate from `JR_AUTH_HEADER`) remains ungated — it is a URL target override for wiremock, not an auth bypass, and has different risk profile.
- Reference: `architecture/security-decisions/SD-002-jr-auth-header-prod-gating.md`.
