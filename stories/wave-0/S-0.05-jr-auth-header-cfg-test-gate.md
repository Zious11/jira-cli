# S-0.05: Gate `JR_AUTH_HEADER` env var read behind `#[cfg(debug_assertions)]` (canonized 2026-05-07)

## Context

`JiraClient::from_config` reads the `JR_AUTH_HEADER` environment variable unconditionally at `src/api/client.rs:64-66`. This means the env var is active in production release binaries. Any process that inherits `JR_AUTH_HEADER` from its parent (CI runner, shell script, test harness) can bypass keychain authentication entirely.

Security Decision SD-002 was resolved at the Phase 1→2 gate (2026-05-04) with Option A (`#[cfg(test)]`). During implementation (2026-05-07), Red Gate analysis found ~151 subprocess integration tests using `Command::cargo_bin("jr").env("JR_AUTH_HEADER", ...)`. These subprocess binaries run without `cfg(test)` active, so Option A would break them all. SD-002 was canonized to Option B-revised: `#[cfg(debug_assertions)]`. This achieves identical release-binary security with zero test migration cost. See SD-002 for full canonization rationale.

**Important:** `JiraClient::new_for_test(base_url, auth_header)` — the primary integration test constructor — passes the auth header directly as a constructor argument and does NOT use `JR_AUTH_HEADER`. Most integration tests already use `new_for_test`. The `#[cfg(debug_assertions)]` gate affects only the narrow code path in `from_config` that reads the env var.

This story also sets up the holdout H-NEW-AUTH-002 (formalized in S-0.07) by verifying the `#[cfg(debug_assertions)]` gate works correctly.

## Behavioral Contract

**SD-002 Resolution** (primary anchor for this story, canonized 2026-05-07):
> Wrap `src/api/client.rs:64-66` env-var read in `#[cfg(debug_assertions)]`. `JR_AUTH_HEADER` is a debug-build-only mechanism excluded from release binaries. (`#[cfg(test)]` was the original gate but was changed to `#[cfg(debug_assertions)]` during implementation to preserve ~151 subprocess integration tests — see SD-002.)

**BC-X.1.001** (`cross-cutting.md`) — context only, not primary anchor:
> Auth header is injected on every API call via `req.header("Authorization", &self.auth_header)` at line 195. Header value is verbatim auth string.
>
> Note: BC-X.1.001 does not itself mandate keychain-only in release mode — that constraint
> comes from SD-002's resolution. The ACs below trace to SD-002, not to BC-X.1.001 postcondition.

## Acceptance Criteria

**AC-001** (traces to SD-002 resolution): In `#[cfg(debug_assertions)]` context (debug builds — `cargo test`, `cargo build`, `cargo_bin`), setting `JR_AUTH_HEADER=fake` causes `JiraClient::from_config` to use `fake` as the auth header value. Existing integration tests using `JR_AUTH_HEADER` via subprocess `.env()` calls continue to work.

**AC-002** (traces to SD-002 resolution — H-NEW-AUTH-002 setup): In `#[cfg(not(debug_assertions))]` context (release build, `cargo build --release`), the `JR_AUTH_HEADER` env var read block does NOT exist in the compiled binary. This is verified by confirming that the `#[cfg(debug_assertions)]` gate is present in source (source-inspection test) and by the H-NEW-AUTH-002 holdout process test.

**AC-003** (traces to SD-002 resolution): After the fix, `JiraClient::new_for_test(base_url, auth_header)` continues to work identically. No change to the test constructor.

**AC-004** (traces to SD-002 precondition): Audit `tests/` for any test that sets `JR_AUTH_HEADER` as a bare env var (not via `new_for_test`). If found, migrate to `new_for_test`. Document the result in this story's implementation notes.

## Holdout Strategy

- **H-NEW-AUTH-002** (formalized in S-0.07) targets this story's implementation.
- H-NEW-AUTH-002 MUST-FAIL pre-fix (current code reads `JR_AUTH_HEADER` in release)
- H-NEW-AUTH-002 MUST-PASS post-fix (`JR_AUTH_HEADER` not active in release build)
- The holdout test: build `jr` with `cargo build --release`; set `JR_AUTH_HEADER=fake`; run a `jr` command that requires auth; assert it does NOT use `fake` (returns a keychain error, not a fake-auth API call)

## Implementation Notes

**Step 1: Audit `JR_AUTH_HEADER` usage in tests**

Run:
```
grep -r "JR_AUTH_HEADER" tests/ src/
```

Expected result: Only `src/api/client.rs:64-66` references it in non-test code. Test files that use it directly (not via `new_for_test`) must be migrated first.

**Step 2: Wrap the env-var read**

Current code (`src/api/client.rs:64-66`):
```rust
// JR_AUTH_HEADER env var overrides keychain auth (used by tests to inject mock auth)
let auth_header = if let Ok(header) = std::env::var("JR_AUTH_HEADER") {
    header
} else {
    match auth_method {
        // ... keychain lookup
    }
};
```

Target code (canonized gate: `#[cfg(debug_assertions)]`):
```rust
// JR_AUTH_HEADER is debug-only: excluded from release builds via #[cfg(debug_assertions)] gate.
// Preserves subprocess integration tests (cargo_bin → debug binary, debug_assertions=true).
#[cfg(debug_assertions)]
if let Ok(header) = std::env::var("JR_AUTH_HEADER") {
    return Ok(header);
}
// Release builds (debug_assertions=false) always use keychain lookup.
```

Choose the implementation that minimizes code duplication. The keychain lookup block can be extracted to a helper `fn build_auth_header(auth_method: &str, profile_name: &str) -> anyhow::Result<String>` to avoid duplicating the oauth/api_token match arms.

**Rust `check-cfg` note (Rust 1.80+):** `#[cfg(debug_assertions)]` is a built-in Rust attribute and does not need `check-cfg` registration. No `build.rs` changes are needed.

## Architecture Compliance Rules

- `#[cfg(debug_assertions)]` is the canonized gate (SD-002 canonization 2026-05-07). Do not use a runtime env var check. Do not use `#[cfg(test)]` — it excludes from release binaries but also breaks ~151 subprocess integration tests. `#[cfg(debug_assertions)]` achieves the identical release-binary security goal while preserving subprocess test compatibility. See SD-002 for full rationale.
- `JiraClient::new_for_test` must not be modified. It is the canonical integration test constructor.
- The `JR_BASE_URL` override (separate from `JR_AUTH_HEADER`) remains ungated — it is a URL target override for wiremock, not an auth bypass, and has different risk profile.
- Reference: `architecture/security-decisions/SD-002-jr-auth-header-prod-gating.md`.
