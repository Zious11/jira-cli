# S-0.02: Paginate `list_worklogs` to prevent silent truncation

## Context

`JiraClient::list_worklogs` in `src/api/jira/worklogs.rs:26-30` fetches exactly one `OffsetPage<Worklog>` and discards the `total`/`start_at`/`max_results` fields. The Jira API returns at most 50 worklogs per page. Issues with >50 worklogs silently return only the first 50.

This is a data-correctness bug for any team that logs work on long-running issues (epics, support tickets, recurring tasks). The user sees 50 worklogs when 80 (or more) exist, with no warning.

BC-X.5.002 describes the fixed (target) behavior. H-045 is the MUST-FAIL holdout pinning the bug; it becomes MUST-PASS after this story ships.

## Behavioral Contract

**BC-X.5.002** (`cross-cutting.md`):
> `client.list_worklogs(key)` MUST paginate in a loop until `page.total <= page.start_at + page.items().len()`. All pages concatenated and returned to caller. No silent truncation.

## Acceptance Criteria

**AC-001** (traces to BC-X.5.002 postcondition): Given wiremock returns page 1 with 50 worklogs (`total: 80, startAt: 0, maxResults: 50`) and page 2 with 30 worklogs (`total: 80, startAt: 50, maxResults: 50`), when `jr worklog list PROJ-1 --output json` is executed, then the JSON array length is 80.

**AC-002** (traces to BC-X.5.002 postcondition): Both pages are fetched from the server (wiremock mock with `expect(2)` for the worklog endpoint).

**AC-003** (traces to BC-X.5.002 postcondition): For a single-page result (total=20, one page), only one HTTP request is made (no spurious second-page fetch).

**AC-004** (traces to BC-X.5.002 postcondition): For an empty issue (total=0), zero items returned with no HTTP errors.

## Holdout Strategy

## Implementation Notes

**Current code** (`src/api/jira/worklogs.rs:26-30`):
```rust
pub async fn list_worklogs(&self, key: &str) -> Result<Vec<Worklog>> {
    let path = format!("/rest/api/3/issue/{}/worklog", urlencoding::encode(key));
    let page: OffsetPage<Worklog> = self.get(&path).await?;
    Ok(page.items().to_vec())
}
```

**Target code** (pagination loop):
```rust
pub async fn list_worklogs(&self, key: &str) -> Result<Vec<Worklog>> {
    let base_path = format!("/rest/api/3/issue/{}/worklog", urlencoding::encode(key));
    let mut all_items: Vec<Worklog> = Vec::new();
    let mut start_at: usize = 0;

    loop {
        let path = format!("{}?startAt={}", base_path, start_at);
        let page: OffsetPage<Worklog> = self.get(&path).await?;
        let count = page.items().len();
        all_items.extend_from_slice(page.items());

        let fetched_up_to = start_at + count;
        if page.total() <= fetched_up_to || count == 0 {
            break;
        }
        start_at = fetched_up_to;
    }

    Ok(all_items)
}
```

**`OffsetPage<T>` contract**: The `OffsetPage` struct (from `src/api/pagination.rs`) already exposes `items()`, and should expose `total()` and `start_at()` accessors. Verify these methods exist; add them if absent (they are fields already present in the JSON response).

**Pagination termination condition**: `page.total <= page.start_at + page.items().len()` per BC-X.5.002. The `count == 0` guard handles the edge case where `total` is incorrectly reported by Jira (defensive).
