# S-0.04: Migrate 14 field-read sites to `config.active_profile()` (multi-profile fields MUST-FIX)

## Context

CRITICAL production bug. Every invocation of `jr auth login`, `jr auth switch`, or `jr init` calls `Config::save_global()`. After that save, the legacy `[fields]` block is dropped from disk because it is marked `#[serde(default, skip_serializing)]` in `src/config.rs:43-48`. All 14 hot-path read sites that reference `config.global.fields.story_points_field_id` or `config.global.fields.team_field_id` then observe `None`.

**Visible symptoms:**
- `jr issue list --points` — points column blank
- `jr issue list` — Team column disappears
- `jr issue view <KEY>` — points and team missing
- `jr sprint current` — points + team missing
- `jr board view` — team column missing
- `jr issue create --points 5` — fails with stale error message referencing `[fields]`
- Multi-profile users: silently get the wrong field IDs when switching profiles

The fix pattern (per ADR-0007): replace every `config.global.fields.X` read with `config.active_profile().X` (or `config.active_profile_or_err()?.X`). The `Config::field_id(FieldKind, profile)` accessor introduced in ADR-0007 is the preferred call site for new code.

Additionally, update two pinned error message strings that reference the deprecated `[fields]` section.

## Behavioral Contract

**BC-6.3.001** (`bc-6-config-cache.md`):
> For all profiles P and all field-id pairs (sp, team):
> After `config.save_global()` followed by `Config::load_with(Some(P))`:
> - `config.active_profile().story_points_field_id` MUST == `Some(sp)`
> - `config.active_profile().team_field_id` MUST == `Some(team)`
>
> AND every read site in the 14-entry table MUST observe (sp, team) from `active_profile()`, not from `global.fields.*`.
>
> Error messages MUST reference `[profiles.<name>]` not deprecated `[fields]`.

## Acceptance Criteria

**AC-001** (traces to BC-6.3.001 postcondition): Given two profiles `prod` (story_points_field_id = "customfield_10005") and `sandbox` (story_points_field_id = "customfield_10099"), when `jr --profile sandbox issue create --summary "Test" --story-points 5 --type Story --project PROJ --no-input` is executed, then the POST body contains `"customfield_10099": 5` and does NOT contain `"customfield_10005"`.

**AC-002** (traces to BC-6.3.001 postcondition): After `Config::save_global()` is called (simulated by running `jr auth login` or equivalent), `jr issue list --points` still shows the points column for the active profile.

**AC-003** (traces to BC-6.3.001 postcondition): `jr sprint current` shows the team and points columns after a `save_global()` round-trip.

**AC-004** (traces to BC-6.3.001 postcondition): `jr board view` shows the team column after a `save_global()` round-trip.

**AC-005** (traces to BC-6.3.001 postcondition — error message): When `story_points_field_id` is not configured for the active profile, the error message references `[profiles.<name>]` not `[fields]`. Specifically: `"Story points field not configured. Run \"jr init\" or set story_points_field_id under [profiles.<name>] in ~/.config/jr/config.toml"`.

**AC-006** (traces to BC-6.3.001 postcondition — error message): BC-2.2.021 error string in `src/cli/issue/list.rs:756-770` is updated from `set [fields].story_points_field_id in ...` to `set story_points_field_id under [profiles.<name>] in ...`.

## Risk / Dependencies

- **No other Wave 0 stories depend on this.** Independent.
- **No breaking change for single-profile users.** `config.active_profile()` for the "default" profile reads from `[profiles.default]`, where the field IDs are already stored correctly after `jr init`.
- **Risk (MEDIUM):** Some call sites may require `config.active_profile_or_err()?` instead of `config.active_profile()` because the function signature returns `Result<&Profile>` in some variants. Check `src/config.rs` before editing.
- **Risk (LOW):** If `Config::field_id(FieldKind)` accessor from ADR-0007 is not yet implemented, the fix still works by directly reading `config.active_profile().story_points_field_id`. Do not wait for the accessor if it doesn't exist — the direct path is acceptable.
- **Risk (LOW):** Snapshot tests (insta) that pin on the old `[fields]` error message string will fail — update the snapshot, don't revert the error message fix.

## Out of Scope

- `Config::save_global()` internals: the `#[serde(default, skip_serializing)]` annotation on the legacy `[fields]` block is correct and should remain. The fix is on the read side, not the write side.
- `jr init` flow: init writes to `profile.story_points_field_id` and `profile.team_field_id` already (per ADR-0007). No change needed there.
- Adding a `Config::field_id(FieldKind)` accessor: if it doesn't exist, the direct `active_profile()` path is sufficient for this story.
