# S-0.01: Fix `handle_open` to use `instance_url()` for OAuth profiles

## Context

`handle_open` in `src/cli/issue/workflow.rs:636` composes the browse URL as:

```rust
let url = format!("{}/browse/{}", client.base_url(), key);
```

For OAuth profiles, `client.base_url()` returns `https://api.atlassian.com/ex/jira/<cloudId>` — the API gateway, not the user-facing Jira instance. The resulting URL (`https://api.atlassian.com/ex/jira/<cloudId>/browse/FOO-1`) is invalid in a browser. This is a regression for every OAuth-mode user who runs `jr issue open`.

The fix is one line: replace `client.base_url()` with `client.instance_url()`. `client.instance_url()` returns the real `*.atlassian.net` URL in both api-token mode and OAuth mode.

BC-3.4.001 describes the fixed (target) behavior. H-046 is the MUST-FAIL holdout that pins the bug before this story ships; it becomes MUST-PASS after.

## Behavioral Contract

**BC-3.4.001** (`bc-3-issue-write.md`):

>

## Acceptance Criteria

**AC-001** (traces to BC-3.4.001 postcondition): Given an OAuth profile where `client.base_url()` returns `https://api.atlassian.com/ex/jira/my-cloud-123` and `client.instance_url()` returns `https://mycompany.atlassian.net`, when `jr issue open FOO-1 --url-only` is executed, then stdout contains `https://mycompany.atlassian.net/browse/FOO-1` and does NOT contain `api.atlassian.com`.

**AC-002** (traces to BC-3.4.001 postcondition): Given an api-token profile where `client.instance_url()` returns `https://mycompany.atlassian.net`, when `jr issue open FOO-1 --url-only` is executed, then stdout contains `https://mycompany.atlassian.net/browse/FOO-1`. (Regression guard: api-token mode is unaffected by this fix.)

**AC-003** (traces to BC-3.4.001 postcondition): The URL does not gain a trailing slash: `https://mycompany.atlassian.net/browse/FOO-1` not `https://mycompany.atlassian.net//browse/FOO-1`.

## Implementation Notes

**Current code** (`src/cli/issue/workflow.rs:636`):
```rust
let url = format!("{}/browse/{}", client.base_url(), key);
```

**Target code** (`src/cli/issue/workflow.rs:636`):
```rust
let url = format!("{}/browse/{}", client.instance_url(), key);
```

That is the entire code change. `instance_url()` is already implemented on `JiraClient` (see `src/api/client.rs`) and returns the profile's configured URL (or the `JR_BASE_URL` override in test mode, which routes to the mock server for both paths).

**Why `instance_url()` is correct for OAuth:** In `JiraClient::from_config`, `instance_url` is set from `profile.url` (the user-configured `*.atlassian.net` URL), while `base_url` for OAuth profiles is overridden to `https://api.atlassian.com/ex/jira/<cloudId>` for API calls. The browse URL must always use the human-facing instance URL.

**Verification of `JiraClient::new_for_test` behavior:** In test mode (when `JR_BASE_URL` is set), `from_config` sets `instance_url = test_override` (the mock server URL). So `client.instance_url()` in tests correctly returns the mock server base, enabling the integration test to verify URL construction without hitting a real Jira instance. H-046 fixture setup uses `JiraClient::new_for_test(base_url, auth_header)` — in that constructor, `instance_url` is also set to `base_url`, which is the mock server. The test sets up a separate check by constructing the expected URL from the known mock base rather than from `api.atlassian.com`.
