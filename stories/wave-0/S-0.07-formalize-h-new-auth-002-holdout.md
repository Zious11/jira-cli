# S-0.07: Formalize H-NEW-AUTH-002 holdout in `holdout-scenarios.md`

## Context

SD-002 resolved DRIFT-002: `JR_AUTH_HEADER` will be gated behind a compile-time flag excluding it from release binaries. Originally resolved as Option A (`#[cfg(test)]`); canonized to Option B-revised (`#[cfg(debug_assertions)]`) during S-0.05 implementation (2026-05-07) to preserve ~151 subprocess integration tests. The resolution document specifies that a new holdout (call it H-NEW-AUTH-002) must be added to `holdout-scenarios.md` to formalize the verification requirement for Phase 4 evaluation.

S-0.05 implements the `#[cfg(debug_assertions)]` gate. This story (S-0.07) closes the loop by:
1. Appending H-NEW-AUTH-002 to `holdout-scenarios.md` (incrementing `total_holdouts` from 48 to 49)
2. Ensuring the Phase 4 holdout evaluator has a precise, unambiguous spec for what to verify

This is a documentation/spec story. No Rust code is changed in this story. The implementation that satisfies the holdout is in S-0.05.

**Note:** `tdd_mode: facade` because this story produces a spec artifact (a markdown holdout spec), not an implementation with a Red Gate. The "test" is whether the holdout spec is unambiguous and correctly specifies the S-0.05 behavior.

## Implementation Notes

Append to `holdout-scenarios.md` in Group 6 (Reliability / MUST-FIX Pins), after H-NEW-MP-001:

```markdown
### H-NEW-AUTH-002: `JR_AUTH_HEADER` not honored in release build (`#[cfg(debug_assertions)]` gate verification)

**NFR source**: NFR-S-B
**BC**: BC-X.1.001
**SD**: SD-002 (Option B-revised — `#[cfg(debug_assertions)]` compile-time gate, canonized 2026-05-07)

**Setup**:
1. Build jr in release mode: `cargo build --release`
2. Set `JR_AUTH_HEADER=Basic dGVzdEBleGFtcGxlLmNvbTpmYWtl` (a Base64-encoded fake credential)
3. Empty `XDG_CONFIG_HOME` (no configured profiles, no keychain entries). Set `JR_SERVICE_NAME=jr-jira-cli-test`.

**Action**: `./target/release/jr auth status`

**Expected (FIXED behavior — MUST-PASS post-S-0.05)**:
- `JR_AUTH_HEADER` is NOT used as the auth header
- Exit non-zero (64 or 78) because no profile is configured
- stderr does NOT contain any reference to `dGVzdEBleGFtcGxlLmNvbTpmYWtl` or `fake`
- The binary behaves as if `JR_AUTH_HEADER` was not set

**Expected (MUST-FAIL pre-S-0.05)**:
- Current code at `src/api/client.rs:64-66` reads `JR_AUTH_HEADER` unconditionally
- The fake header is used, bypassing keychain lookup
- (This would only be visible if a mock server at `JR_BASE_URL` was also set — without it, the command exits with a URL error, but the fake header IS loaded into the client struct)

**Practical test note**: The gate is `#[cfg(debug_assertions)]`. Debug binaries (including `cargo_bin` subprocess binaries) still honor `JR_AUTH_HEADER`. The holdout must therefore use a RELEASE binary:
1. Spawns `./target/release/jr auth status` (release binary — `debug_assertions=false`)
2. Sets `JR_AUTH_HEADER=fake` in the child process environment
3. Asserts the exit code is 64 (no profile configured — NOT a fake-auth success)
4. Asserts stderr does not contain `api.atlassian.com` or any response from a real server

A debug subprocess test (`cargo_bin`) would NOT verify this holdout because debug binaries still honor `JR_AUTH_HEADER`. This requires a release binary to be present. The test can be gated behind `#[ignore]` and `JR_RUN_RELEASE_AUTH_GATE_TEST=1` to avoid requiring a release build in CI unit test runs.

**Status**: MUST-FIX (NFR-S-B). Current code fails this holdout.
**BC refs**: BC-X.1.001, SD-002
**Added**: Phase 2 Burst 1 (2026-05-06), resolves DRIFT-002
```
