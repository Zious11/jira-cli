# S-0.03: Fix multi-workspace asset HashMap to use composite `(workspace_id, oid)` key

## Context

`src/cli/issue/list.rs` enriches CMDB assets linked to Jira issues. The enrichment pipeline uses two HashMaps:

- `to_enrich: HashMap<(String, String), ()>` — tracks which assets need fetching (correctly uses `(wid, oid)` composite key at line 398)
- `resolved: HashMap<String, (String, String, String)>` — stores resolved asset data (incorrectly uses `oid` alone at line 446)

When two workspaces both contain an asset with `oid = "OBJ-88"` but different names/types, the `resolved` map experiences last-write-wins: the second workspace's data overwrites the first. Issues linked to workspace A's `OBJ-88` then display workspace B's asset name — silent data mis-attribution.

`api/assets/linked.rs::enrich_assets` (the library-level function) already uses the correct composite key. The bug is only in the `list.rs` inline enrichment path.

ADR-0008 documents the multi-workspace composite key design. BC-4.3.001 describes the fixed behavior. H-036 pins the MUST-FAIL holdout.

## Behavioral Contract

**BC-4.3.001** (`bc-4-assets-cmdb.md`):
> `resolved: HashMap<(String, String), (String, String, String)>` MUST use composite `(workspace_id, oid)` key.
>
> - Line 446: `resolved: HashMap<(String, String), _>` (change type from `HashMap<String, _>`)
> - Line 449: `resolved.insert((wid.clone(), oid.clone()), ...)` — workspace preserved
> - Line 456: `resolved.get(&(wid.clone(), oid.clone()))` — workspace-qualified lookup
>
> `to_enrich` (line 398) is already correct.

## Acceptance Criteria

**AC-001** (traces to BC-4.3.001 postcondition): Given wiremock setup with two workspaces `ws-A` and `ws-B`, both containing an asset with `oid = "OBJ-88"` but different names (`"Acme Corp"` for ws-A, `"Widgets Inc"` for ws-B), when `jr issue list --project PROJ --output json` is executed with issues linked to both workspace assets, then each issue's asset field shows the correct workspace-specific name (no last-write-wins collision).

**AC-002** (traces to BC-4.3.001 postcondition): Single-workspace tenants are unaffected: when all assets are from one workspace, the output is identical to pre-fix behavior.

**AC-003** (traces to BC-4.3.001 postcondition): The `to_enrich` HashMap (line 398, already correct) is not modified by this fix.
